一句话定位：Scene 是 DB-GPT 应用层最重要的"业务封装单元"。 它把 prompt 模板 + 输入预处理 + 模型调用 + 输出解析 + 副作用执行 五件套 封装成一个可插拔的类。每新增一种对话能力（如 NL2SQL、Excel 分析、Dashboard 报表）， 只需要新建一个 Scene 子包，而不是改动核心框架代码。

一、Scene 是什么：把业务封装成场景模板

如果你用过 DB-GPT 的 Web 界面，你会看到聊天框旁边有一个"对话模式"下拉框， 里面有 Chat Normal、Chat Data、Chat DB、 Dashboard、Chat Knowledge 等选项。 每选一种模式，整个后端的 prompt、数据注入方式、输出渲染逻辑都会完全不同。 这背后就是 Scene 在起作用。

本质上，一个 Scene 回答这样一个工程问题：

"怎么把一类业务（比如 NL2SQL）封装成可复用、可切换的模块？"

DB-GPT 的答案是把业务沉淀成 场景模板。每个模板由四个要素组成：

  1. Prompt 模板：System 指令 + 占位符变量（如 {table_info}、{context}）
  2. 输入预处理逻辑：把 select_param（数据库、文件、知识库）转换成 prompt 变量字典
  3. 输出解析器：把 LLM 返回的文本解析成结构化结果（如 SqlAction、ChartItem）
  4. 副作用执行器：在 do_action() 中执行 SQL、渲染图表、拼装引用

这四个要素在源码中对应四组文件，每个 Scene 子包的目录结构高度一致：

scene/chat_db/auto_execute/
  ├── __init__.py
  ├── chat.py          # BaseChat 子类（业务逻辑）
  ├── prompt.py        # prompt 模板 + 注册
  ├── out_parser.py    # 输出解析器
  ├── config.py        # Scene 专属配置
  └── example.py       # few-shot 示例（可选）

这种"约定大于配置"的目录布局让新 Scene 的接入成本极低，也使得 ChatFactory 可以用 BaseChat.__subclasses__() 自动发现所有已注册 Scene，无需维护一堆 if-else 分支。

二、Scene 抽象：ChatScene 与 AppScenePromptTemplateAdapter

整个 Scene 体系的"入口"是 base.py 中的两个核心类。 第一个是 ChatScene 枚举，它定义了所有支持的对话模式； 第二个是 AppScenePromptTemplateAdapter，它是每个 Scene 的"配置容器"。

2.1 ChatScene 枚举：对话模式的"户籍簿"

ChatScene 是一个 Enum，每个成员封装了一个 Scene 值对象。 Scene 类的字段包括 code（模式标识）、name（显示名）、describe（描述）、 param_types（前端要选择什么参数）、is_inner（是否内部 Scene）、prepare_scene_code（前置 Scene）等。

class Scene:
    def __init__(self, code, name, describe,
                 param_types=[], is_inner=False,
                 show_disable=False, prepare_scene_code=None):
        self.code = code
        self.name = name
        self.describe = describe
        self.param_types = param_types
        self.is_inner = is_inner
        self.prepare_scene_code = prepare_scene_code

class ChatScene(Enum):
    ChatWithDbExecute = Scene(
        code="chat_with_db_execute",
        name="Chat Data",
        describe="Dialogue with your private data...",
        param_types=["DB Select"],
    )
    ChatExcel = Scene(
        code="chat_excel",
        name="Chat Excel",
        param_types=["File Select"],
        prepare_scene_code="excel_learning",  # 前置场景
    )
    ChatDashboard = Scene(
        code="chat_dashboard",
        name="Dashboard",
        param_types=["DB Select"],
    )
    ChatKnowledge = Scene(
        code="chat_knowledge",
        param_types=["Knowledge Space Select"],
    )
    # ... 还有 ChatNormal / ChatAgent / ChatFlow / Extract* / QueryRewrite 等

    def value(self):       return self._value_.code
    def scene_name(self): return self._value_.name
    def param_types(self): return self._value_.param_types
    def is_inner(self):  return self._value_.is_inner
关键设计：ChatScene.ChatExcel 的 prepare_scene_code="excel_learning" 是一个非常重要的字段。它表示 ChatExcel 在正式对话前会先跑一个前置 Scene， 让 LLM 先学习 Excel 的列结构、做列名标准化。这种"Scene 串接"机制是 DB-GPT 独有的设计。

2.2 AppScenePromptTemplateAdapter：Scene 的"配置容器"

每个 Scene 不只是 prompt，还需要声明温度、max_new_tokens、是否流式、 用哪个输出解析器等。AppScenePromptTemplateAdapter 就是把这些"非业务逻辑" 的配置打包在一起。它在 prompt.py 中实例化，然后注册到 CFG.prompt_template_registry，供 BaseChat.__init__ 时按 chat_mode.value() 取出。

class AppScenePromptTemplateAdapter(BaseModel):
    prompt: ChatPromptTemplate              # 模板本体（System + History + Human）
    template_scene: Optional[str]            # 场景 code（如 "chat_normal"）
    template_is_strict: Optional[bool] = True  # 模板渲染是否严格模式
    output_parser: Optional[BaseOutputParser]  # 输出解析器
    stream_out: Optional[bool] = True          # 是否流式输出
    example_selector: Optional[ExampleSelector] # few-shot 示例选择器
    temperature: Optional[float] = 0.6       # 默认温度
    max_new_tokens: Optional[int] = 1024     # 默认最大新 token
    str_history: Optional[bool] = False       # 历史是否拍成字符串

注意 temperature / max_new_tokens 的三层覆盖优先级： API 请求 > app 配置文件 > prompt 模板默认值。这由 BaseChat.llm_temperature() 和 BaseChat.llm_max_new_tokens() 实现，保证运维侧和应用侧都能调参。

四、ChatFactory：Scene 的工厂模式实例化

ChatFactory 是一个单例（metaclass=Singleton）， 提供唯一入口 get_implementation(chat_mode, system_app, chat_param)。 它的策略非常巧妙：不做 if-else 分发，而是遍历 BaseChat.__subclasses__() 自动匹配。

class ChatFactory(metaclass=Singleton):
    @staticmethod
    def get_implementation(chat_mode, system_app, chat_param, **kwargs):
        # 懒加载：先 import 所有 Scene 子模块（让 __subclasses__ 生效）
        from dbgpt_app.scene.chat_normal.chat import ChatNormal
        from dbgpt_app.scene.chat_db.auto_execute.chat import ChatWithDbAutoExecute
        from dbgpt_app.scene.chat_dashboard.chat import ChatDashboard
        from dbgpt_app.scene.chat_data.chat_excel.excel_analyze.chat import ChatExcel
        from dbgpt_app.scene.chat_knowledge.v1.chat import ChatKnowledge
        # ... 其他 Scene import

        chat_classes = BaseChat.__subclasses__()
        implementation = None
        for cls in chat_classes:
            if cls.chat_scene == chat_mode:
                config = parse_config(system_app, chat_mode, type_class=cls.param_class())
                chat_param.app_config = config
                implementation = cls(**kwargs, chat_param=chat_param, system_app=system_app)
        if implementation is None:
            raise Exception(f"Invalid implementation name:{chat_mode}")
        return implementation
懒加载的意义：所有 Scene 的 import 都放在方法内部而非模块顶部， 是为了 避免循环依赖。Scene 子类依赖 BaseChat，而 BaseChat 又依赖 operators/app_operator，如果都顶部 import，初始化链会非常复杂。懒加载让 "第一次调用 get_implementation 时"才真正建立依赖图。

六、chat_normal：普通聊天（骨架详解）

chat_normal 是最简单的 Scene，几乎只做"用户输入 → LLM → 输出"。 理解它等于理解所有 Scene 的骨架。整个子包只有 4 个文件，加起来不到 50 行核心代码。

6.1 chat.py —— 极简的 BaseChat 子类

class ChatNormal(BaseChat):
    chat_scene: str = ChatScene.ChatNormal.value()

    @classmethod
    def param_class(cls) -> Type[ChatNormalConfig]:
        return ChatNormalConfig

    def __init__(self, chat_param: ChatParam, system_app: SystemApp):
        super().__init__(chat_param=chat_param, system_app=system_app)

注意 每个 BaseChat 子类必须声明类属性 chat_scene， 这是 ChatFactory 匹配的"身份证"。还要实现 param_class() 返回该 Scene 的配置类， 供 parse_config 从 app 配置文件反序列化。

6.2 prompt.py —— 模板与注册

PROMPT_SCENE_DEFINE_EN = "You are a helpful AI assistant."
PROMPT_SCENE_DEFINE_ZH = "你是一个有用的 AI 助手。"
PROMPT_SCENE_DEFINE = PROMPT_SCENE_DEFINE_ZH if CFG.LANGUAGE == "zh" else PROMPT_SCENE_DEFINE_EN

prompt = ChatPromptTemplate(
    messages=[
        SystemPromptTemplate.from_template(PROMPT_SCENE_DEFINE),
        MessagesPlaceholder(variable_name="chat_history"),
        MessagesPlaceholder(variable_name="media_input"),
        HumanPromptTemplate.from_template("{input}"),
    ]
)

prompt_adapter = AppScenePromptTemplateAdapter(
    prompt=prompt,
    template_scene=ChatScene.ChatNormal.value(),
    stream_out=True,
    output_parser=NormalChatOutputParser(),
)

CFG.prompt_template_registry.register(prompt_adapter, language=CFG.LANGUAGE, is_default=True)

这里有几个关键点：

  • ChatPromptTemplate 由四条消息组成：System 指令、历史消息占位符、媒体输入占位符、用户输入模板
  • MessagesPlaceholder 是一种特殊的模板变量，运行时会被替换成一组 Message 对象（而非字符串）
  • 注册到 registry 时要传 language 和 is_default，registry 会按 (scene_code, language) 建索引

6.3 out_parser.py —— 最朴素的解析器

class NormalChatOutputParser(BaseOutputParser):
    def parse_prompt_response(self, model_out_text) -> T:
        return model_out_text  # 直接返回原文，不做结构化解析

    def get_format_instructions(self) -> str:
        pass

普通聊天不需要结构化输出，所以解析器只是"原样返回"。对比之下， NL2SQL 的解析器要做 JSON 解析 + SQL 提取，Dashboard 解析器要解析 JSON 数组， 复杂度差异巨大。这是 Scene 设计的精髓：解析器的复杂度 = 业务的复杂度。

七、chat_data：Excel/CSV 数据问答

chat_data 场景让用户上传 Excel/CSV 文件后，用自然语言提问， 系统自动生成 DuckDB SQL 并执行，返回表格或图表。它是最复杂的 Scene 之一， 因为涉及 Excel 解析 → 列名标准化 → DuckDB 建表 → SQL 生成 → 执行 → 渲染 全链路。

7.1 ChatExcel 的初始化与 ExcelReader

class ChatExcel(BaseChat):
    chat_scene: str = ChatScene.ChatExcel.value()

    def __init__(self, chat_param: ChatParam, system_app: SystemApp):
        self.fs_client = FileStorageClient.get_instance(system_app)
        self.select_param = chat_param.select_param
        # 解析文件路径（支持绝对路径 / dict / JSON / file:// URI）
        file_path, file_name, database_file_path, database_file_id = self._resolve_path(...)
        self._curr_table = "data_analysis_table"
        self.excel_reader = ExcelReader(
            chat_param.chat_session_id, file_path, file_name,
            read_type="direct",
            database_name=database_file_path,  # DuckDB 文件路径
            table_name=self._curr_table,
        )
        self.api_call = ApiCall()
        super().__init__(chat_param=chat_param, system_app=system_app)

ExcelReader 是这个 Scene 的核心模块。它用 DuckDB 作为内存分析引擎， 把 Excel/CSV 数据导入到一张名为 data_analysis_table 的 DuckDB 表中， 后续所有 SQL 都跑在这张表上。DuckDB 的列式存储和向量化执行让单机分析百万行数据也能秒级响应。

7.2 前置 Scene：ExcelLearning

ChatExcel 重写了 prepare()，在正式对话前先跑一个内部的 ExcelLearning Scene。 ExcelLearning 让 LLM 分析 Excel 的每一列，做列名标准化（中文 → 英文小写下划线）， 生成数据分析建议。这样后续对话时，prompt 里就能提供更规范的表结构。

async def prepare(self):
    if self.has_history_messages():
        return None  # 只在首轮学习
    chat_param = ChatParam(
        chat_session_id=self.chat_session_id,
        current_user_input="[" + self.excel_reader.excel_file_name + "] Analyze！",
        select_param=self.select_param,
        chat_mode=ChatScene.ExcelLearning,  # 切换到学习模式
        model_name=self.model_name,
    )
    learn_chat = ExcelLearning(chat_param, system_app=self.system_app,
                               parent_mode=self.chat_mode, excel_reader=self.excel_reader)
    result = await learn_chat.nostream_call()
    return result

八、chat_db：NL2SQL 经典场景

chat_db 是 DB-GPT 最经典的场景：用户选择数据库后用自然语言提问， LLM 生成 SQL，系统自动执行并返回结果。这个场景分为两个子模式：

  • auto_execute（chat_with_db_execute）：生成 SQL 后自动执行
  • professional_qa（chat_with_db_qa）：只生成 SQL 不执行，纯问答

8.1 表结构检索：DBSummaryClient

一个数据库可能有几百张表，全部塞进 prompt 会爆 token。 ChatWithDbAutoExecute 的 generate_input_values() 通过 DBSummaryClient.get_db_summary() 做语义检索，只返回 TOP-K 张最相关的表结构。 如果检索失败，降级到 database.table_simple_info() 并按 token 截断。

async def generate_input_values(self) -> Dict:
    client = DBSummaryClient(system_app=self.system_app)
    try:
        table_infos = await blocking_func_to_async(
            self._executor, client.get_db_summary,
            self.db_name, user_input, self.curr_config.schema_retrieve_top_k,
        )
    except Exception:
        table_infos = await blocking_func_to_async(
            self._executor, self.database.table_simple_info
        )
        if len(table_infos) > self.curr_config.schema_max_tokens:
            table_infos = table_infos[:self.curr_config.schema_max_tokens]

    input_values = {
        "db_name": self.db_name,
        "user_input": user_input,
        "top_k": self.curr_config.max_num_results,  # 默认 50
        "dialect": self.database.dialect,           # mysql/postgresql/...
        "table_info": table_infos,
        "display_type": self._generate_numbered_list(),
    }
    return input_values

8.3 do_action：返回 SQL 执行闭包

注意 do_action 不是直接执行 SQL，而是返回一个闭包 self.database.run_to_df。这个闭包会在 parse_view_response 中被调用，传入 LLM 生成的 SQL 来执行并获取 DataFrame。

DbChatOutputParser.parse_view_response 中调用这个闭包执行 SQL， 把结果 DataFrame 转 JSON，组装成 <chart-view> XML 标签返回给前端。 如果 SQL 执行失败，会返回错误信息的红色标签。

九、chat_dashboard：图表看板生成

chat_dashboard 是 chat_db 的"升级版"：不是只生成一条 SQL， 而是一次生成 4-8 条 SQL，每条对应一个图表，组成一个完整的看板（Dashboard）。 这个场景的 do_action 会批量执行 SQL 并组装 ReportData 对象。

9.2 do_action：批量执行 + ReportData 组装

def do_action(self, prompt_response):
    chart_datas: List[ChartData] = []
    dashboard_data_loader = DashboardDataLoader()
    for chart_item in prompt_response:  # 遍历每个图表项
        try:
            field_names, values = dashboard_data_loader.get_chart_values_by_conn(
                self.database, chart_item.sql
            )
            chart_datas.append(ChartData(
                chart_uid=str(uuid.uuid1()),
                chart_name=chart_item.title,
                chart_type=chart_item.showcase,    # 如 response_pie_chart
                chart_desc=chart_item.thoughts,
                chart_sql=chart_item.sql,
                column_name=field_names,
                values=values,
            ))
        except Exception as e:
            logger.warning(f"Failed to get chart data: {str(e)}")
    return ReportData(
        conv_uid=self.chat_session_id,
        template_name=self.report_name,
        charts=chart_datas,
    )

注意 单条 SQL 失败不会中断整个看板生成， 失败的图表会被跳过，成功的图表正常展示。这种容错策略保证了即使 LLM 生成了一条错误 SQL， 用户也能看到部分结果。Dashboard 的输出解析器 ChatDashboardOutputParser 还实现了 JSON 清洗和正则兜底提取， 应对 LLM 输出格式不规范的情况。

十、chat_knowledge：RAG 知识库问答

chat_knowledge 是 RAG 在应用层的落地。用户选择知识库后提问， 系统先做向量检索（可选 QueryRewrite 和 Rerank），再把检索到的 chunk 作为 context 拼入 prompt，最后让 LLM 基于上下文回答，并附带引用来源。

10.1 KnowledgeSpaceRetriever 的初始化

self._space_retriever = KnowledgeSpaceRetriever(
    space_id=space.id,
    embedding_model=self.model_config.default_embedding,
    top_k=retriever_top_k,       # 检索数量（开启 rerank 时会放大）
    query_rewrite=query_rewrite, # 可选的 QueryRewrite
    rerank=reranker,             # 可选的 RerankEmbeddingsRanker
    llm_model=self.llm_model,
    system_app=self.system_app,
)
Rerank 时的 top_k 放大策略：如果开启了 reranker， 且检索 top_k 小于 rerank_top_k 或小于 20，会把检索 top_k 提升到 max(rerank_top_k, 20)。这是因为 rerank 需要更大的候选集才能有效排序。

10.2 generate_input_values：检索 + 拼接 context

async def generate_input_values(self) -> Dict:
    # 如果知识库有自定义 prompt，替换默认模板
    if self.space_context and self.space_context.get("prompt"):
        self.prompt_template.prompt = ChatPromptTemplate(
            messages=[
                SystemPromptTemplate.from_template(self.space_context["prompt"]["template"]),
                MessagesPlaceholder(variable_name="chat_history"),
                HumanPromptTemplate.from_template("{question}"),
            ]
        )

    user_input = self.current_user_input.last_text
    tasks = [self.execute_similar_search(user_input)]
    candidates_with_scores = await run_async_tasks(tasks=tasks, concurrency_limit=1)
    candidates_with_scores = reduce(lambda x, y: x + y, candidates_with_scores)

    context = "\n".join([doc.content for doc in candidates_with_scores])
    relations = list(set([os.path.basename(str(d.metadata.get("source", "")))
                         for d in candidates_with_scores]))
    input_values = {
        "context": context,
        "question": user_input,
        "relations": relations,
    }
    return input_values

十一、输出解析器：out_parser.py

输出解析器是 Scene 设计的"第三条腿"。BaseChat 的 stream_call 和 nostream_call 在拿到 LLM 输出后，会依次调用解析器的三个方法：

  1. parse_model_stream_resp_ex(output) —— 流式解析（逐 token）
  2. parse_model_nostream_resp(model_output) —— 非流式解析（一次性）
  3. parse_prompt_response(ai_response_text) —— 把纯文本解析成结构化对象
  4. parse_view_response(speak, data, prompt_response) —— 把结构化对象转成前端可渲染的 view

不同 Scene 的解析器复杂度差异巨大。下面是三个典型例子：

Scene解析器解析策略结构化结果
chat_normal NormalChatOutputParser 直接返回原文 纯字符串
chat_db DbChatOutputParser JSON 解析 + SQL 语法兜底检测 SqlAction(sql, thoughts, display, direct_response)
chat_dashboard ChatDashboardOutputParser JSON 数组解析 + 清洗 + 正则兜底 List[ChartItem(sql, title, thoughts, showcase)]

DbChatOutputParser 中有一个巧妙的兜底逻辑：如果 LLM 返回的不是 JSON 而是纯 SQL （社区模型常见行为），is_sql_statement() 会用 sqlparse 检测， 如果是合法 SQL 语句就直接包装成 SqlAction：

def parse_prompt_response(self, model_out_text):
    clean_str = super().parse_prompt_response(model_out_text)
    # 兼容社区纯 SQL 输出模型
    if self.is_sql_statement(clean_str):
        return SqlAction(clean_str, "", "", "")
    else:
        response = json.loads(clean_str, strict=False)
        # 提取 sql / thoughts / display_type / direct_response
        return SqlAction(sql=sql, thoughts=thoughts, display=display, direct_response=resp)

ChatDashboardOutputParser 则更进一步，实现了三级降级策略： 先尝试 json.loads，失败后调用 _clean_json_string （去除 markdown 代码块、修复转义），再失败用 _extract_json_fallback 正则提取。这种"防御性编程"反映了 LLM 输出的不确定性 —— 即使 prompt 要求严格 JSON， 模型也可能输出带 markdown 包裹或格式错误的内容。

十二、Scene 与 AWEL 的衔接：operators/app_operator.py

Scene 不是孤立的体系，它通过 operators/app_operator.py 中的两个核心 Operator 与 AWEL 工作流无缝衔接。这两个 Operator 是 AppChatComposerOperator 和 build_cached_chat_operator。

12.1 AppChatComposerOperator：消息编排器

AppChatComposerOperator 是一个 MapOperator[ChatComposerInput, ModelRequest]， 它的职责是把 (历史消息 + prompt 变量字典) 编排成最终发给 LLM 的 ModelRequest。 它内部会构建一个子 DAG：

flowchart TD Input(["ChatComposerInput (messages, prompt_dict)"]) MsgMap["MapOperator(λ x → x.messages)"] HistoryTransform["HistoryTransformTask
(BufferWindow 或 TokenBuffer 裁剪历史)"] PromptBuilder["HistoryPromptBuilderOperator
(拼装 System + History + Human)"] DictMap["MapOperator(λ x → x.prompt_dict)"] Output(["ModelRequest"]) Input --> MsgMap MsgMap --> HistoryTransform HistoryTransform --> PromptBuilder Input --> DictMap DictMap --> PromptBuilder PromptBuilder --> Output

历史消息的裁剪策略由 memory 配置决定：

  • BufferWindowGPTsAppMemoryConfig：按轮次裁剪（保留前 N 轮 + 后 M 轮）
  • TokenBufferGPTsAppMemoryConfig：按 token 数裁剪（超过限制就裁掉最旧消息）

裁剪后的历史消息和 prompt 变量一起喂给 HistoryPromptBuilderOperator， 它会用 ChatPromptTemplate 把变量填入模板，生成最终的 Message 列表。 最终输出是一个完整的 ModelRequest，包含模型名、温度、max_new_tokens、消息列表等。

12.2 build_cached_chat_operator：带缓存的 LLM 调用

build_cached_chat_operator 构建了一个带缓存分支的 DAG， 在调用 LLM 前先查缓存，命中则直接返回，未命中则调用 LLM 并写回缓存。 支持流式和非流式两种模式：

flowchart TD Input(["Input"]) CacheBranch{"CacheBranch"} LLMOperator["LLMOperator"] SaveCacheOperator["SaveCacheOperator"] CachedModelOperator["CachedModelOperator
(命中缓存直接返回)"] Join(["Join"]) Input --> CacheBranch CacheBranch -- "未命中" --> LLMOperator LLMOperator --> SaveCacheOperator SaveCacheOperator --> Join CacheBranch -- "命中" --> CachedModelOperator CachedModelOperator --> Join 
with DAG("dbgpt_awel_app_model_infer_with_cached"):
    input_task = InputOperator(SimpleCallDataInputSource())

    if is_streaming:
        llm_task = StreamingLLMOperator(llm_client, task_name="llm_model_node")
        cache_task = CachedModelStreamOperator(cache_manager, ...)
        save_cache_task = ModelStreamSaveCacheOperator(cache_manager)
    else:
        llm_task = LLMOperator(llm_client, task_name="llm_model_node")
        cache_task = CachedModelOperator(cache_manager, ...)
        save_cache_task = ModelSaveCacheOperator(cache_manager)

    cache_check_branch_task = ModelCacheBranchOperator(
        cache_manager,
        model_task_name="llm_model_node",
        cache_task_name="llm_model_cache_node",
    )
    join_task = BranchJoinOperator()

    input_task >> cache_check_branch_task
    cache_check_branch_task >> llm_task >> save_cache_task >> join_task
    cache_check_branch_task >> cache_task >> join_task
    return join_task
Scene 与 AWEL 的分层关系：Scene 在上层负责业务编排 （选择 prompt、注入数据、解析输出），AWEL 在下层负责通用编排 （缓存分支、历史裁剪、模型调用）。BaseChat 是两层之间的桥梁， 它在 _build_model_request 中实例化 AppChatComposerOperator， 在 call_llm_operator / call_streaming_operator 中调用 build_cached_chat_operator。

十三、实战指南：怎么扩展一个新 Scene

假设你要新增一个 "chat_api" Scene —— 用户提问后， LLM 判断是否需要调用外部 API（如天气、汇率），如果需要就生成 API 调用指令并执行。 按照 DB-GPT 的 Scene 约定，你需要做以下几步：

步骤 1：在 ChatScene 枚举中注册新模式

# scene/base.py
ChatApi = Scene(
    code="chat_api",
    name="Chat API",
    describe="Call external APIs through natural language.",
    param_types=["API Select"],
)

步骤 2：创建子包目录

scene/chat_api/
  ├── __init__.py
  ├── chat.py        # ChatApi(BaseChat)
  ├── prompt.py      # prompt 模板 + 注册
  ├── out_parser.py  # ApiCallOutputParser
  └── config.py      # ChatApiConfig(GPTsAppCommonConfig)

步骤 3：编写 ChatApi 子类

# scene/chat_api/chat.py
class ChatApi(BaseChat):
    chat_scene: str = ChatScene.ChatApi.value()

    @classmethod
    def param_class(cls):
        return ChatApiConfig

    def __init__(self, chat_param, system_app):
        self.api_config = chat_param.select_param  # 用户选择的 API
        super().__init__(chat_param=chat_param, system_app=system_app)

    async def generate_input_values(self) -> Dict:
        return {
            "api_list": self.api_config.to_dict(),
            "user_input": self.current_user_input.last_text,
        }

    def do_action(self, prompt_response):
        # prompt_response 是解析后的 ApiAction 对象
        return lambda api_name, params: self._call_api(api_name, params)

步骤 4：编写 prompt 和注册

# scene/chat_api/prompt.py

    SystemPromptTemplate.from_template("""

完成后，前端只要在"对话模式"下拉框里加上 chat_api 选项， 后端就能自动路由到你的新 Scene。整个过程不需要修改 BaseChat 或任何核心代码， 只是在子包里新增文件。这就是 Scene 设计的核心价值 —— 开闭原则。

扩展检查清单： 1. ChatScene 枚举里注册 code 和 param_types； 2. chat.py 里 chat_scene 类属性 == ChatScene.XXX.value()； 3. prompt.py 里注册到 CFG.prompt_template_registry； 4. out_parser.py 里实现 parse_prompt_response 和 parse_view_response； 5. config.py 里定义 GPTsAppCommonConfig 子类（即使是空的也要有）； 6. chat_factory.py 里加一行 import 让 __subclasses__ 能发现你的类。
