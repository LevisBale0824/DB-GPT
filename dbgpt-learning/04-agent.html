一、Agent 是什么 · 与 AWEL 的关系

在 DB-GPT 中，Agent 是一个具有角色（Role）、能通过 LLM 思考（Thinking）、调用资源（Resource）执行动作（Action）、并基于记忆（Memory）自我演化的自治单元。它把"自然语言意图 → 可验证结果"的过程封装为一条标准管线：

  • 接收消息（receive） → 生成回复（generate_reply）→ 思考（thinking）→ 审查（review）→ 行动（act）→ 校验（verify）→ 循环直至通过或终止
  • 每一轮的中间产物以 AgentMessage 的形式在 Agent 之间、Agent 与 Memory 之间流转

Agent 与 AWEL 的关系是 DB-GPT 架构中最容易被误解的部分：

  • AWEL（第 03 章）是横向的 DAG 编排层，负责把 Operator 串成工作流
  • Agent 是纵向的自治单元，内部自带一个 "Thinking → Act → Verify" 循环
  • 两者通过 core/plan/awel/agent_operator.py 中的 WrappedAgentOperator / AWELAgentOperator 桥接：把一个 Agent 包成一个 AWEL MapOperator，这样 Agent 就能在 DAG 中被串联
设计哲学：AWEL 适合确定性的数据流编排（如 RAG → LLM → 后处理），而 Agent 适合需要 LLM 自主决策的循环任务（如"先查库，若空则搜索，再综合作答"）。两者在 team_awel_layout.py 中融合为"AWEL Layout Team"，即用 AWEL DAG 描述团队协作拓扑。

1.1 Agent 的本质：一个被 LLM 驱动的状态机

从计算机科学的角度看，DB-GPT 的 Agent 本质上是一个由 LLM 驱动的有限状态机。每一轮 generate_reply 循环就是一次状态转换：

  • 输入状态：当前轮次的消息历史（AgentMessage 列表）+ 资源 prompt + 记忆
  • 转换函数：LLM 推理（thinking）+ 动作执行（act）
  • 输出状态：新的 AgentMessage（含 action_report 和 observation）
  • 终止条件：verify 通过、terminate 动作、retry 耗尽、超时

与传统的状态机不同，这里的"转换函数"是不确定性的——同样的输入，LLM 可能给出不同的推理路径。这就是为什么需要 verify 环节和 max_retry_count 保护：它们是在用确定性框架约束非确定性推理。

1.2 ConversableAgent：Agent 接口的标准实现

Agent 只是 ABC 接口，真正的实现是 ConversableAgent（继承 Role + Agent）。Role 来自 core/role.py，提供身份信息（name、role、desc、goal）和运行模式（AgentRunMode）。ConversableAgent 在此基础上添加了 LLM 配置、Action 列表、Resource 引用、Memory 句柄、线程池执行器等运行时状态。

理解这一点很重要：Agent 的"身份"和"能力"是分离的。身份由 ProfileConfig 描述（名字、角色、系统提示词），能力由 actions + resource + llm_config 决定。这意味着同一个 Agent 类（如 ReActAgent），配上不同的 Profile 和 Resource，就能表现为完全不同的角色——这是 DB-GPT Agent 系统灵活性的根源。

2.1 Agent 接口定义

所有 Agent 的契约都在 core/agent.py 里：Agent 是一个纯抽象基类（ABC），只定义行为契约，不持有状态。

packages/dbgpt-core/src/dbgpt/agent/core/agent.py L16-L60 
class Agent(ABC):
    """Agent Interface."""

    @abstractmethod
    async def send(
        self,
        message: AgentMessage,
        recipient: "Agent",
        reviewer: Optional["Agent"] = None,
        request_reply: Optional[bool] = True,
        is_recovery: Optional[bool] = False,
        silent: Optional[bool] = False,
        is_retry_chat: bool = False,
        last_speaker_name: Optional[str] = None,
        rely_messages: Optional[List[AgentMessage]] = None,
        historical_dialogues: Optional[List[AgentMessage]] = None,
    ) -> None:
        """Send a message to recipient agent."""

    @abstractmethod
    async def receive(self, message, sender, ...) -> None:
        """Receive a message from another agent."""

    @abstractmethod
    async def generate_reply(self, received_message, sender, ...) -> AgentMessage:
        """Generate a reply based on the received messages."""

    @abstractmethod
    async def thinking(self, messages, sender=None, prompt=None, ...) -> Tuple:
        """Think and reason about the current task goal."""

    @abstractmethod
    async def act(self, message, sender, reviewer=None, ...) -> "ActionOutput":
        """Act based on the LLM inference results."""

    @abstractmethod
    async def verify(self, message, sender, reviewer=None, ...) -> Tuple[bool, Optional[str]]:
        """Verify whether the current execution results meet the target expectations."""

注意签名中有大量"协作参数"：reviewer（审查者）、rely_messages（依赖的前序消息）、historical_dialogues（历史对话）、is_retry_chat（是否重试）、last_speaker_name（上一个发言者）—— 这些都是为多 Agent 团队协作设计的，单 Agent 场景大部分留空即可。

三、Agent 生命周期与 a_call 主循环

真正实现所有逻辑的是 ConversableAgent（core/base_agent.py，约 900+ 行），它同时继承自 Role 和 Agent。其核心方法 generate_reply 被标记为 @final，子类不可覆盖——这是为了强制所有 Agent 走同一条经过精心打磨的主循环。

3.1 build()：构建阶段

Agent 在被使用前必须经过 build() 完成初始化：

packages/dbgpt-core/src/dbgpt/agent/core/base_agent.py L190-L224 
async def build(self, is_retry_chat: bool = False) -> "ConversableAgent":
    # 1. 预加载资源（如连接数据库、加载知识库索引）
    await self.preload_resource()
    # 2. 检查可用性：identity、context、action、resource、llm 五项检查
    self.check_available()
    _language = self.not_null_agent_context.language
    if _language:
        self.language = _language

    # 3. 为每个 Action 注入资源引用
    for action in self.actions:
        action.init_resource(self.resource)

    # 4. 初始化 LLM 客户端和记忆系统
    if not self.is_human:
        if not self.llm_config or not self.llm_config.llm_client:
            raise ValueError("LLM client is not initialized！")
        self.llm_client = AIWrapper(llm_client=self.llm_config.llm_client)
        real_conv_id, _ = parse_conv_id(self.not_null_agent_context.conv_id)
        memory_session = f"{real_conv_id}_{self.role}_{self.name}"
        self.memory.initialize(
            self.name,
            self.llm_config.llm_client,
            importance_scorer=self.memory_importance_scorer,
            insight_extractor=self.memory_insight_extractor,
            session_id=memory_session,
        )
        # 克隆一份记忆结构（避免多会话间共享）
        self.memory = self.memory.structure_clone()
        action_outputs = await self.memory.gpts_memory.get_agent_history_memory(
            real_conv_id, self.role
        )
        await self.recovering_memory(action_outputs)
    return self

3.2 check_available()：五项安全检查

在 build() 阶段，check_available() 会执行五项严格的可用性检查，任何一项不通过都会抛出 ValueError：

  • 身份检查（identity_check）：确保 Agent 有 name 和 role
  • 上下文检查：确保 agent_context 不为 None
  • Action 检查：确保每个 Action 所需的 Resource 类型都已绑定
  • 人类/团队豁免：如果是 human 或 team 类型，跳过 Action 检查
  • LLM 检查：确保非人类 Agent 有 llm_config 和 llm_client

这种"构建时检查"的设计避免了运行时才发现缺少关键依赖的问题，是工程化 Agent 系统的重要实践。

3.3 generate_reply()：不可覆盖的主循环

这是整个 Agent 系统的"心脏"，它在 max_retry_count 次循环中反复执行"思考 → 审查 → 行动 → 校验"，直到通过或耗尽重试次数：

packages/dbgpt-core/src/dbgpt/agent/core/base_agent.py L502-L726（精简版） 
while current_retry_counter < self.max_retry_count:
    # 加载思考所需的消息（含历史记忆、资源 prompt）
    thinking_messages, resource_info = await self._load_thinking_messages(
        received_message=received_message,
        sender=sender,
        observation=observation,
        rely_messages=rely_messages,
        ...
    )

    # 1. THINKING：让 LLM 推理下一步动作
    llm_reply, model_name = await self.thinking(
        thinking_messages, sender, stream_callback=_llm_stream_callback
    )
    reply_message.content = llm_reply

    # 2. REVIEW：审查 LLM 输出是否合规
    approve, comments = await self.review(llm_reply, self)
    reply_message.review_info = AgentReviewInfo(approve=approve, comments=comments)

    # 3. ACT：解析并执行动作
    act_out: ActionOutput = await self.act(
        message=reply_message,
        sender=sender,
        reviewer=reviewer,
        **act_extent_param,
    )
    reply_message.action_report = act_out

    # 4. VERIFY：校验执行结果
    check_pass, reason = await self.verify(reply_message, sender, reviewer)

    # 5. 自我优化：未通过则把 fail_reason 作为新的 observation 进入下一轮
    if not check_pass:
        if not act_out.have_retry:
            break  # 无法重试，放弃
        fail_reason = reason
        observation = fail_reason
        await self.write_memories(...)
    else:
        observation = act_out.observations
        await self.write_memories(...)
        # 非 LOOP 模式或动作要求终止，则结束
        if self.run_mode != AgentRunMode.LOOP or act_out.terminate:
            break

    # 超时保护
    if time.time() - start_time > self.max_timeout:
        break

    current_retry_counter += 1

# 6. 最终消息调整
await self.adjust_final_message(is_success, reply_message)
return reply_message

这个循环的精妙之处在于自我修复能力：当 verify 失败时，失败原因会被注入为新的 observation，LLM 在下一轮会看到"上一步失败了，原因是 XXX"，从而调整策略。这正是 ReAct 模式的核心。

流式输出集成：主循环中 _emit_stream 函数会在每个阶段（thinking、act）发射事件（thinking_chunk、thinking、act），前端 SSE 层接收这些事件后可以实时展示推理过程。每个事件都带 round 字段标识当前轮次，方便用户区分"首次推理"和"重试推理"。

3.4 thinking()：LLM 推理的重试策略

thinking 方法负责调用 LLM 生成回复。它内置了 3 次自动重试机制，以应对速率限制和网络波动：

  • 每次重试会切换到不同的 LLM 模型（通过 _a_select_llm_model）
  • 两次重试之间有 10 秒的等待
  • 当 LLM 返回 context_too_long 错误时，会触发被动上下文压缩（Layer 4），由 ContextManager.reactive_compact 处理
  • 所有参数（max_new_tokens、temperature、verbose）都从 AgentContext 中读取，保证整个会话的推理参数一致

3.5 act()：动作执行的流水线

ConversableAgent.act() 会遍历所有注册的 Action，依次执行。每个 Action 可以通过 parse_action 方法判断是否应该处理当前消息——如果不应该，返回 None 跳过。前一个 Action 的输出（rely_action_out）会作为后一个 Action 的输入，形成一条动作流水线：

class Resource(ABC, Generic[P]):
    """Resource for the agent."""

    @classmethod
    @abstractmethod
    def type(cls) -> ResourceType:
        """Return the resource type."""

    @property
    @abstractmethod
    def name(self) -> str:
        """Return the resource name."""

    async def preload_resource(self):
        """预加载（如连接数据库、加载索引）。"""
        pass

    @abstractmethod
    async def get_prompt(self, *, lang="en", question=None, ...) -> Tuple[str, Optional[Dict]]:
        """返回要注入 LLM prompt 的文本和资源引用信息。"""

    def execute(self, *args, resource_name=None, **kwargs) -> Any:
        """同步执行资源（如执行 SQL）。"""
        raise NotImplementedError

    async def async_execute(self, *args, resource_name=None, **kwargs) -> Any:
        """异步执行资源。"""
        raise NotImplementedError

    def get_resource_by_type(self, resource_type: ResourceType) -> List["Resource"]:
        """按类型过滤子资源。"""

4.2 DBResource 示例

以数据库资源为例，它的 get_prompt 会返回表结构 Schema，execute 会执行 SQL：

packages/dbgpt-core/src/dbgpt/agent/resource/database.py L31-L117（精简） 
class DBResource(Resource[P], Generic[P]):
    """Database resource class."""

    def __init__(self, name, db_type=None, db_name=None, dialect=None, ...):
        self._name = name
        self._db_type = db_type
        self._db_name = db_name
        self._dialect = dialect or db_type
        self._executor = executor or ThreadPoolExecutor()

    @classmethod
    def type(cls) -> ResourceType:
        return ResourceType.DB

    @cached(cachetools.TTLCache(maxsize=100, ttl=10))
    async def get_prompt(self, *, lang="en", question=None, ...) -> Tuple:
        if not self._db_name:
            return "No database name provided.", None
        # 调用 schema_link 获取相关表结构（10 秒缓存）
        schema_info = await blocking_func_to_async(
            self._executor, self.get_schema_link, db=self._db_name, question=question
        )
        return (
            self._prompt_template.format(db_type=self._db_type, schemas=schema_info),
            None,
        )

    def execute(self, *args, resource_name=None, **kwargs):
        copy_kwargs = kwargs.copy()
        if "db" not in copy_kwargs:
            copy_kwargs["db"] = self._db_name
        return self._sync_query(*args, **copy_kwargs)

    @property
    def is_async(self) -> bool:
        return True  # DBResource 支持异步执行
缓存策略：@cached(cachetools.TTLCache(maxsize=100, ttl=10)) 表示 Schema prompt 有 10 秒的 TTL 缓存——因为表结构不会频繁变化，但在开发期可能手动改表， 所以不能永久缓存。这是 DB-GPT 在性能与正确性之间的平衡选择。

4.3 ResourcePack：资源聚合

当一个 Agent 需要同时使用数据库 + 知识库 + 多个工具时，就用 ResourcePack 把它们打包：

packages/dbgpt-core/src/dbgpt/agent/resource/pack.py L21-L80（精简） 
class ResourcePack(Resource[PackResourceParameters]):
    """Resource pack class."""

    def __init__(self, resources: List[Resource], name="Resource Pack", ...):
        self._resources: Dict[str, Resource] = {
            resource.name: resource for resource in resources
        }

    @classmethod
    def type(cls) -> ResourceType:
        return ResourceType.Pack

    @property
    def is_pack(self) -> bool:
        return True

    async def get_prompt(self, *, lang="en", question=None, ...):
        # 遍历所有子资源，拼接 prompt
        prompt_list = []
        info_map = {}
        for name, resource in self._resources.items():
            prompt, resource_reference = await resource.get_prompt(
                lang=lang, question=question, ...
            )
            prompt_list.append(prompt)
            if resource_reference is not None:
                info_map.update(resource_reference)
        return self._prompt_separator.join(prompt_list), info_map

ReActAgent（expand/react_agent.py）是 DB-GPT 中最常用的 Agent 实现，它采用 Reasoning + Acting 范式，让 LLM 在"思考"和"行动"之间交替推进。

5.6 容错与降级解析

ReActAction 内置了多层容错机制来处理 LLM 输出不规范的情况：

  • JSON 解析失败时：_fallback_parse_args 方法会根据工具参数定义，用正则表达式提取参数值。对于单参数工具，直接把原始输入作为参数值；对于多参数工具，按参数名在文本中的位置定位并提取值
  • HTML 内容特殊处理：_extract_html_interpreter_args 方法专门处理 html_interpreter 工具——因为 HTML 内容中大量双引号会破坏 JSON 解析，它采用反向搜索策略：先找到最后一个 "title" 键，再向前提取 HTML 内容
  • 多步骤容错：当 LLM 在一次输出中包含多个 ReAct 步骤时，只执行第一步并给出警告
  • 空输出保护：如果解析不到任何步骤，返回 is_exe_success=False 的 ActionOutput，让主循环进入重试

这些容错机制体现了防御性编程的思想：永远不要假设 LLM 的输出是完美的，必须为每种可能的错误模式准备降级方案。

7.3 EnhancedShortTermMemory：相似度增强

短期记忆不只是简单的 FIFO 缓冲——它会计算新记忆与已有记忆的余弦相似度，当相似度超过阈值时，触发"增强"机制：

packages/dbgpt-core/src/dbgpt/agent/core/memory/short_term.py L22-L101（精简） 
class EnhancedShortTermMemory(ShortTermMemory[T]):
    """Enhanced short term memory."""

    def __init__(self, embeddings, executor, buffer_size=10,
                 enhance_similarity_threshold=0.7, enhance_threshold=3):
        self._embeddings = embeddings
        self.short_embeddings: List[List[float]] = []
        self.enhance_cnt: List[int] = [0 for _ in range(buffer_size)]
        self.enhance_memories: List[List[T]] = [[] for _ in range(buffer_size)]
        self.enhance_similarity_threshold = enhance_similarity_threshold
        self.enhance_threshold = enhance_threshold

    @mutable
    async def write(self, memory_fragment, now=None, op=WriteOperation.ADD):
        # 1. 计算新记忆片段的 embedding
        memory_fragment_embeddings = await blocking_func_to_async(
            self._executor,
            memory_fragment.calculate_current_embeddings,
            self._embeddings.embed_documents,
        )
        memory_fragment.update_embeddings(memory_fragment_embeddings)

        async with self._lock:
            # 2. 与已有记忆计算余弦相似度
            for idx, memory_embedding in enumerate(self.short_embeddings):
                similarity = await blocking_func_to_async(
                    self._executor, cosine_similarity,
                    memory_embedding, memory_fragment_embeddings,
                )
                sigmoid_prob = await blocking_func_to_async(
                    self._executor, sigmoid_function, similarity
                )
                # 3. 高相似度 + 概率命中 → 增强计数 +1
                if (sigmoid_prob >= self.enhance_similarity_threshold
                        and random.random() < sigmoid_prob):
                    self.enhance_cnt[idx] += 1
                    self.enhance_memories[idx].append(memory_fragment)

            # 4. 超过 enhance_threshold 的记忆转移到长期记忆
            discard_memories = await self.transfer_to_long_term(memory_fragment)
            if op == WriteOperation.ADD:
                self._fragments.append(memory_fragment)
                self.short_embeddings.append(memory_fragment_embeddings)
                await self.handle_overflow(self._fragments)
            return discard_memories
关键参数： buffer_size=10（短期记忆容量）、enhance_similarity_threshold=0.7（相似度阈值）、 enhance_threshold=3（增强次数阈值——被相似记忆"命中"3 次就会被转移到长期记忆）。 这套机制确保频繁出现的相似记忆会被自动"固化"为长期记忆，类似人类的记忆巩固过程。

7.5 AgentMemoryFragment：记忆的最小单元

AgentMemoryFragment 是记忆的基本单元，每条记忆片段包含以下字段：

  • observation：原始观察文本（如"用户问了 XXX，我用了 YYY 工具，结果是 ZZZ"）
  • memory_id：雪花算法生成的全局唯一 ID（可从中解析出创建时间戳）
  • importance：重要性评分（由 LLM 打分，范围 0-1，影响检索排序）
  • embeddings：文本的向量表示（用于相似度计算）
  • last_accessed_time：最后访问时间（用于时间衰减加权）
  • is_insight：是否为洞察记忆（从多次相似记忆中提取的抽象总结）
  • snapshot_path：快照文件路径（大文本记忆的完整内容存放在文件中）

ReActAgent 使用的是 StructuredAgentMemoryFragment——它把 ReAct 循环的结构化数据（question、thought、action、observation）序列化为 JSON 后存入 observation 字段。读取时再反序列化还原为对话消息格式，这样就能完整恢复历史推理过程。

7.6 GptsMemory：对话日志层

除了三层记忆模型，还有一套独立的 GptsMemory（core/memory/gpts/），它记录的是完整的对话日志，不做任何压缩或抽象：

  • GptsMessageMemory：每条消息的完整记录（发送者、接收者、内容、动作报告）
  • GptsPlansMemory：计划执行记录（每个子任务的分配、状态、结果）
  • default_gpts_memory.py：默认实现，使用 SQLite 持久化

GptsMemory 的主要用途是会话恢复——当用户中断后重新打开对话时，Agent 可以从 GptsMemory 中恢复完整的上下文。这与 HybridMemory 的"语义记忆"不同，GptsMemory 是"事件日志"。

7.4 LongTermRetriever：时间加权检索

长期记忆的检索基于 TimeWeightedEmbeddingRetriever，它同时考虑语义相关性和时间衰减：

packages/dbgpt-core/src/dbgpt/agent/core/memory/long_term.py L24-L111（精简） 
class LongTermRetriever(TimeWeightedEmbeddingRetriever):
    """长期记忆检索器，支持持久化。"""

    @mutable
    def _retrieve(self, query, filters=None) -> List[Chunk]:
        current_time = self.now

        # 1. 遍历 memory_stream，赋予默认显著性
        docs_and_scores = {}
        for doc in self.memory_stream:
            if _METADATA_BUFFER_IDX in doc.metadata:
                buffer_idx = doc.metadata[_METADATA_BUFFER_IDX]
                docs_and_scores[buffer_idx] = (doc, self.default_salience)

        # 2. 对语义相关的文档，更新显著性分数
        docs_and_scores.update(self.get_salient_docs(query, filters))

        # 3. 计算组合分数 = 语义相关性 × 时间衰减因子
        rescored_docs = [
            (doc, self._get_combined_score(doc, relevance, current_time))
            for doc, relevance in docs_and_scores.values()
        ]
        rescored_docs.sort(key=lambda x: x[1], reverse=True)

        # 4. 跳过被标记为 FORGET 或 MERGE 的记忆
        result = []
        for doc, _ in rescored_docs:
            if (retrieved_num < self._k
                    and doc.content.find(_FORGET_PLACEHOLDER) == -1
                    and doc.content.find(_MERGE_PLACEHOLDER) == -1):
                retrieved_num += 1
                # 更新最后访问时间（用于后续时间衰减计算）
                buffered_doc.metadata[_METADATA_LAST_ACCESSED_AT] = current_time
                result.append(buffered_doc)

        # 5. 持久化 memory_stream
        self._save_memory_stream()
        return result

class SkillResource(Resource[SkillParameters]):
    @classmethod
    def type(cls) -> ResourceType:
        return ResourceType.Skill

    def __init__(self, name, skill=None, skill_name=None, system_app=None, **kwargs):

        if skill:
            self._skill = skill
        elif skill_name:
            from dbgpt.agent.skill.manage import get_skill_manager
            skill_manager = get_skill_manager(system_app)
            self._skill = skill_manager.get_skill(name=skill_name)

    async def get_prompt(self, *, lang="en", question=None, ...):

        if not self._skill or not self._skill.prompt_template:
            return "", None
        prompt = self._skill.prompt_template.format(question=question or "")
        return prompt, None

    async def async_execute(self, *args, **kwargs):
        # Skill 本身不执行，具体行为由 actions/tools 实现
        raise NotImplementedError("SkillResource is readonly wrapper")
Skill vs Action 的关系：Skill 是更高层级的抽象——一个 Skill 可以包含多个 Action、多个 Tool、多个 Knowledge Source 以及一个完整的 prompt 模板。Skill 像是一个"Agent 模板"或"角色定义"，而 Action 是 Agent 在单步中执行的具体动作。

九、中间件机制

中间件系统（middleware/base.py）允许插件在 Agent 生命周期的各个阶段插入逻辑，而不需要修改 Agent 本身的代码——这是典型的 AOP（面向切面编程）思想。

class AgentMiddleware(ABC):
    """Agent 中间件基类。

    可以挂钩的 Agent 生命周期事件：
    - before_init / after_init              初始化前后
    - before_generate_reply / after_generate_reply  生成回复前后
    - before_thinking / after_thinking      思考前后
    - before_act / after_act                行动前后
    - modify_system_prompt                  修改系统提示词
    """

    def __init__(self):
        self._enabled: bool = True

    @property
    def enabled(self) -> bool:
        return self._enabled

    def enable(self):  self._enabled = True
    def disable(self): self._enabled = False

    async def before_init(self, agent, **kwargs) -> Optional[Dict]:
        return None

    async def after_init(self, agent, **kwargs) -> Optional[Dict]:
        return None

    async def before_generate_reply(self, agent, context, **kwargs):
        return None

    async def after_generate_reply(self, agent, context, reply_message, **kwargs):
        return None

    async def before_thinking(self, agent, context, **kwargs):
        return None

    async def after_thinking(self, agent, context, llm_reply, model_name, **kwargs):
        return None

    async def before_act(self, agent, context, message, **kwargs):
        return None

    async def after_act(self, agent, context, action_output, **kwargs):
        return None

    async def modify_system_prompt(self, agent, original_prompt, context=None):
        return original_prompt  # 默认不修改

9.2 MiddlewareManager：注册与执行

MiddlewareManager 负责中间件的注册、排序和批量执行。它对每个生命周期事件都有对应的 execute_* 方法，会遍历所有已注册且 enabled 的中间件：

packages/dbgpt-core/src/dbgpt/agent/middleware/base.py L211-L482（精简） 
class MiddlewareManager:
    """中间件管理器：注册、排序、批量执行。"""

    def __init__(self):
        self._middlewares: List[AgentMiddleware] = []
        self._state: Dict[str, Any] = {}

    def register(self, middleware) -> "MiddlewareManager":
        if middleware not in self._middlewares:
            self._middlewares.append(middleware)
        return self  # 链式调用

    def unregister(self, middleware) -> "MiddlewareManager":
        if middleware in self._middlewares:
            self._middlewares.remove(middleware)
        return self

    async def execute_before_thinking(self, agent, context, **kwargs) -> Dict:
        combined_state = {}
        for middleware in self._middlewares:
            if middleware.enabled:
                state = await middleware.before_thinking(agent, context, **kwargs)
                if state:
                    combined_state.update(state)
        return combined_state

    async def execute_modify_system_prompt(self, agent, original_prompt, context=None):
        # 链式修改：每个中间件看到的是上一个中间件的输出
        prompt = original_prompt
        for middleware in self._middlewares:
            if middleware.enabled:
                prompt = await middleware.modify_system_prompt(agent, prompt, context)
        return prompt
典型用途：
  • 安全审计中间件：在 before_act 中检查 LLM 输出是否包含危险操作（如 DROP TABLE）
  • 日志中间件：在 after_thinking 中记录 LLM 调用耗时和 token 消耗
  • 多语言中间件：在 modify_system_prompt 中注入语言偏好
  • Skill 集成中间件：在 before_generate_reply 中加载 Skill 所需的工具和知识

十、思考题

思考题 04-1：Agent 与 AWEL 的边界

给定一个"用户提问 → 查数据库 → 如果查不到就搜索网页 → 生成图表 → 返回"的任务，你会用 AWEL DAG 还是 Agent 来实现？为什么？如果两者混用，边界在哪里？提示：考虑"分支决策是否需要 LLM 推理"。

思考题 04-2：generate_reply 为什么是 @final

ConversableAgent.generate_reply 被标记为 @final，子类不能覆盖。但 ReActAgent 的行为明显和普通 Agent 不同，它是如何实现差异化逻辑的？提示：看 _a_init_reply_message、act、prepare_act_param 这些可覆盖的钩子方法。

思考题 04-3：ReAct 循环的终止条件

ReActAgent 的 run_mode = AgentRunMode.LOOP。如果 LLM 永远不输出 Action: terminate，循环会怎样结束？有哪几道保护机制？提示：看 max_retry_count、max_timeout、have_retry。

思考题 04-4：短期记忆的 enhance_threshold

EnhancedShortTermMemory 中 enhance_threshold=3 表示一条记忆被相似记忆"命中" 3 次后会被转移到长期记忆。如果把这个值设为 1，会发生什么？设为 100 呢？提示：考虑"记忆固化速度"与"噪声记忆"之间的权衡。

思考题 04-5：ResourcePack 的 prompt 拼接顺序

ResourcePack.get_prompt 遍历 self._resources（一个 dict）拼接 prompt。在 Python 3.7+ 中 dict 保持插入顺序，但如果资源列表很长，这个顺序是否可预测？如何确保数据库 Schema 总是在知识库 chunk 前面？提示：考虑 Resource.apply 方法。

思考题 04-6：中间件的执行顺序

MiddlewareManager.execute_modify_system_prompt 是链式修改——中间件 A 改完传给 B，B 改完传给 C。如果 A 和 B 都在 modify_system_prompt 中注入安全规则，最终 prompt 中的顺序由什么决定？如何控制中间件的执行优先级？提示：看 _middlewares 的注册顺序。
