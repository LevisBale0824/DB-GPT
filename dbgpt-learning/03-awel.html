一、AWEL 是什么 & 为什么需要它

翻开 AWEL 模块的 docstring，官方给出的定义非常精准：

awel/__init__.py L1-L9 
"""Agentic Workflow Expression Language (AWEL).

Agentic Workflow Expression Language(AWEL) is a set of intelligent agent workflow
expression language specially designed for large model application development. It
provides great functionality and flexibility. Through the AWEL API, you can focus on
the development of business logic for LLMs applications without paying attention to
cumbersome model and environment details.
"""
解读AWEL 是一套专为 LLM 应用设计的工作流表达式语言。 核心思想：把 LLM 应用拆成一个个可复用的 Operator（算子），用 DAG（有向无环图）把它们编排起来。

对比其他编排框架

框架 编排方式 优点 缺点
LangChain Chain 线性链式：A → B → C 简单直观 无法表达分支、并行、JOIN
AutoGen 对话式：Agent 之间互发消息 灵活、适合多轮讨论 执行路径不透明、难以可视化
AWEL (DB-GPT) DAG 算子图 可视化编排、支持分支/并行/JOIN/流式、可序列化为 JSON 概念较多、学习曲线稍陡
设计哲学：AWEL 把 LLM 应用当成数据流水线来处理。 每一步的输入和输出有明确的类型契约（泛型 IN/OUT）， 上游输出自动流入下游输入，框架负责调度。开发者只需关注每个算子的业务逻辑。

三、DAG 容器：节点、边、拓扑序

3.1 DAGNode —— 图的节点基类

所有能放进 DAG 的东西都继承 DAGNode。它实现了 Python 运算符重载， 让你用 << 和 >> 像写管道一样连接节点。

awel/dag/base.py L51-L133 
class DependencyMixin(ABC):
    """mixin：定义上下游关系的接口 + 运算符重载"""

    def __lshift__(self, nodes) -> DependencyType:
        # node << input_node  等价于  node.set_upstream(input_node)
        self.set_upstream(nodes)
        return nodes

    def __rshift__(self, nodes) -> DependencyType:
        # node >> next_node  等价于  node.set_downstream(next_node)
        self.set_downstream(nodes)
        return nodes

class DAGNode(DAGLifecycle, DependencyMixin, ViewMixin, ABC):
    def __init__(self, dag=None, node_id=None, node_name=None, ...):
        self._upstream: List[DAGNode] = []
        self._downstream: List[DAGNode] = []
        self._dag: Optional[DAG] = dag or DAGVar.get_current_dag()
        # 如果在 DAG 上下文里创建，自动把自己加进去
        if self._dag:
            self._dag._append_node(self)
L84__lshift__ 实现 node << input：等价于"把 input 设为我的上游"。
L101__rshift__ 实现 node >> next：等价于"把 next 设为我的下游"。
L325每个节点维护 _upstream 和 _downstream 两个列表，这就是 DAG 的边。
L327如果不在构造时传 dag，就从 DAGVar 获取当前上下文 DAG（类似 Airflow 的隐式 context manager）。

3.2 DAGVar —— 隐式 DAG 上下文

DAGVar 使用 threading.local（同步）和 contextvars.ContextVar（异步）来维护 "当前正在构建的 DAG"栈。这让你可以用 with DAG(...) 语法隐式注册节点：

awel/dag/base.py L136-L170 
class DAGVar:
    # 双轨制：线程本地（同步）+ ContextVar（异步）
    _thread_local = threading.local()
    _async_local: contextvars.ContextVar = contextvars.ContextVar(
        "current_dag_stack", default=deque()
    )

    @classmethod
    def enter_dag(cls, dag):
        is_async = _is_async_context()
        if is_async:
            stack = cls._async_local.get()
            stack.append(dag)
            cls._async_local.set(stack)
        else:
            if not hasattr(cls._thread_local, "current_dag_stack"):
                cls._thread_local.current_dag_stack = deque()
            cls._thread_local.current_dag_stack.append(dag)
解读DAGVar 是 AWEL 的"暗线"——你在 with DAG("my_dag"): 内创建的所有 Operator， 构造函数都会自动从 DAGVar 拿到当前 DAG 并把自己注册进去。这就是为什么写 AWEL 代码时不需要显式 dag.add_node()。 双轨制的原因：Python 的 threading.local 在 asyncio 中无法正确传播， 所以异步路径用 contextvars（PEP 567），同步路径用 threading.local。

3.3 DAG 类 —— 图容器

DAG 是所有节点的容器，负责维护 node_map、计算 root/leaf/trigger 节点、管理运行时上下文。

awel/dag/base.py L788-L872 
class DAG:
    def __init__(self, dag_id, resource_group=None, tags=None, ...):
        self._dag_id = dag_id
        self.node_map: Dict[str, DAGNode] = {}         # 所有节点
        self._root_nodes: List[DAGNode] = []            # 无上游的节点
        self._leaf_nodes: List[DAGNode] = []            # 无下游的节点
        self._trigger_nodes: List[DAGNode] = []         # TriggerOperator 节点
        self._event_loop_task_id_to_ctx: Dict[int, DAGContext] = {}  # 运行时上下文

    def _build(self) -> None:
        # 遍历 node_map，递归收集所有可达节点，
        # 然后按 upstream/downstream 是否为空分类
        from ..operators.common_operator import TriggerOperator
        nodes: Set[DAGNode] = set()
        for _, node in self.node_map.items():
            nodes = nodes.union(_get_nodes(node))
        self._root_nodes = list(set(filter(lambda x: not x.upstream, nodes)))
        self._leaf_nodes = list(set(filter(lambda x: not x.downstream, nodes)))
        self._trigger_nodes = list(
            set(filter(lambda x: isinstance(x, TriggerOperator), nodes))
        )
L806node_map 是 DAG 的核心数据结构：{"node_id": DAGNode}。
L862_build() 是惰性计算：当 root_nodes/leaf_nodes/trigger_nodes 第一次被访问时才遍历全图。这避免了每次加节点都重算。
L870trigger_nodes 的判定：类型为 TriggerOperator 的节点。这是 DAG 和 HTTP API 的连接点。
循环检测：AWEL 的 DAG 在构建时不做显式的环检测。它依赖拓扑序执行来隐式保证无环： DefaultWorkflowRunner._execute_node() 递归先跑完所有 upstream 再跑自己， 如果有环就会无限递归导致栈溢出。这是"用约定代替检查"的工程取舍。

四、Operator 类型家族

Operator 是 DAGNode 的可执行子类。BaseOperator 定义了 _do_run() 抽象方法， 各种子类用不同策略实现数据转换。

4.1 BaseOperator —— 所有算子的基类

awel/operators/base.py L161-L198 
class BaseOperator(DAGNode, ABC, Generic[OUT], metaclass=BaseOperatorMeta):
    streaming_operator: bool = False      # 是否产生流式输出
    incremental_output: bool = False      # 增量输出标记

    def __init__(self, task_id=None, task_name=None, dag=None,
                 runner=None, can_skip_in_branch=True, ...):
        # DAGNode.__init__ 自动注册到 dag.node_map
        super().__init__(node_id=task_id, node_name=task_name, dag=dag, **kwargs)
        if not runner:
            runner = DefaultWorkflowRunner()
        self._runner: WorkflowRunner = runner

    async def _run(self, dag_ctx: DAGContext, task_log_id: str) -> TaskOutput[OUT]:
        CURRENT_DAG_CONTEXT.set(dag_ctx)           # 设置 ContextVar
        await self._resolve_variables(dag_ctx)    # 解析变量占位符
        return await self._do_run(dag_ctx)        # 子类实现

    async def call(self, call_data=EMPTY_DATA, ...) -> OUT:
        # 同步调用入口
        out_ctx = await self._runner.execute_workflow(self, call_data)
        return out_ctx.current_task_context.task_output.output

    async def call_stream(self, call_data=EMPTY_DATA, ...) -> AsyncIterator[OUT]:
        # 流式调用入口
        out_ctx = await self._runner.execute_workflow(
            self, call_data, streaming_call=True
        )
        return out_ctx.current_task_context.task_output.output_stream
L167streaming_operator = False：类属性，子类设为 True 表示输出流。
L240_run() 是 Runner 调用的入口：先设 ContextVar，再解析变量，最后调 _do_run()。
L262call() 是对外的同步接口：执行完返回最终 output。
L316call_stream() 是流式接口：传 streaming_call=True 给 Runner。

4.2 BaseOperatorMeta —— 元类自动注入依赖

awel/operators/base.py L91-L158 
class BaseOperatorMeta(ABCMeta):
    """元类：在 __init__ 前自动注入 dag / runner / system_app / executor"""

    @classmethod
    def _apply_defaults(cls, func: F) -> F:
        def apply_defaults(self, *args, **kwargs):
            dag = kwargs.get("dag") or DAGVar.get_current_dag()
            system_app = kwargs.get("system_app") or DAGVar.get_current_system_app()
            executor = kwargs.get("executor") or DAGVar.get_executor()
            # ... 自动填充 kwargs 里缺失的字段
            if not kwargs.get("dag"): kwargs["dag"] = dag
            if not kwargs.get("runner"): kwargs["runner"] = default_runner
            return func(self, *args, **kwargs)
        return cast(F, apply_defaults)

    def __new__(cls, name, bases, namespace, **kwargs):
        new_cls = super().__new__(cls, name, bases, namespace, **kwargs)
        new_cls.__init__ = cls._apply_defaults(new_cls.__init__)  # 包装 __init__
        return new_cls
解读这个元类是 AWEL "少写样板代码"的关键。每个 Operator 子类的 __init__ 都被包了一层， 自动从 DAGVar 注入 dag / system_app / executor / runner。用户写 MapOperator(map_function=fn) 就够了， 不用传 dag、不用传 runner。

4.4 MapOperator 精读 —— 最常用的算子

awel/operators/common_operator.py L128-L198 
class MapOperator(BaseOperator, Generic[IN, OUT]):
    """1→1 转换算子，把 IN 映射为 OUT"""

    def __init__(self, map_function: Optional[MapFunc] = None, **kwargs):
        super().__init__(**kwargs)
        if map_function and not callable(map_function):
            raise ValueError("map_function must be callable")
        self.map_function = map_function

    async def _do_run(self, dag_ctx: DAGContext) -> TaskOutput[OUT]:
        curr_task_ctx = dag_ctx.current_task_context
        # MapOperator 只允许单上游
        if not curr_task_ctx.call_data and not curr_task_ctx.task_input.check_single_parent():
            raise ValueError("MapOperator expects single parent")

        map_function = self.map_function or self.map  # 有函数用函数，否则用子类重写的 map()
        # 对上游输出应用 map_function
        input_ctx = await curr_task_ctx.task_input.map(map_function)
        output = input_ctx.parent_outputs[0].task_output
        curr_task_ctx.set_task_output(output)
        return output

    async def map(self, input_value: IN) -> OUT:
        raise NotImplementedError  # 子类必须实现 map 或传 map_function
L135两种用法：① 传 map_function=lambda x: x.upper()；② 继承 MapOperator 重写 map()。
L172check_single_parent() 确保只有1个上游输出——这是 MapOperator 的数据契约。
L178self.map_function or self.map：优先用构造函数传入的函数，否则用子类重写的 map() 方法。
L190task_input.map(fn) 会自动处理流式/非流式：如果是流，map 会逐元素应用并返回新流。

4.5 BranchOperator 精读 —— 条件分支

awel/operators/common_operator.py L206-L314 
class BranchOperator(BaseOperator, Generic[IN, OUT]):
    """根据条件函数决定下游哪些节点被执行、哪些被跳过"""

    def __init__(self, branches: Optional[Dict[BranchFunc[IN], BranchTaskType]] = None, **kwargs):
        # branches = {条件函数: "下游节点名"}
        super().__init__(**kwargs)
        self._branches = branches

    async def _do_run(self, dag_ctx) -> TaskOutput[OUT]:
        curr_task_ctx = dag_ctx.current_task_context
        branches = self._branches or (await self.branches())

        # 对每个分支条件并行求值
        branch_func_tasks = []
        for func, node_name in branches.items():
            branch_func_tasks.append(
                curr_task_ctx.task_input.predicate_map(func, failed_value=None)
            )
        branch_input_ctxs = await asyncio.gather(*branch_func_tasks)

        # 条件为 False 的分支，其下游节点名被加入 skip_node_names
        skip_node_names = []
        for i, ctx in enumerate(branch_input_ctxs):
            if ctx.parent_outputs[0].task_output.is_none:
                skip_node_names.append(branch_nodes[i])
        curr_task_ctx.update_metadata("skip_node_names", skip_node_names)
        return parent_output  # 透传上游输出
L220branches 是一个字典：key 是条件函数 IN -> bool，value 是目标节点名。
L275对每个条件并行求值（asyncio.gather），返回 None 表示该分支为 False。
L298把条件为 False 的下游节点名写入 metadata 的 skip_node_names。
解读BranchOperator 本身不做转换，它只决定"后续谁该跳过"。Runner 在执行下游时读 skip_node_names 并跳过对应节点。

5.2 DefaultWorkflowRunner 精读

awel/runner/local_runner.py L43-L123 
async def execute_workflow(self, node, call_data=None, streaming_call=False,
                          exist_dag_ctx=None, dag_variables=None):
    # ① 从 end_node 反向遍历，收集所有节点
    job_manager = JobManager.build_from_end_node(node, call_data)

    # ② 创建（或复用）DAGContext
    if not exist_dag_ctx:
        node_outputs: Dict[str, TaskContext] = {}
        share_data: Dict[str, Any] = {}
        event_loop_task_id = id(asyncio.current_task())
    else:
        node_outputs = exist_dag_ctx._node_to_outputs
        share_data = exist_dag_ctx._share_data

    dag_ctx = DAGContext(event_loop_task_id, node_outputs, share_data,
                         streaming_call, ...)

    # ③ 生命周期钩子：before_dag_run
    await job_manager.before_dag_run()

    # ④ 递归执行（核心！）
    skip_node_ids: Set[str] = set()
    await self._execute_node(job_manager, node, dag_ctx,
                               node_outputs, skip_node_ids, system_app)

    # ⑤ 非流式调用才触发 after_dag_end
    if not streaming_call and node.dag and exist_dag_ctx is None:
        await node.dag._after_dag_end(dag_ctx._event_loop_task_id)

    return dag_ctx
L65JobManager.build_from_end_node 从叶子节点开始递归 node.upstream，收集整张图。
L66如果传了 exist_dag_ctx，说明这是一个子 DAG 嵌套调用，复用父 DAG 的上下文。
L102before_dag_run() 是生命周期钩子，所有节点的 before 钩子并行执行。
L117关键：流式调用不触发 after_dag_end——因为流还没消费完，DAG 不能结束。

5.3 _execute_node —— 拓扑序递归

awel/runner/local_runner.py L125-L212 
async def _execute_node(self, job_manager, node, dag_ctx, node_outputs, skip_node_ids, ...):
    # 幂等：如果该节点已经跑过，直接返回
    if node.node_id in node_outputs:
        return

    # 先递归跑完所有上游（深度优先 = 拓扑序）
    for upstream_node in node.upstream:
        await self._execute_node(job_manager, upstream_node, dag_ctx, ...)

    # 收集上游输出，构建 InputContext
    inputs = [node_outputs[up.node_id] for up in node.upstream]
    input_ctx = DefaultInputContext(inputs)

    # 创建当前节点的 TaskContext
    task_ctx = DefaultTaskContext(node.node_id, TaskState.INIT, ...)
    if current_call_data:
        task_ctx.set_call_data(current_call_data)
    task_ctx.set_task_input(input_ctx)
    dag_ctx.set_current_task_context(task_ctx)
    task_ctx.set_current_state(TaskState.RUNNING)

    # 检查跳过
    if node.node_id in skip_node_ids:
        task_ctx.set_current_state(TaskState.SKIP)
        task_ctx.set_task_output(SimpleTaskOutput(SKIP_DATA))
        node_outputs[node.node_id] = task_ctx
        return

    # 执行！
    await node._run(dag_ctx, task_ctx.log_id)
    node_outputs[node.node_id] = dag_ctx.current_task_context
    task_ctx.set_current_state(TaskState.SUCCESS)

    # 如果是 BranchOperator，把跳过的下游加入 skip_node_ids
    if isinstance(node, BranchOperator):
        skip_nodes = task_ctx.metadata.get("skip_node_names", [])
        _skip_current_downstream_by_node_name(node, skip_nodes, skip_node_ids)
L135幂等检查：用 node_outputs 字典充当 visited 集合，防止节点重复执行。
L141拓扑序：先递归跑完所有 upstream，再跑自己。这天然保证了执行顺序与 DAG 拓扑序一致。
L169跳过逻辑：如果节点在 skip_node_ids 中，输出 SKIP_DATA 并直接返回。下游的 MapOperator 看到 SKIP_DATA 可以继续跳过。
L200BranchOperator 特殊处理：执行完后读取 metadata 中的 skip_node_names，调用 _skip_current_downstream_by_node_name 把下游该跳的节点 ID 加入 skip 集合。
L206异常处理：try/except 捕获异常，设为 FAILED 状态后重新抛出。

5.4 分支跳过的传播规则

_skip_downstream_by_id 实现了一个精巧的多父节点跳过规则：

awel/runner/local_runner.py L241-L257 
def _skip_downstream_by_id(node: BaseOperator, skip_node_ids: Set[str]):
    if not node.can_skip_in_branch():
        return  # 当前节点不允许跳过（如 BranchJoinOperator），停止传播

    # 多父节点（如 JoinOperator）：只有所有上游都被跳过，才跳过自己
    for parent in node.upstream:
        if isinstance(parent, BaseOperator) and parent.node_id not in skip_node_ids:
            return  # 还有父节点没被跳过，当前节点不能跳过

    skip_node_ids.add(node.node_id)
    for child in node.downstream:
        _skip_downstream_by_id(child, skip_node_ids)
L242can_skip_in_branch() 默认 True，但 BranchJoinOperator 设为 False —— 合并点不能被跳过，否则会丢失另一条分支的数据。
L251多父节点规则：只有所有上游都被跳过，当前节点才跳过。这保证了 JoinOperator 只要有一条分支还活着就不会被跳过。

六、流式输出实现

LLM 应用中流式输出是刚需。AWEL 用 TaskOutput 的双形态设计来统一处理流和非流。

6.1 TaskOutput 双形态

awel/task/task_impl.py L61-L99, L152-L194 
class SimpleTaskOutput(TaskOutput[T]):
    """非流式输出：包装一个标量值"""
    def __init__(self, data=EMPTY_DATA):
        self._data = data

    @property
    def output(self) -> T:
        return cast(T, self._data)

    async def map(self, map_func) -> TaskOutput[OUT]:
        # 标量 map：直接把函数应用到值上
        out = await self._apply_func(map_func)
        return SimpleTaskOutput(out)

    async def streamify(self, transform_func) -> TaskOutput[OUT]:
        # 标量 → 流：用 transform_func 生成 AsyncIterator
        out = await self._apply_func(transform_func)
        return SimpleStreamTaskOutput(out)


class SimpleStreamTaskOutput(TaskOutput[T]):
    """流式输出：包装一个 AsyncIterator"""
    def __init__(self, data=EMPTY_DATA):
        self._data = data

    @property
    def is_stream(self) -> bool:
        return True

    @property
    def output_stream(self) -> AsyncIterator[T]:
        return cast(AsyncIterator[T], self._data)

    async def map(self, map_func) -> TaskOutput[OUT]:
        # 流 map：逐元素应用，返回新流
        async def new_iter():
            async for out in self.output_stream:
                yield await map_func(out) if asyncio.iscoroutinefunction(map_func) else map_func(out)
        return SimpleStreamTaskOutput(new_iter())

    async def reduce(self, reduce_func) -> TaskOutput[OUT]:
        # 流 → 标量：遍历整个流累积
        out = await _reduce_stream(self.output_stream, reduce_func)
        return SimpleTaskOutput(out)
解读这是 AWEL 流式系统的核心设计：SimpleTaskOutput（标量）和 SimpleStreamTaskOutput（流）都实现相同的 TaskOutput 接口。 map() 方法在两种形态下表现不同：标量直接应用函数返回新标量，流则逐元素应用返回新流。
L137streamify() 是标量→流的转换器。
L212流的 map() 返回一个 lazy 的 new_iter() generator——只有在最终消费时才真正执行，这就是流式惰性求值。

6.2 三种流式算子

awel/operators/stream_operator.py L11-L118 
class StreamifyAbsOperator(BaseOperator[OUT], ABC, Generic[IN, OUT]):
    """标量 IN → 流 AsyncIterator[OUT]"""
    streaming_operator = True  # 标记为流式算子

    async def _do_run(self, dag_ctx):
        # 对上游输出调用 streamify() 方法
        output = await curr_task_ctx.task_input.parent_outputs[0].task_output.streamify(
            self.streamify
        )
        curr_task_ctx.set_task_output(output)

    async def streamify(self, input_value: IN) -> AsyncIterator[OUT]:
        raise NotImplementedError


class UnstreamifyAbsOperator(BaseOperator[OUT], Generic[IN, OUT]):
    """流 AsyncIterator[IN] → 标量 OUT"""

    async def _do_run(self, dag_ctx):
        output = await curr_task_ctx.task_input.parent_outputs[0].task_output.unstreamify(
            self.unstreamify
        )
        curr_task_ctx.set_task_output(output)


class TransformStreamAbsOperator(BaseOperator[OUT], Generic[IN, OUT]):
    """流 → 流转换"""
    streaming_operator = True

    async def _do_run(self, dag_ctx):
        output = await curr_task_ctx.task_input.parent_outputs[0].task_output.transform_stream(
            self.transform_stream
        )
        curr_task_ctx.set_task_output(output)
L14streaming_operator = True 告诉框架这个算子输出是流。
L28StreamifyAbsOperator 调 task_output.streamify(self.streamify)，底层就是 SimpleTaskOutput.streamify → 返回 SimpleStreamTaskOutput。
L52UnstreamifyAbsOperator 反过来：流→标量，调用 unstreamify()。
L82TransformStreamAbsOperator 是流→流，调用 transform_stream()，对应 SimpleStreamTaskOutput 的同名方法。
解读三种流式算子 + TaskOutput 的双形态设计，让流可以在 DAG 中自由传递、转换和终止，而不需要每个算子关心"上游是不是流"。

7.4 _trigger_dag —— HTTP 请求如何触发 DAG

async def _trigger_dag(body, dag, streaming_response=False, ...):
    from fastapi.responses import StreamingResponse

    leaf_nodes = dag.leaf_nodes
    if len(leaf_nodes) != 1:
        raise ValueError("HttpTrigger just support one leaf node in dag")

class TaskContext(ABC, Generic[T]):
    """运行时每个节点的上下文：持有 task_input 和 task_output"""
    @property
    @abstractmethod
    def task_id(self) -> str: ...
    @property
    @abstractmethod
    def task_input(self) -> InputContext: ...
    @property
    @abstractmethod
    def task_output(self) -> TaskOutput[T]: ...
    @property
    def call_data(self) -> Optional[Dict]:
        return self.metadata.get("call_data")


class InputContext(ABC):
    """所有上游输出的集合，提供 map/reduce/filter 操作"""
    @property
    @abstractmethod
    def parent_outputs(self) -> List[TaskContext]: ...

    def check_single_parent(self) -> bool:
        return len(self.parent_outputs) == 1

    def check_stream(self) -> bool:
        for out in self.parent_outputs:
            if not (out.task_output and out.task_output.is_stream):
                return False
        return True
解读TaskContext 是运行时每个节点的"工作台"：input 来自上游，output 是自己的产出。 InputContext 包装了所有上游的 TaskContext 列表，提供 map() / map_all() / reduce() / filter() / predicate_map() 等操作。 check_single_parent() 和 check_stream() 是数据契约校验方法。

十、思考题

Q1 — DAG 编译

AWEL 没有 dag.compile() 方法，那么它是如何保证执行顺序正确的？ 如果用户在代码中创建了环（A → B → A），会发生什么？

参考答案

AWEL 用 Runner 的递归拓扑遍历代替显式编译：先递归跑完所有 upstream 再跑自己。 如果存在环，_execute_node 会无限递归导致 RecursionError。 node_outputs 的幂等检查（L135: if node.node_id in node_outputs: return） 实际上也会让第二次访问直接返回，但此时输出可能尚未填充，导致 KeyError。AWEL 依赖约定而非检查来保证无环。

Q2 — 流式传递

一个 DAG 中同时有 StreamifyAbsOperator（标量→流）和 MapOperator（1→1）， 当流经过 MapOperator 时，MapOperator 的行为是什么？最终输出是流还是标量？

参考答案

MapOperator 调用 task_input.map(map_function)。当上游是 SimpleStreamTaskOutput（流）时， 其 map() 方法会返回一个新的 SimpleStreamTaskOutput（逐元素应用函数的 lazy generator）。 所以 MapOperator 透传流：输出仍然是流。流-非流由 TaskOutput 的多态自动处理。

Q3 — 循环检测

如果要在 AWEL 中加入显式的环检测，应该在哪个类的哪个方法里加？用什么算法？

参考答案

在 DAG._build() 方法（L862）中加最合适，因为它已经在遍历全图。 算法用DFS 三色标记法（白/灰/黑）：遇到灰色节点说明有回边=环。 或者用 Kahn 算法（拓扑排序），如果排序后剩余节点数 < 总节点数，说明有环。

Q4 — Operator 数据契约

MapOperator 要求 check_single_parent() 为 True， JoinOperator 没有这个要求。BranchOperator 也有这个要求。 为什么 BranchOperator 要求单上游？

参考答案

BranchOperator 的语义是"根据输入决定走哪条分支"。如果有多个上游， 就无法确定用哪个输入来做分支判断。BranchOperator 把自己当作一个"路由器"， 只接受一个输入流，然后把它分发到不同下游。

Q5 — 元类注入

BaseOperatorMeta._apply_defaults 会在每次 __init__ 时自动从 DAGVar 注入 dag/runner/system_app。 如果你在 with DAG("x"): 块外面创建 Operator，会发生什么？

参考答案

DAGVar.get_current_dag() 返回 None，所以 self._dag = None。 节点不会被注册到任何 DAG，但它仍然可以独立运行（通过 operator.call() 直接调用）。 Runner 会从 node.dag 取 DAG 信息，如果为 None 就跳过 DAG 级操作（如 before_dag_run）。

Q6 — 分支跳过传播

BranchOperator 下游有两条分支，最终汇入一个 BranchJoinOperator。 如果只有一条分支被跳过，BranchJoinOperator 会被跳过吗？为什么？

参考答案

不会。因为 BranchJoinOperator 的 can_skip_in_branch 默认为 False （在 __init__ 中 can_skip_in_branch=False）。 _skip_downstream_by_id 在 L242 检查这个标记，为 False 就直接 return 停止传播。 此外，即使允许跳过，多父节点规则也会阻止：只要有一个上游没被跳过，当前节点就不会被跳过。

Q7 — HttpTrigger 约束

_trigger_dag() 函数要求 len(leaf_nodes) == 1。 如果一个 DAG 有两个叶子节点（两条独立链路），HttpTrigger 为什么不支持？

参考答案

因为 HTTP 是请求-响应模型：一个请求只能返回一个响应。如果有多个叶子节点， 就不知道该把哪个叶子节点的输出作为 HTTP 响应体。如果你确实需要并行两条链路， 应该用一个 JoinOperator 把它们合并成一个叶子节点。

Q8 — DAGContext 隔离

DAGContext 用 event_loop_task_id = id(asyncio.current_task()) 作为唯一键。 为什么不用线程 ID？为什么用 asyncio.current_task() 的 id？

参考答案

因为 AWEL 全异步执行。一个 event loop 可能同时处理多个 HTTP 请求（多个 asyncio Task）， 它们共享同一个线程但属于不同的 Task。id(asyncio.current_task()) 能区分同一个 event loop 中不同的并发 Task， 从而实现请求级隔离。用线程 ID 会导致同一 event loop 中的所有并发请求共享同一个 DAGContext，造成数据串扰。

本章总结

AWEL 的五层抽象回顾：
  • DAG（dag/base.py）：有向无环图容器，维护节点、边、root/leaf/trigger 分类，提供 ContextManager 语法。
  • Operator（operators/）：可执行算子，10+ 子类覆盖 Map/Branch/Join/Streamify/TransformStream 等场景。BaseOperatorMeta 元类自动注入依赖。
  • Trigger（trigger/）：HTTPTrigger 把 DAG 变成 FastAPI 路由，一个 DAG = 一个 REST API。
  • Task（task/）：TaskOutput 双形态（SimpleTaskOutput / SimpleStreamTaskOutput）统一流式与非流式，InputContext 提供函数式数据操作。
  • Flow（flow/）：ViewMetadata 描述算子的前端展示信息，支持可视化拖拽编排。

理解了 AWEL，你就理解了 DB-GPT 的运行时。Agent 系统的每一步推理、RAG 的每次检索、 Chat Scene 的每次对话，底层都是 AWEL DAG 在驱动。下一章我们看 Agent 如何在 AWEL 之上构建自主决策能力。
