09 · Serve 服务层 Serve

把 core 抽象封装为 REST API —— DB-GPT 如何用"模板化四层"批量生产 17+ 个 HTTP 服务模块

一句话定位：Serve 层是 DB-GPT 的"产品化外壳"。 dbgpt-core 提供能力（Agent / RAG / Model / Datasource）， dbgpt_serve 把这些能力包装成可被前端/外部系统调用的 REST API。 每个 Serve 模块都是同一套模板的实例：DAO + Service + Endpoint + Schema 四层， 配上 FastAPI 路由、Pydantic 模型、SQLAlchemy 表、统一错误处理。 全仓库 17+ 个 Serve 模块，90% 的代码结构是复制粘贴再改业务。

一、Serve 层的职责与定位

打开 packages/dbgpt-serve 的目录结构，你会看到 17 个名字相似的子包： agent / conversation / datasource / evaluate / feedback / file / flow / libro / model / prompt / rag / scheduled_task / connector / dbgpts / evaluate / feedback。 它们是 DB-GPT 的"服务化封装层"。

换句话说：dbgpt_serve 不实现"AI 能力"，它只负责把已有的 AI 能力 HTTP 化。 这是它与 dbgpt-app（早期承担一部分 HTTP 路由）的关键区别—— 长期方向是所有 REST API 都搬到 serve 层，dbgpt-app 只做启动和组装。

三、CRUD 四层架构：DAO + Service + Endpoint + Schema

Serve 层最显著的特征是高度的模板化。打开任意一个 Serve 子模块， 你都能看到同样的四个文件，做几乎同样的事。下面用 conversation 做例子展示这个套路。

flowchart TD REQ["HTTP 请求\nPOST /api/v1/serve/conversation/query_page"] EP["Endpoint 层 api/endpoints.py\n@router.post\nFastAPI 路由声明 / 参数校验、鉴权\n统一返回 Result.succ / Result.failed"] SVC["Service 层 service/service.py\n业务逻辑编排\n跨模块调用\n不直接碰 SQLAlchemy session"] DAO["DAO 层 models/models.py\n所有 SQLAlchemy 操作集中在这里\nEntity ↔ Request / Response 转换"] SCH["Schema 层 api/schemas.py + models/models.py\nRequest / Response 用 Pydantic\nEntity 用 SQLAlchemy 声明式建表"] REQ --> EP --> SVC --> DAO --> SCH

这四层分别对应 4 个固定文件： api/endpoints.py（路由）、 api/schemas.py（Pydantic）、 service/service.py（业务）、 models/models.py（SQLAlchemy + DAO）。 另有两个辅助文件：config.py（常量 + ServeConfig）、 serve.py（生命周期管理，下文详述）。

core 提供的公共基类

四层架构的"骨架"来自 dbgpt_serve/core/。它导出了几个关键工具：

# dbgpt_serve/core/__init__.py
from dbgpt_serve.core.config import BaseServeConfig as BaseServeConfig
from dbgpt_serve.core.schemas import (
    ResourceParameters, ResourceTypes, Result,
    add_exception_handler,
)
from dbgpt_serve.core.serve import BaseServe as BaseServe
from dbgpt_serve.core.service import BaseService as BaseService

# 异步包装阻塞函数（DAO 多为同步 SQLAlchemy 调用）
async def blocking_func_to_async(system_app, func, *args, **kwargs):
    executor = DefaultExecutorFactory.get_instance(system_app).create()
    return await _blocking_func_to_async(executor, func, *args, **kwargs)

BaseService 定义了 Service 层的契约：必须提供 dao 和 config 属性， 默认实现 create() 直接转给 DAO：

# dbgpt_serve/core/service.py
class BaseService(BaseComponent, Generic[T, REQ, RES], ABC):
    name = "dbgpt_serve_base_service"
    _dag_manager: Optional[DAGManager] = None

    @property
    @abstractmethod
    def dao(self) -> BaseDao[T, REQ, RES]: ...

    @property
    @abstractmethod
    def config(self) -> BaseServeConfig: ...

    def create(self, request: REQ) -> RES:
        return self.dao.create(request)  # 默认实现

    def before_start(self):
        self._dag_manager = DAGManager.get_instance(self._system_app)

注意 BaseService 还引入了 DAGManager——这意味着所有 Service 都能拿到全局 DAG 管理器， 这对 flow 模块尤其重要。

四、通用 API 模式：分页、错误、鉴权、OpenAPI

4.1 统一返回 Result

所有 Endpoint 返回值都被 Result[T] 包装（来自 dbgpt.core.schema.api）：

# 典型用法
@router.post("/query", response_model=Result[ServerResponse])
async def query(request: ServeRequest, service: Service = Depends(get_service)):
    return Result.succ(service.get(request))

# 失败时
return Result.failed(msg=str(e))
return Result.fail("Invalid JSON format")  # 简版

Result 的 JSON 形态是 {"success": bool, "err_code": str, "err_msg": str, "data": T}， 方便前端统一处理成功/失败。

4.2 统一异常处理

dbgpt_serve/core/schemas.py 提供了三类异常处理器，由 add_exception_handler(app) 一次性挂到 FastAPI 上：

async def validation_exception_handler(request, exc: RequestValidationError):
    # Pydantic 校验失败 → 400 + E0001
    message = ""
    for error in exc.errors():
        loc = ".".join(map(str, error.get("loc")))
        message += loc + ":" + error.get("msg") + ";"
    return JSONResponse(status_code=400, content=Result.failed(msg=message, err_code="E0001").to_dict())

async def http_exception_handler(request, exc: HTTPException):
    # FastAPI 抛出的 HTTPException（如 404、403）
    return JSONResponse(status_code=exc.status_code, content=Result.failed(...).to_dict())

async def common_exception_handler(request, exc: Exception):
    # 兜底：未知异常 → 400 + E0003
    return JSONResponse(status_code=400, content=Result.failed(msg=str(exc), err_code="E0003").to_dict())

def add_exception_handler(app):
    app.add_exception_handler(RequestValidationError, validation_exception_handler)
    app.add_exception_handler(HTTPException, http_exception_handler)
    app.add_exception_handler(Exception, common_exception_handler)
错误码约定：E0001 = 参数校验失败； E0003 = 未捕获异常；HTTP 状态码会被 JSONResponse(status_code=...) 覆盖。 注意：这种把所有异常都变成 400 的做法对前端简单，但牺牲了 RESTful 语义，调试时需要看 err_code 区分。

4.3 鉴权：Bearer Token + 配置开关

几乎所有 Serve 模块的 endpoints.py 都有一段几乎一字不差的 check_api_key 函数：

# conversation/api/endpoints.py（所有 Serve 模块复制的模板）
get_bearer_token = HTTPBearer(auto_error=False)

@cache
def _parse_api_keys(api_keys: str) -> List[str]:
    if not api_keys: return []
    return [key.strip() for key in api_keys.split(",")]

async def check_api_key(
    auth: Optional[HTTPAuthorizationCredentials] = Depends(get_bearer_token),
    request: Request = None,
    service: Service = Depends(get_service),
) -> Optional[str]:
    # /api/v1 路径下放行（兼容旧接口）
    if request.url.path.startswith("/api/v1"): return None
    if service.config.api_keys:
        api_keys = _parse_api_keys(service.config.api_keys)
        if auth is None or (token := auth.credentials) not in api_keys:
            raise HTTPException(status_code=401, detail={"error": {...}})
        return token
    return None  # api_keys 未设置，放行

@router.post("/query_page", dependencies=[Depends(check_api_key)])
async def query_page(...): ...

鉴权开关来自 ServeConfig.api_keys（逗号分隔的 key 列表），可以在 .env 或 dbgpt.serve.conversation.api_keys 配置项里设置。未设置时默认放行， 这是为了本地开发方便；生产环境强烈建议配置。

4.4 分页：PaginationResult 约定

分页查询统一返回 PaginationResult[T]：

@router.post("/query_page", response_model=Result[PaginationResult[ServerResponse]])
async def query_page(
    request: ServeRequest,
    page: Optional[int] = Query(default=1, description="current page"),
    page_size: Optional[int] = Query(default=10, description="page size"),
    service: Service = Depends(get_service),
):
    return Result.succ(service.get_list_by_page(request, page, page_size))

DAO 层实现分页的典型写法（conversation 模块）：

# conversation/models/models.py
def get_conv_by_page(self, req, page, page_size) -> PaginationResult[ServerResponse]:
    with self.session(commit=False) as session:
        query = self._create_query_object(session, req)
        query = query.order_by(ServeEntity.gmt_created.desc())
        total_count = query.count()
        items = query.offset((page - 1) * page_size).limit(page_size)
        total_pages = (total_count + page_size - 1) // page_size
        result_items = [self.to_response(it) for it in items]
        return PaginationResult(
            items=result_items, total_count=total_count,
            total_pages=total_pages, page=page, page_size=page_size,
        )

4.5 OpenAPI 文档：FastAPI 自动生成

因为 Schema 用的是 Pydantic，FastAPI 会自动把所有 Endpoint 注册到 /docs（Swagger UI） 和 /redoc。response_model=Result[ServerResponse] 这种写法让返回结构也在文档里可见。 这是 Serve 层选择 FastAPI 而不是 Flask 的核心收益之一。

五、Serve 生命周期：init_app → on_init → before_start → after_start

每个 Serve 模块都是 BaseServe 的子类，它的生命周期由 SystemApp 驱动：

flowchart LR INIT["① init_app(system_app)\n- include_router\n- 注册 Service 组件\n- 读取 ServeConfig\n- 设置 _app_has_initiated"] ONINIT["② on_init()\n- import Model\n- 预加载依赖\n- 尚未可访问别的 Serve 组件"] BEFORE["③ before_start()\n- create DB tables\n- 初始化 storage\n- 注册 Operator\n- 构建 VariablesProvider"] AFTER["④ after_start()\n- flow 特有：\n load_dag_from_db\n load_dag_from_dbgpts"] INIT --> ONINIT --> BEFORE --> AFTER

以 conversation 为例，init_app 挂载路由：

# conversation/serve.py
class Serve(BaseServe):
    name = SERVE_APP_NAME  # "dbgpt_serve_conversation"

    def __init__(self, system_app, config=None,
                 api_prefix="/api/v1/serve/conversation",  # 默认前缀
                 api_tags=None, db_url_or_db=None, try_create_tables=False):
        if api_tags is None: api_tags = [SERVE_APP_NAME_HUMP]
        super().__init__(system_app, api_prefix, api_tags, db_url_or_db, try_create_tables)
        self._conv_storage = None
        self._message_storage = None

    def init_app(self, system_app):
        if self._app_has_initiated: return
        self._system_app = system_app
        self._system_app.app.include_router(
            router, prefix=self._api_prefix, tags=self._api_tags
        )
        self._config = self._config or ServeConfig.from_app_config(
            system_app.config, SERVE_CONFIG_KEY_PREFIX
        )
        init_endpoints(self._system_app, self._config)
        self._app_has_initiated = True

before_start 里做实质性的初始化（创建表 + 构建 storage）：

# conversation/serve.py
def before_start(self):
    from dbgpt.storage.chat_history.chat_history_db import (
        ChatHistoryEntity, ChatHistoryMessageEntity,
    )
    from dbgpt.storage.metadata.db_storage import SQLAlchemyStorage
    from dbgpt.util.serialization.json_serialization import JsonSerializer

    self._db_manager = self.create_or_get_db_manager()

    # 两个 storage 分别存"会话列表"和"消息明细"
    self._conv_storage = SQLAlchemyStorage(
        self._db_manager, ChatHistoryEntity,
        DBStorageConversationItemAdapter(), JsonSerializer(),
    )
    self._message_storage = SQLAlchemyStorage(
        self._db_manager, ChatHistoryMessageEntity,
        DBMessageStorageItemAdapter(), JsonSerializer(),
    )

create_or_get_db_manager 来自 BaseServe，它优先使用系统级 UnifiedDBManagerFactory（保证所有 Serve 共享同一个 SQLite/MySQL 实例）， 回退到自身 db_url_or_db。try_create_tables=True 时会调用 init_db.create_all()——这就是 Serve 的"轻量建表"机制。

六、精读 conversation 模块：聊天会话管理

conversation 模块是最常用的 Serve 模块——每次用户打开聊天页面，前端都要先调它的 /list 拉历史会话、调 /new 新建会话、调 /messages/history 拉某会话的全部消息。

6.1 数据模型

conversation 没有自己建表，而是复用 dbgpt/storage/chat_history/chat_history_db.py 里的 ChatHistoryEntity 和 ChatHistoryMessageEntity。在 conversation/models/models.py 里直接 import as 成 ServeEntity：

# conversation/models/models.py
from dbgpt.storage.chat_history.chat_history_db import ChatHistoryEntity as ServeEntity
from dbgpt.storage.chat_history.chat_history_db import ChatHistoryMessageEntity

class ServeDao(BaseDao[ServeEntity, ServeRequest, ServerResponse]):
    def from_request(self, request):
        request_dict = request.to_dict() if isinstance(request, ServeRequest) else request
        return ServeEntity(**request_dict)

    def to_response(self, entity) -> ServerResponse:
        gmt_created = entity.gmt_created.strftime("%Y-%m-%d %H:%M:%S") if entity.gmt_created else None
        return ServerResponse(
            app_code=entity.app_code, conv_uid=entity.conv_uid,
            user_input=entity.summary, # 注意：summary → user_input
            chat_mode=entity.chat_mode, sys_code=entity.sys_code,
            gmt_created=gmt_created, gmt_modified=...,
        )
命名映射的坑：数据库里叫 summary，API 返回叫 user_input。 这种字段重命名大量出现在 DAO 的 to_response 里——是为了兼容旧的前端字段约定。 阅读 Serve 代码时不要假设字段名就是数据库列名，一定要看 DAO 转换函数。

6.2 Service 层：把 storage 和 DAO 串起来

conversation 的 Service 有个特别之处：它同时持有 DAO 和两个 Storage， DAO 管"会话列表"，Storage 管"消息流"。这是因为消息流需要走 StorageConversation 这个领域对象而不是裸 SQL：

# conversation/service/service.py
class Service(BaseService[ServeEntity, ServeRequest, ServerResponse]):
    name = SERVE_SERVICE_COMPONENT_NAME

    @property
    def conv_storage(self) -> StorageInterface:
        if self._storage: return self._storage
        from ..serve import Serve
        return Serve.call_on_current_serve(
            self._system_app, lambda serve: serve.conv_storage
        )

    def create_storage_conv(self, request, load_message=True) -> StorageConversation:
        return StorageConversation(
            conv_uid=request.conv_uid, chat_mode=request.chat_mode,
            user_name=request.user_name, sys_code=request.sys_code,
            conv_storage=self.conv_storage,
            message_storage=self.message_storage,
            load_message=load_message,
        )

    def delete(self, request):
        conv = self.create_storage_conv(request)
        conv.delete()  # 同时删会话和消息

    def clear(self, request):
        conv = self.create_storage_conv(request)
        conv.clear()  # 只删消息保留会话

get_history_messages 是 conversation Service 里最复杂的方法，它要： (1) 从 storage 读出所有消息； (2) 调 _append_view_messages 合并 view 消息； (3) 调 feedback 模块拿每条消息的反馈； (4) 调 file 模块的 replace_uri 把文件 URI 换成可访问 URL。

# conversation/service/service.py
def get_history_messages(self, request) -> List[MessageVo]:
    from ...file.serve import Serve as FileServe
    file_serve = FileServe.get_instance(self.system_app)

    conv: StorageConversation = self.create_storage_conv(request)
    messages = _append_view_messages(conv.messages)

    # 跨模块：拿 feedback
    feedback_service = get_service()
    feedbacks = feedback_service.list_conv_feedbacks(conv_uid=request.conv_uid)
    fb_map = {fb.message_id: fb.to_dict() for fb in feedbacks}

    result = []
    for msg in messages:
        feedback = fb_map.get(str(msg.round_index), {})
        result.append(MessageVo(
            role=msg.type,
            context=vis_name_change(msg.get_view_markdown_text(file_serve.replace_uri)),
            order=msg.round_index,
            model_name=self.config.default_model,
            feedback=feedback,
        ))
    return result
跨模块调用范式：Service 层通过两种方式获取别的 Serve： (1) Serve.call_on_current_serve(system_app, lambda s: s.xxx)——拿 Serve 组件实例； (2) Service.get_instance(system_app)——拿 Service 组件。 这两种都依赖 SystemApp 的组件注册表，是 DB-GPT 的 IoC 容器。

6.3 Endpoint：导出会话为 JSON 文件

conversation 提供了 /export_messages 端点把全部会话打包成 JSON 下载—— 这是 Serve 层"小工具型"端点的典型写法（流式响应、Content-Disposition 头）：

@router.get("/export_messages", dependencies=[Depends(check_api_key)])
async def export_all_messages(user_name=None, user_id=None, sys_code=None,
                            format: Literal["file", "json"] = Query("file"),
                            service: Service = Depends(get_service)):
    request = ServeRequest(user_name=user_name or user_id, sys_code=sys_code)
    page, page_size, all_conversations = 1, 100, []
    while True:  # 分页把所有会话捞完
        pr = service.get_list_by_page(request, page, page_size)
        all_conversations.extend(pr.items)
        if page >= pr.total_pages: break
        page += 1
    result = {"user_name": ..., "conversations": []}
    for conv in all_conversations:
        messages = service.get_history_messages(ServeRequest(conv_uid=conv.conv_uid))
        result["conversations"].append({"conv_uid": conv.conv_uid, "messages": ...})

    if format == "json":
        return JSONResponse(content=result)
    return StreamingResponse(
        io.BytesIO(json.dumps(result).encode("utf-8")),
        media_type="application/file",
        headers={"Content-Disposition": f"attachment;filename=....json"},
    )

6.4 Operator：把 conversation 接入 AWEL

conversation 还提供了 ServePreChatHistoryLoadOperator——这是 Serve 模块与 AWEL 工作流的衔接点。这个 Operator 让 DAG 中的对话能自动加载历史消息：

# conversation/operators.py
class ServePreChatHistoryLoadOperator(PreChatHistoryLoadOperator):
    def __init__(self, storage=None, message_storage=None, **kwargs):
        super().__init__(storage, message_storage,
                            use_in_memory_storage_if_not_set=False, **kwargs)

    @property
    def storage(self):
        if self._storage: return self._storage
        # 反查当前 Serve 组件，拿 conv_storage
        storage = Serve.call_on_current_serve(
            self.system_app, lambda serve: serve.conv_storage
        )
        if not storage:
            self._storage = InMemoryStorage()  # 兜底
        return self._storage

这种"Operator 延迟从 Serve 拿 storage"的模式在多个 Serve 模块里都能看到—— 它解决了"Operator 在 DAG 构建时尚未注入 system_app"的时序问题。

七、精读 flow 模块：JSON DAG 的持久化、加载、执行

flow 是 Serve 层最复杂的模块。它负责把前端拖拽出的 JSON DAG（工作流）保存到数据库、 在启动时重新加载、按需注册到 DAGManager 让它真正"可执行"。

7.1 实体模型：flow_data 是个 JSON blob

flow 模块的核心表 dbgpt_serve_flow 字段非常丰富—— 除了常规的 uid / name / user_name / sys_code，最重要的是 flow_data：

# flow/models/models.py
class ServeEntity(Model):
    __tablename__ = SERVER_APP_TABLE_NAME  # "dbgpt_serve_flow"
    __table_args__ = (UniqueConstraint("uid", name="uk_uid"),)

    id = Column(Integer, primary_key=True)
    uid = Column(String(128), index=True, nullable=False)
    dag_id = Column(String(128), index=True)  # FlowFactory.build 生成
    name = Column(String(128), index=True)
    flow_category = Column(String(64))  # CHAT_AGENT / CHAT_FLOW / COMMON
    flow_data = Column(Text)  # ★ 整个 DAG 的 JSON 表示
    state = Column(String(32))  # INITIALIZING / DEPLOYED / RUNNING / LOAD_FAILED / DISABLED
    error_message = Column(String(512))
    source = Column(String(64))  # flow 来源: dbgpts / web / ...
    define_type = Column(String(32), default="json")  # json or python
    editable = Column(Integer)  # 0 可编辑, 1 不可编辑
    variables = Column(Text)  # 变量定义 JSON
    gmt_created = Column(DateTime, default=datetime.now)
    gmt_modified = Column(DateTime, default=datetime.now)

注意 define_type 有两种值："json"（前端拖拽产生）和 "python"（DBGPTs 包里的 Python DAG 定义）。这两种来源的 flow 共用同一张表。

7.2 状态机：State 枚举

flow 的 state 字段由 FlowFactory.State 管理， State.can_change_state 定义了合法的状态迁移：

stateDiagram-v2 [*] --> INITIALIZING INITIALIZING --> DEPLOYED: deploy INITIALIZING --> LOAD_FAILED: load failed DEPLOYED --> RUNNING: register OK DEPLOYED --> LOAD_FAILED: build failed RUNNING --> DISABLED: disable DISABLED --> DEPLOYED: enable

7.3 持久化：create_and_save_dag 的两阶段

当用户在前端点"保存并部署"时，前端会 POST /api/v2/serve/awel/flows， 进入 create_and_save_dag。这个方法是 flow Service 的灵魂：

# flow/service/service.py
def create_and_save_dag(self, request, save_failed_flow=False) -> ServerResponse:
    try:
        # 阶段 1: 构建内存中的 DAG 对象
        if request.define_type == "json":
            dag = self._flow_factory.build(request)  # JSON → DAG
        else:
            dag = request.flow_dag  # Python 定义直接拿
        request.dag_id = dag.dag_id
        request.flow_category = self._parse_flow_category(dag)
    except Exception as e:
        if save_failed_flow:  # 允许保存失败 flow（调试用）
            request.state = State.LOAD_FAILED
            request.error_message = str(e)
            return self.dao.create(request)
        raise ValueError(...)

    # 阶段 2: 写库
    self.dao.create(request)
    res = self.get({"uid": request.uid})

    # 阶段 3: 如果 state=DEPLOYED，注册到 DAGManager（真正"激活"）
    state = request.state
    try:
        if state == State.DEPLOYED:
            self.dag_manager.register_dag(dag, request.uid)
            request.state = State.RUNNING  # 注册成功 → RUNNING
            request.error_message = ""
            self.dao.update({"uid": request.uid}, request)
        else:
            logger.info(f"Flow state is {state}, skip register DAG")
    except Exception as e:
        if save_failed_flow:
            request.state = State.LOAD_FAILED
            self.dao.update(...)
        else:
            self.delete(request.uid)  # 回滚
        raise e
    return res
三阶段事务的脆弱点：这个方法在 dao.create 之后还做了 dag_manager.register_dag。如果中间崩溃，会出现"DB 里有但 DAGManager 里没注册"的脏数据。 代码用 try/except + delete 回滚 兜底，但回滚本身可能失败。 生产环境部署 flow 时建议监控 LOAD_FAILED 状态。

7.4 启动加载：从 DB 和 dbgpts 包两处恢复

应用启动时，flow Service 的 before_start 和 after_start 会做两轮加载： _pre_load_dag_from_db 先预热依赖（pre_load_requirements）， 然后 load_dag_from_db 真正 build + register：

# flow/service/service.py
def load_dag_from_db(self):
    entities = self.dao.get_list({})
    for entity in entities:
        try:
            if entity.define_type != "json": continue  # 跳过 Python 定义
            dag = self._flow_factory.build(entity)
            if entity.state in [State.DEPLOYED, State.RUNNING] or (
                entity.version == "0.1.0" and entity.state == State.INITIALIZING
            ):
                self.dag_manager.register_dag(dag, entity.uid)
                entity.state = State.RUNNING
                entity.error_message = ""
                self.dao.update({"uid": entity.uid}, entity)
        except Exception as e:
            logger.warning(f"Load DAG({entity.name}) error: {e}")

def load_dag_from_dbgpts(self, is_first_load=False):
    flows = self.dbgpts_loader.get_flows()  # 扫描已安装的 dbgpts 包
    for flow in flows:
        try:
            flow.state = State.DEPLOYED
            exist_inst = self.dao.get_one({"name": flow.name})
            if not exist_inst:
                self.create_and_save_dag(flow, save_failed_flow=True)
            elif is_first_load or exist_inst.state != State.RUNNING:
                flow.uid = exist_inst.uid
                self.update_flow(flow, check_editable=False, save_failed_flow=True)
        except Exception as e:
            logger.warning(f"Load DAG {flow.name} from dbgpts error")

def after_start(self):
    self.load_dag_from_db()
    self.load_dag_from_dbgpts(is_first_load=True)
    # 周期性扫 dbgpts 包（默认 5 秒）
    schedule.every(self._serve_config.load_dbgpts_interval).seconds.do(
        self.load_dag_from_dbgpts
    )

这套加载机制让 DB-GPT 支持"把工作流像 npm 包一样分发"—— 别人装了你的 dbgpts 包，启动时自动加载并部署里面的 flow。

7.6 VariablesService：变量加密与作用域

flow 还附带 VariablesService——管理 DAG 里复用的变量（如 LLM 的 API Key）。 变量有作用域（global / flow / app / agent / datasource）和加密（Fernet）：

# flow/serve.py
class Serve(BaseServe):
    def __init__(self, system_app, config=None,
                 api_prefix=None,  # flow 有两个前缀
                 api_tags=None, db_url_or_db=None, try_create_tables=False):
        if api_prefix is None:
            api_prefix = ["/api/v1/serve/awel", "/api/v2/serve/awel"]
        ...
        self._variables_provider = StorageVariablesProvider(
            storage=None,
            encryption=FernetEncryption(self._config.encrypt_key),
            system_app=system_app,
        )
        system_app.register_instance(self._variables_provider)

    def init_app(self, system_app):
        for prefix in self._api_prefix:  # ★ 同一份 router 挂两个前缀
            self._system_app.app.include_router(router, prefix=prefix, tags=...)

这里有个关键技巧：flow 的路由同时挂在 /api/v1/serve/awel 和 /api/v2/serve/awel 两个前缀下。v1 给老前端用（鉴权放行）， v2 给新前端和 OpenAI 兼容客户端用（必须带 API Key）。

7.7 其他模块的差异化设计

除了 conversation 和 flow 两个"明星模块"，其他 Serve 子模块虽然走同样的四层模板， 但在对接的 core 子系统和特有的业务逻辑上各有差异。 下面对照着读源码，看看它们的"同"与"不同"。

agent 模块：多 DAO 协作

agent 模块的目录结构是 Serve 层里最复杂的——它包含 chat / app / hub / team / resource / evaluation 六个子子模块，每个都是一套独立的小 Serve。agent/chat/ 负责聊天记录的 CRUD， agent/app/ 负责"应用"（独立部署的 Agent）管理，agent/hub/ 是 Agent 插件市场。 这种" Serve 里嵌套 Serve"的结构，是为了让每个子能力都能独立挂路由。

# agent/chat/service/service.py
class Service(BaseService[ServeEntity, ServeRequest, ServerResponse]):
    name = SERVE_SERVICE_COMPONENT_NAME  # "dbgpt_serve_agent_chat_service"

    def init_app(self, system_app):
        super().init_app(system_app)
        self._dao = self._dao or ServeDao(self._serve_config)
        self._system_app = system_app

agent 的 Service 类结构和其他模块几乎一样，但它的业务方法（list_apps / install_app / uninstall_app）会调用 dbgpt.agent 下的 AgentManager、 PluginManager，而不只是单纯查库。这是 "Serve = core 能力的 HTTP 外壳" 这句话的最好例证。

model 模块：桥接 WorkerManager

model Serve 不像其他模块那样把全部业务放在 Service——它大量使用 Depends 注入的方式 把 WorkerManager / BaseModelController / ModelStorage 直接拉到 Endpoint 层：

# model/api/endpoints.py
def get_worker_manager() -> WorkerManager:
    # 从 SystemApp 拿全局唯一的 WorkerManagerFactory
    return WorkerManagerFactory.get_instance(global_system_app).create()

def get_model_controller() -> BaseModelController:
    return BaseModelController.get_instance(global_system_app)

def get_model_storage() -> ModelStorage:
    from ..serve import Serve as ModelServe
    model_serve = ModelServe.get_instance(global_system_app)
    return ModelStorage(model_serve.model_storage)

这是因为模型启停 / 列表查询是"重资源"操作，每个端点需要拿到的是不同的 core 组件。 model Serve 展示了"不必把所有逻辑塞进 Service 层"的灵活性—— 当 Endpoint 直接对接 core 组件比绕一层 Service 更清晰时，允许"越层"。

rag 模块：多 DAO + 异步同步任务

rag Serve 是除 flow 外第二复杂的 Serve。它的 Service 类同时持有 3 个 DAO：

# rag/service/service.py
class Service(BaseService[KnowledgeSpaceEntity, SpaceServeRequest, SpaceServeResponse]):
    def __init__(self, system_app, config,
                 dao: Optional[KnowledgeSpaceDao] = None,
                 document_dao: Optional[KnowledgeDocumentDao] = None,
                 chunk_dao: Optional[DocumentChunkDao] = None):
        self._dao = dao                      # 知识空间
        self._document_dao = document_dao    # 文档
        self._chunk_dao = chunk_dao          # 切片

这是因为 RAG 的数据模型有三层：知识空间 → 文档 → 切片。每层都有独立的表， 所以需要三个 DAO。文档同步（/documents/sync）涉及异步长任务： 上传文档 → 切分 → embedding → 入库，全程走 blocking_func_to_async 包装， 状态用 SyncStatus.TODO/RUNNING/FINISHED/FAILED 枚举追踪。 这种"长任务 + 状态轮询"是 rag Serve 特有的模式。

datasource 模块：v2 only

datasource 是少有的"只用 v2 前缀"的 Serve：

# datasource/serve.py
api_prefix: Optional[str] = "/api/v2/serve"  # 没有 /api/v1 版本

这是因为 datasource 是后加的 Serve，没有兼容老前端的负担。 它的 Endpoint 直接用 DatasourceServeRequest 接收连接参数， 调 ConnectorManager 测试连通性。这种"纯 v2"的 Serve 是未来新模块的推荐做法—— v1 兼容路径最终会被清理。

八、数据库迁移与建表策略

DB-GPT 在 serve 层没有使用 alembic（虽然 storage 层支持）。 每个 Serve 模块走的是 SQLAlchemy create_all() 自动建表：

# dbgpt_serve/core/serve.py
def create_or_get_db_manager(self) -> DatabaseManager:
    from dbgpt.storage.metadata import Model, UnifiedDBManagerFactory, db

    db_manager_factory = self._system_app.get_component(
        ComponentType.UNIFIED_METADATA_DB_MANAGER_FACTORY,
        UnifiedDBManagerFactory, default_component=None,
    )
    if db_manager_factory is not None and db_manager_factory.create():
        init_db = db_manager_factory.create()
    else:
        init_db = self._db_url_or_db or db
        init_db = DatabaseManager.build_from(init_db, base=Model)

    if self._try_create_tables and self._not_create_table:
        try:
            init_db.create_all()  # ★ SQLAlchemy.MetaData.create_all
        except Exception as e:
            logger.warning(f"Failed to create tables: {e}")
        finally:
            self._not_create_table = False
    return init_db
为什么不用 alembic？
  • create_all 只建不迁移——表存在就跳过，不会改字段。这给"加字段"造成麻烦：老数据库升级时新字段不会自动加。
  • DB-GPT 的策略是"字段变更走 try/except"——运行时如果缺列报错，开发者手动跑 SQL 或删表重建。
  • 这是 serve 层的一个已知技术债。生产部署前建议在测试库验证表结构。

Serve 模块还提供了 模板工具（dbgpt_serve/utils/_template_files/default_serve_template/） 用于生成新的 Serve 子模块——包含 serve.py / service.py / endpoints.py / schemas.py / models.py / config.py 的脚手架。这是保持"四层模板"一致性的关键。

调试技巧：要在浏览器里直接测试 Serve API，最简单的方式是：
  1. 启动后访问 http://localhost:5670/docs——FastAPI 自动生成的 Swagger UI。

Serve 层与 Scene 层的区别： Scene（第 8 章）是聊天场景（chat_normal / chat_with_db / chat_excel...）， 处理的是"一次对话用什么模式"。Serve 是HTTP 接口层，提供所有场景共用的 会话/消息/Flow 等 CRUD。Scene 是业务维度的切分，Serve 是技术维度的切分。 两者正交：同一个 Scene 下的对话，依然会调 conversation Serve 的 API。

基础题

设计题

小结：Serve 层是 DB-GPT 工程化最成熟的一层。
  • 模板化：17+ 个模块共用一套四层架构，90% 代码结构一致
  • FastAPI 优先：用 Pydantic 自动生成 OpenAPI 文档，用 Depends 做依赖注入
  • 组件化：所有 Serve 通过 SystemApp 注册，跨模块调用走 IoC
  • 待改进：缺 alembic 迁移、错误码语义不强、轮询而非事件驱动
理解了 Serve 层，你就能定位 DB-GPT 90% 的 HTTP 接口 bug；想扩展新功能， "复制粘贴 conversation 模块改名" 是最快的起步方式。
