05 · RAG 检索系统 Core

五层架构、九大 Knowledge、AWEL Operator 化 —— DB-GPT 如何把"检索增强生成"拆成可编排的积木

一句话定位：DB-GPT 的 RAG 子系统把"文档 → 向量 → 检索 → 重排 → 拼上下文"这条链路， 拆成 五层抽象（Knowledge / Embedding / VectorStore / Retriever / Reranker）， 再用 AWEL Operator 把每一层封装成可拖拽的 DAG 节点。 这让"换一个 Embedding 模型"或"加一个 Reranker"变成只改一行 Operator 配置。

设计哲学：每一层都是"抽象 + 实现"的模板方法模式。 Knowledge / Embeddings / IndexStoreBase / BaseRetriever / Ranker 五个抽象类， 圈定了每一层的契约边界。替换任何一层不影响其它层 —— 这是 DB-GPT RAG 可扩展的根基。

对比业界其它 RAG 框架（如 LangChain 的 RetrievalQA、LlamaIndex 的 QueryEngine）， DB-GPT 的独特之处在于把每一层都 AWEL Operator 化。 在 LangChain 中，RAG 是一条写死的 Chain（RetrievalQA.from_chain_type）； 而在 DB-GPT 中，RAG 是一组可自由编排的 Operator 节点。 你可以在检索前插入 Query Rewrite，在检索后追加 Reranker， 甚至并行跑两个不同参数的 Retriever 再合并结果 —— 这些都只需改 DAG 拓扑，无需改业务代码。

此外，DB-GPT 的 RAG 子系统还承担了一个特殊职责：为 Agent 提供知识工具。 在 Agent 章节我们会看到，KnowledgeSpaceRetrievalTool 就是把 RAG 封装成一个 Agent 可调用的工具。 Agent 在推理循环中决定"是否需要检索知识库"，如果需要，就调用这个工具， 背后执行的正是本章讨论的 EmbeddingRetrieverOperator 链路。 所以理解 RAG 五层架构，也是理解 Agent 知识增强能力的基础。

二、第一层：Knowledge —— 知识源抽象

文件：dbgpt-core/src/dbgpt/rag/knowledge/base.py

Knowledge 是所有知识源的基类。它定义了"从某种数据源加载文档"的统一接口：

class Knowledge(ABC):
    """Knowledge Base Class."""

    def __init__(self, path, knowledge_type, knowledge_type, loader, metadata, **kwargs):
        self._path = path
        self._type = knowledge_type
        self._loader = loader
        self._metadata = metadata

    def load(self) -> List[Document]:
        """Load knowledge from data loader."""
        documents = self._load()        # 子类实现：真正的解析逻辑
        return self._postprocess(documents)  # 钩子：后处理

    @classmethod
    @abstractmethod
    def type(cls) -> KnowledgeType:
        """Get knowledge type."""

    @abstractmethod
    def _load(self) -> List[Document]:
        """子类必须实现：真正加载文档的逻辑"""

    @classmethod
    def support_chunk_strategy(cls) -> List[ChunkStrategy]:
        return ChunkStrategy.CHUNK_BY_SIZE  # 默认只支持按大小切

    @classmethod
    def default_chunk_strategy(cls) -> ChunkStrategy:
        return ChunkStrategy.CHUNK_BY_SIZE

这里有一个标准的模板方法模式：load() 是 final 方法，它先调子类的 _load()， 再调 _postprocess() 做后处理。子类只需实现 _load() 即可。

CSVKnowledge 示例（精读）

文件：dbgpt-ext/src/dbgpt_ext/rag/knowledge/csv.py

class CSVKnowledge(Knowledge):
    """CSV Knowledge."""

    def _load(self) -> List[Document]:
        """Load csv document from loader."""
        if self._loader:
            documents = self._loader.load()
        else:
            docs = []
            with open(self._path, newline="", encoding=self._encoding) as csvfile:
                csv_reader = csv.DictReader(csvfile)
                for i, row in enumerate(csv_reader):
                    strs = []
                    for k, v in row.items():
                        if k is None or v is None:
                            continue
                        strs.append(f"{k.strip()}: {v.strip()}")
                    content = "\n".join(strs)  # 每行变成 "列名: 值" 的文本
                    metadata = {"source": self._path, "row": i}
                    docs.append(Document(content=content, metadata=metadata))
            return docs

    @classmethod
    def support_chunk_strategy(cls) -> List[ChunkStrategy]:
        return [ChunkStrategy.CHUNK_BY_SIZE, ChunkStrategy.CHUNK_BY_SEPARATOR]

    @classmethod
    def document_type(cls) -> DocumentType:
        return DocumentType.CSV
关键设计：CSV 每一行被拼成 "列名: 值\n列名: 值\n..." 的文本， 而不是保留表格结构。这让 Embedding 模型能更好地理解语义。 support_chunk_strategy() 允许子类收敛支持的分块策略（CSV 额外支持按分隔符切）。

再来看 URLKnowledge（文件 dbgpt_ext/rag/knowledge/url.py）。 它的 _load() 方法用 requests 抓取网页，再用 BeautifulSoup 解析 HTML， 去掉 <script> 和 <style> 标签，提取纯文本。 同时还会提取 <title> 和 <meta name="description"> 作为 metadata。 文本清洗逻辑（clean_text）会去除零宽字符（\u200b 等）、控制字符， 并做 Unicode NFKC 归一化。这些细节决定了网页类知识源的检索质量。

而 PDFKnowledge 是最复杂的 Knowledge 实现，包含一个完整的 PDFProcessor 类。 它用 pdfplumber 解析 PDF，不仅提取文本，还能识别表格（excel 类型）， 把表格转成 Markdown 格式拼入正文。还有正则匹配一级/二级标题的 process_text_data 逻辑， 以及页眉页脚检测。这说明 DB-GPT 对 PDF 这类复杂格式做了深度适配， 不是简单调用 pdfplumber.extract_text() 就完事。

KnowledgeFactory：根据扩展名自动分发

文件：dbgpt-ext/src/dbgpt_ext/rag/knowledge/factory.py

class KnowledgeFactory:
    @classmethod
    def from_file_path(cls, file_path, knowledge_type=KnowledgeType.DOCUMENT, ...) -> Knowledge:
        factory = cls(file_path=file_path, ...)
        return factory._select_document_knowledge(file_path=file_path, ...)

    def _select_document_knowledge(self, **kwargs):
        extension = self._file_path.rsplit(".", 1)[-1]  # 取扩展名
        knowledge_classes = self._get_knowledge_subclasses()  # 扫描所有子类
        for cls in knowledge_classes:
            if cls.document_type() and cls.document_type().value() == extension:
                return cls(**kwargs)  # 匹配扩展名，实例化
        raise Exception(f"Unsupported knowledge document type '{extension}'")

这里用了一个"注册表模式"：Knowledge.__subclasses__() 自动收集所有子类， 再按 document_type() 匹配扩展名。新增一种文档类型只需写一个子类，无需修改 Factory。

三、分块策略：ChunkStrategy 与 TextSplitter

分块（Chunking）是 RAG 中最影响检索质量的环节。DB-GPT 把分块策略定义为枚举：

class ChunkStrategy(Enum):
    CHUNK_BY_SIZE = (RecursiveCharacterTextSplitter, [...], "chunk size", ...)
    CHUNK_BY_PAGE = (PageTextSplitter, [], "page", ...)
    CHUNK_BY_PARAGRAPH = (ParagraphTextSplitter, [...], "paragraph", ...)
    CHUNK_BY_SEPARATOR = (SeparatorTextSplitter, [...], "separator", ...)
    CHUNK_BY_MARKDOWN_HEADER = (MarkdownHeaderTextSplitter, [], "markdown header", ...)

    def __init__(self, splitter_class, parameters, alias, description):
        self.splitter_class = splitter_class  # 对应的 TextSplitter 类
        self.parameters = parameters          # 参数定义（chunk_size 等）
        self.alias = alias
        self.description = description

    def match(self, *args, **kwargs) -> TextSplitter:
        """Match and build splitter."""
        return self.value[0](*args, **kwargs)  # 实例化对应的 Splitter
策略Splitter 类核心参数适用场景
CHUNK_BY_SIZERecursiveCharacterTextSplitterchunk_size=512, chunk_overlap=50通用默认，递归按分隔符切到指定大小
CHUNK_BY_PAGEPageTextSplitter—PDF / PPT 按页切
CHUNK_BY_PARAGRAPHParagraphTextSplitterseparator="\n"按段落切
CHUNK_BY_SEPARATORSeparatorTextSplitterseparator, enable_mergeCSV / JSON 按自定义分隔符切
CHUNK_BY_MARKDOWN_HEADERMarkdownHeaderTextSplitter—Markdown 按标题层级切

ChunkManager（位于 dbgpt-ext/src/dbgpt_ext/rag/chunk_manager.py）是分块的协调器：

class ChunkManager:
    def __init__(self, knowledge, chunk_parameter, extractor=None):
        self._knowledge = knowledge
        self._chunk_strategy = (
            chunk_parameter.chunk_strategy
            or knowledge.default_chunk_strategy().name  # 没指定就用默认
        )

    def split(self, documents: List[Document]) -> List[Chunk]:
        text_splitter = self._select_text_splitter()
        if self._splitter_type == SplitterType.LANGCHAIN:
            documents = text_splitter.split_documents(documents)
            return [Chunk.langchain2chunk(d) for d in documents]
        elif self._splitter_type == SplitterType.LLAMA_INDEX:
            nodes = text_splitter.split_documents(documents)
            return [Chunk.llamaindex2chunk(n) for n in nodes]
        else:
            return text_splitter.split_documents(documents)
三种 Splitter 生态：ChunkManager 支持三种后端 —— LANGCHAIN（LangChain splitter）、 LLAMA_INDEX（LlamaIndex splitter）、USER_DEFINE（DB-GPT 自带的 TextSplitter）。 默认用自带实现，但可以通过 SplitterType 切换到其它生态的分块器。

分块参数通过 ChunkParameters（一个 Pydantic Model）统一管理。 它被注册为 AWEL 资源（@register_resource("Chunk Parameters", "chunk_parameters", ...)）， 所以可以在可视化编排 UI 中作为参数面板使用。核心参数包括 chunk_size（默认 512）、 chunk_overlap（默认 50，表示相邻 chunk 的重叠字符数，用于避免在关键句子中间切断）。 separator 用于 CHUNK_BY_SEPARATOR 和 CHUNK_BY_PARAGRAPH 策略， enable_merge 控制按分隔符切完后是否要再按 chunk_size 合并小片段。

选对分块策略对 RAG 效果影响巨大。经验法则：长文档（PDF/Word）用 CHUNK_BY_SIZE 递归切； 结构化数据（CSV/JSON）用 CHUNK_BY_SEPARATOR 按记录切；Markdown 用 CHUNK_BY_MARKDOWN_HEADER 按标题层级切； 页面清晰的 PDF 可以用 CHUNK_BY_PAGE。如果不确定，用默认的 CHUNK_BY_SIZE 是最安全的选择。

四、第二层：Embedding —— 向量化

文件：dbgpt-core/src/dbgpt/core/interface/embeddings.py

Embeddings 是 Embedding 模型的核心抽象，只有两个必须实现的方法：

class Embeddings(ABC):
    """Interface for embedding models."""

    @abstractmethod
    def embed_documents(self, texts: List[str]) -> List[List[float]]:
        """Embed search docs."""

    @abstractmethod
    def embed_query(self, text: str) -> List[float]:
        """Embed query text."""

    # 异步默认实现：用线程池包装同步方法
    async def aembed_documents(self, texts):
        return await asyncio.get_running_loop().run_in_executor(
            None, self.embed_documents, texts
        )

接口极度简洁：embed_documents 用于批量向量化文档，embed_query 用于向量化查询语句。 两者可能用不同的 instruction（如 BGE 模型查询时要加前缀）。

以 HuggingFaceBgeEmbeddings 为例，看 query 与 document 的向量化差异：

class HuggingFaceBgeEmbeddings(BaseModel, Embeddings):
    query_instruction: str = DEFAULT_QUERY_BGE_INSTRUCTION_EN
    # "Represent this question for searching relevant passages: "

    def embed_documents(self, texts):
        texts = [t.replace("\n", " ") for t in texts]
        embeddings = self.client.encode(texts, **self.encode_kwargs)
        return embeddings.tolist()

    def embed_query(self, text):
        text = text.replace("\n", " ")
        embedding = self.client.encode(
            self.query_instruction + text,  # 查询时加 instruction 前缀！
            **self.encode_kwargs
        )
        return embedding.tolist()
易错点：BGE 模型的查询和文档用不同 instruction。如果忘了给 query 加前缀， 会导致相似度分数偏低、检索效果差。DB-GPT 还会根据 -zh 后缀自动切换中文 instruction。

Embedding 层还有一个配套的抽象：RerankEmbeddings（同样在 embeddings.py 中）。 它只有一个方法 predict(query, candidates) -> List[float]，用于给 query-candidate 对打分。 这和 Embeddings 的区别是：Embeddings 把文本变成向量（向量之间算余弦相似度）， 而 RerankEmbeddings 直接接收 query 和 candidates，内部做交叉编码，输出原始分数。 前者是双塔模型（Bi-Encoder），后者是交叉模型（Cross-Encoder）。 交叉模型精度更高但速度更慢，所以通常只用于对小批量候选做精排。

所有 Embedding 实现都通过 register_embedding_adapter 注册到全局适配器注册表。 注册时需要提供 EmbeddingModelMetadata（模型名、维度、上下文长度、支持语言等）， 这让 DB-GPT 能在启动时自动发现可用的 Embedding 模型，并在 APIServer 中暴露它们。 用户通过 DefaultEmbeddingFactory 可以根据模型名自动创建对应的 Embeddings 实例。

五、第三层：VectorStore —— 向量存储

文件：dbgpt-core/src/dbgpt/storage/base.py

IndexStoreBase 是向量存储的抽象基类。它定义了写入、检索、删除的核心契约：

class IndexStoreBase(ABC):
    """Index store base class."""

    @abstractmethod
    def load_document(self, chunks: List[Chunk]) -> List[str]:
        """写入文档到向量库，返回 chunk ids"""

    @abstractmethod
    async def aload_document(self, chunks, file_id=None) -> List[str]:
        """异步写入"""

    @abstractmethod
    def similar_search_with_scores(self, text, topk, score_threshold, filters=None) -> List[Chunk]:
        """带分数的相似度检索"""

    @abstractmethod
    def delete_by_ids(self, ids: str) -> List[str]:
        """按 id 删除"""

    def similar_search(self, text, topk, filters=None) -> List[Chunk]:
        """不带分数的检索 = similar_search_with_scores(threshold=1.0)"""
        return self.similar_search_with_scores(text, topk, 1.0, filters)

    def full_text_search(self, text, topk, filters=None) -> List[Chunk]:
        """全文检索（可选实现）"""
        raise NotImplementedError(...)  # 不是所有向量库都支持

IndexStoreBase 还内置了分批写入和容错能力：

def load_document_with_limit(self, chunks, max_chunks_once_load=None, max_threads=None):
    # 把 chunks 按 max_chunks_once_load 分组
    chunk_groups = [chunks[i:i+max_chunks_once_load]
                    for i in range(0, len(chunks), max_chunks_once_load)]
    with ThreadPoolExecutor(max_workers=max_threads) as executor:
        tasks = [executor.submit(self._safe_load_group, group) for group in chunk_groups]
        for future in tasks:
            ids.extend(future.result())
    return ids

def _safe_load_group(self, chunk_group):
    # 先整组写入，失败则逐条重试，跳过真正的坏数据
    try:
        return self.load_document(chunk_group)
    except Exception:
        ids = []
        for chunk in chunk_group:
            try:
                ids.extend(self.load_document([chunk]))
            except Exception:
                logger.warning(f"Skipping bad chunk")
        return ids
生产级容错：FP16 量化模型在个别 chunk 上可能产生 NaN，导致整批写入失败。 _safe_load_group 先整组试，失败后退化为逐条写入，只丢弃真正有问题的 chunk， 而不是让整个索引任务失败。这是工程上非常实用的防御策略。

值得注意的是，IndexStoreBase 还提供了 full_text_search 和 is_support_full_text_search 两个方法。全文检索（BM25 / TF-IDF）是向量检索的补充， 适合精确关键词匹配场景（如产品编号、人名）。不是所有向量库都内置了全文检索能力， 所以基类默认抛 NotImplementedError，由具体实现按需覆盖。 这为 Hybrid Retrieval（混合检索）提供了基础设施 —— 同时跑向量检索和全文检索，再用 RRF 或加权融合结果，兼顾语义相似和关键词精确匹配。

六、第四层：Retriever —— 召回与打分

文件：dbgpt-core/src/dbgpt/rag/retriever/base.py

BaseRetriever 定义了检索的模板方法：retrieve() 是公开 API， _retrieve() 是子类实现的私有方法。

class BaseRetriever(ABC):
    """Base retriever."""

    def retrieve(self, query: str, filters=None) -> List[Chunk]:
        """公开 API：检索（不带分数）"""
        return self._retrieve(query, filters)

    def retrieve_with_scores(self, query, score_threshold, filters=None):
        """公开 API：检索（带分数 + 阈值过滤）"""
        return self._retrieve_with_score(query, score_threshold, filters)

    @abstractmethod
    def _retrieve(self, query, filters=None) -> List[Chunk]:
        """子类实现"""

    @abstractmethod
    def _retrieve_with_score(self, query, score_threshold, filters=None) -> List[Chunk]:
        """子类实现：带分数 + 阈值过滤"""

EmbeddingRetriever 精读

文件：dbgpt-core/src/dbgpt/rag/retriever/embedding.py

class EmbeddingRetriever(BaseRetriever):
    def __init__(self, index_store, top_k=4, query_rewrite=None, rerank=None,
                 retrieve_strategy=RetrieverStrategy.EMBEDDING):
        self._top_k = top_k
        self._query_rewrite = query_rewrite
        self._index_store = index_store
        self._rerank = rerank or DefaultRanker(self._top_k)

    def _retrieve_with_score(self, query, score_threshold, filters=None):
        queries = [query]
        # 1. 多查询召回（如果有 query_rewrite 会先改写再搜索）
        candidates_with_score = [
            self._index_store.similar_search_with_scores(
                query, self._top_k, score_threshold, filters
            )
            for query in queries
        ]
        # 2. 合并多个查询的召回结果
        new_candidates = reduce(lambda x, y: x + y, candidates_with_score)
        # 3. 重排（去重 + 按分数排序 + 取 top_k）
        new_candidates = self._rerank.rank(new_candidates, query)
        return new_candidates
score_threshold 的语义：分数小于阈值的 chunk 会被过滤掉。 默认阈值 0.3（在 Operator 层定义）。不同向量库的分数含义不同 —— Chroma 返回的是距离（越小越相似），Milvus 返回的是相似度（越大越相似）。 DB-GPT 在各 VectorStore 实现内部做了统一化处理。

EmbeddingRetriever 默认使用 DefaultRanker 做重排（如果不显式传 rerank 参数）。 这意味着即使你不配置 Reranker，检索结果也会经过去重和按分数排序。 如果你想要更强的重排效果，可以传入 CrossEncoderRanker 或 RerankEmbeddingsRanker。 top_k 参数控制召回数量（从向量库取多少条），而 Ranker 的 topk 控制输出数量 （重排后保留多少条）。典型配置是召回 top-20、重排后取 top-3。

异步检索与 Query Rewrite

_aretrieve_with_score 支持先做一次检索拿 context，再用 LLM 改写查询，然后重新检索：

async def _aretrieve_with_score(self, query, score_threshold, filters=None):
    queries = [query]
    if self._query_rewrite:
        # 先用原查询检索，拿到 context
        chunks = await self._run_async_tasks([...])
        context = "\n".join([chunk.content for chunk in chunks])

            origin_query=query, context=context, nums=1
        )
        queries.extend(new_queries)  # 改写后的查询加入列表

    # 多查询并行检索，合并结果
    candidates = [self._similarity_search_with_score(q, ...) for q in queries]
    # 重排
    return await self._rerank.arank(merged_candidates, query)

七、第五层：Reranker —— 重排

文件：dbgpt-core/src/dbgpt/rag/retriever/rerank.py

Ranker 是重排的抽象基类：

class Ranker(ABC):
    def __init__(self, topk: int, rank_fn=None):
        self.topk = topk
        self.rank_fn = rank_fn

    @abstractmethod
    def rank(self, candidates_with_scores, query=None) -> List[Chunk]:
        """返回 top k chunks"""

    def _filter(self, candidates_with_scores) -> List[Chunk]:
        """去重：按 content 去重，按 score 降序"""
        candidates = sorted(candidates_with_scores, key=lambda x: x.score, reverse=True)
        visited_docs = set()
        new_candidates = []
        for c in candidates:
            if c.content not in visited_docs:
                new_candidates.append(c)
                visited_docs.add(c.content)
        return new_candidates

Ranker 实现家族

Ranker算法说明
DefaultRanker按 score 降序 + 去重最简单，不需要额外模型
RRFRankerReciprocal Rank Fusion多路召回融合（预留实现）
CrossEncoderRankerCrossEncoder（如 BAAI/bge-reranker-base）用 query-doc 对做交叉编码，效果最好但慢
RerankEmbeddingsRankerRerankEmbeddings 接口抽象的 rerank 模型接口，支持 API 调用
RetrieverNameRanker按 retriever 名过滤多路召回时只保留指定 retriever 的结果

CrossEncoderRanker 的核心逻辑：

class CrossEncoderRanker(Ranker):
    def __init__(self, topk=4, model="BAAI/bge-reranker-base", device="cpu", ...):
        from sentence_transformers import CrossEncoder
        self._model = CrossEncoder(model, max_length=512, device=device)

    def rank(self, candidates_with_scores, query=None) -> List[Chunk]:
        if len(candidates_with_scores) <= 1:
            return candidates_with_scores
        # 把 query 和每个 chunk 拼成 [query, doc] 对
        query_content_pairs = [[query, c.content] for c in candidates_with_scores]
        # CrossEncoder 打分
        rank_scores = self._model.predict(sentences=query_content_pairs)
        # 用新分数覆盖，排序，取 top_k
        for candidate, score in zip(candidates_with_scores, rank_scores):
            candidate.score = float(score)
        new_candidates = sorted(candidates_with_scores, key=lambda x: x.score, reverse=True)
        return new_candidates[:self.topk]
双阶段检索：Embedding 检索是"双塔模型"（query 和 doc 独立编码，用余弦相似度）， 速度快但精度有限。CrossEncoder 是"交互模型"（query 和 doc 拼在一起送入模型）， 精度高但慢。所以典型流程是：先用 Embedding 粗召 top-20，再用 CrossEncoder 精排取 top-3。

除了重排算法，DB-GPT 还提供了 Query Rewrite（查询改写）机制。 QueryRewrite 类（dbgpt/rag/retriever/rewrite.py）用 LLM 根据上下文改写原始查询， 生成 N 个相关的新查询。这些新查询会被分别用于检索，结果合并后再重排。 这对于多轮对话场景特别重要 —— 用户可能在第二轮说"那它的价格呢？"， 这时需要根据上下文把"它"指代消解，改写成完整查询再检索。 中英文 prompt 模板都已内置（REWRITE_PROMPT_TEMPLATE_EN/ZH）。

EmbeddingRetrieverOperator 精读

class EmbeddingRetrieverOperator(RetrieverOperator[Union[str, List[str]], List[Chunk]]):
    """The Embedding Retriever Operator."""

    metadata = ViewMetadata(
        label="Embedding Retriever Operator",
        name="embedding_retriever_operator",
        category=OperatorCategory.RAG,
        parameters=[
            Parameter.build_from("Storage Index Store", "index_store", IndexStoreBase, ...),
            Parameter.build_from("Top K", "top_k", int, ...),
            Parameter.build_from("Score Threshold", "score_threshold", float,
                            default=0.3, ...),
            Parameter.build_from("Query Rewrite", "query_rewrite", QueryRewrite, optional=True, ...),
            Parameter.build_from("Rerank", "rerank", Ranker, optional=True, ...),
        ],
        inputs=[IOField.build_from("Query", "query", str, ...)],
        outputs=[IOField.build_from("Candidates", "candidates", Chunk, is_list=True, ...)],
    )

    def __init__(self, index_store, top_k, score_threshold=0.3,
                 query_rewrite=None, rerank=None, **kwargs):
        super().__init__(**kwargs)
        self._score_threshold = score_threshold
        self._retriever = EmbeddingRetriever(
            index_store=index_store, top_k=top_k,
            query_rewrite=query_rewrite, rerank=rerank,
        )

    def retrieve(self, query):
        if isinstance(query, str):
            return self._retriever.retrieve_with_scores(query, self._score_threshold)
        elif isinstance(query, list):
            # 多查询并行检索，结果合并
            candidates = [self._retriever.retrieve_with_scores(q, self._score_threshold)
                          for q in query]
            return reduce(lambda x, y: x + y, candidates)
ViewMetadata 的意义：每个 Operator 都携带 metadata（标签、参数定义、输入输出类型）。 这让可视化编排 UI 能自动渲染节点和连线。AWEL DAG 不只是 Python 代码，它是可序列化、可拖拽的。

RerankOperator：最简单的 Operator

class RerankOperator(MapOperator[List[Chunk], List[Chunk]]):
    """The Rerank Operator."""

    def __init__(self, topk=3, algorithm="default", rank_fn=None, **kwargs):
        super().__init__(**kwargs)
        self._rerank = DefaultRanker(topk=topk, rank_fn=rank_fn)

    async def map(self, candidates_with_scores: List[Chunk]) -> List[Chunk]:
        return await self.blocking_func_to_async(
            self._rerank.rank, candidates_with_scores
        )

九、评估器：RetrieverEvaluator

文件：dbgpt-core/src/dbgpt/rag/evaluation/retriever.py

DB-GPT 内置了三种检索质量评估指标：

指标含义计算方式
RetrieverSimilarityMetric余弦相似度用 Embedding 计算预测与标准答案的余弦相似度均值
RetrieverMRRMetricMRR（Mean Reciprocal Rank）1/rank，第一个相关文档的位置倒数
RetrieverHitRateMetric命中率top-k 中是否包含正确答案（0 或 1）

评估器本身就是一条 AWEL DAG：

async def evaluate(self, dataset, metrics=None, ...):
    with DAG("relevancy_evaluation_dag"):
        input_task = IteratorTrigger(dataset)
        query_task = MapOperator(lambda x: x["query"])
        retriever_task = self._operator_cls(**self._operator_kwargs)
        retriever_eva_task = RetrieverEvaluatorOperator(
            evaluation_metrics=metrics, llm_client=self.llm_client
        )
        # 构建 DAG 连线
        input_task >> query_task
        query_task >> retriever_eva_task        # query 送入评估器
        query_task >> retriever_task >> retriever_eva_task  # query → 检索 → 评估器
        input_task >> MapOperator(lambda x: x["contexts"]) >> retriever_eva_task
        input_task >> retriever_eva_task        # 原始数据也送入

    results = await input_task.trigger(parallel_num=parallel_num)
    return [item for _, item in results]
评估也是 DAG：RetrieverEvaluatorOperator 是一个 JoinOperator， 它同时接收 query、检索结果（prediction）、标准答案（contexts），然后并行计算多个指标。 整个评估过程通过 IteratorTrigger 流式触发数据集。

评估器的用法非常直观：你只需要准备一个数据集（包含 query 和标准答案 contexts）， 传入 RetrieverEvaluator，它就会自动构建一条 DAG 来并行评估。 默认指标是 RetrieverSimilarityMetric（余弦相似度）， 你也可以传入自定义指标列表（比如同时跑 MRR + HitRate + Similarity）。 parallel_num 参数控制并发评估数。这种"评估即 DAG"的设计， 让你可以把评估嵌入到 CI/CD 流程中 —— 每次切换 Embedding 模型或调整 chunk 策略后， 自动跑一遍评估集，确保检索质量没有退化。

除了检索评估，DB-GPT 在 dbgpt/rag/evaluation/answer.py 中还提供了 答案评估能力（AnswerRelevancyMetric）， 用于评估最终生成的回答与问题的相关性。这需要 LLM 参与（让 LLM 判断答案是否切题）， 属于更高级的 RAG 端到端评估。

from dbgpt.core.awel import DAG, MapOperator
from dbgpt_ext.rag.operators.knowledge import KnowledgeOperator, ChunksToStringOperator
from dbgpt_ext.rag.operators.embedding import EmbeddingAssemblerOperator, EmbeddingRetrieverOperator

# ---- 离线：构建知识库 ----
with DAG("build_kb"):
    knowledge_task = KnowledgeOperator(datasource="data/report.pdf")
    assembler_task = EmbeddingAssemblerOperator(
        index_store=chroma_store,
        chunk_parameters=ChunkParameters(chunk_strategy="CHUNK_BY_SIZE")
    )
    knowledge_task >> assembler_task  # 执行后 chunks 已写入 Chroma

# ---- 在线：查询检索 ----
with DAG("rag_query"):
    # 用户 query 直接进入检索
    retriever_task = EmbeddingRetrieverOperator(
        index_store=chroma_store,
        top_k=10,
        score_threshold=0.3,
        rerank=CrossEncoderRanker(topk=3)  # 内置 reranker
    )
    rerank_task = RerankOperator(topk=3)
    to_string_task = ChunksToStringOperator(separator="\n\n")

    retriever_task >> rerank_task >> to_string_task
    # to_string_task 的输出直接拼入 LLM prompt
两阶段合一：EmbeddingRetrieverOperator 已经内置了 reranker（通过 rerank 参数）， 外面再套 RerankOperator 是二次重排。实际使用时二选一即可。 如果检索效果不够好，可以先加 QueryRewriteOperator 改写查询，再检索。

除了通用的文档 RAG，DB-GPT 还为 Text-to-SQL 场景提供了专用的 Assembler 和 Retriever： DBSchemaAssembler 和 DBSchemaRetriever（位于 dbgpt_ext/rag/assembler/db_schema.py）。 它们不是检索文档内容，而是检索数据库的表结构信息（表名、列名、注释、外键关系）。 当用户问"查询销售额前十的客户"时，LLM 需要知道有哪些表、列叫什么、怎么 JOIN， 这些 schema 信息通过 RAG 的方式检索最相关的部分送入 prompt， 而不是把整个数据库 schema 都塞进去。这是 DB-GPT 作为"数据助理"平台的核心 RAG 应用之一。

类似地，SummaryAssembler（dbgpt_ext/rag/assembler/summary.py）用于 文档摘要检索 —— 先用 LLM 为每篇文档生成摘要，再把摘要向量化。 检索时先匹配摘要，命中后再返回完整文档。这对于长文档库特别有效， 因为摘要比全文更紧凑，向量匹配更准确。

class BaseAssembler(ABC):
    def __init__(self, knowledge, chunk_parameters=None, extractor=None, **kwargs):
        self._knowledge = knowledge
        self._chunk_manager = ChunkManager(
            knowledge=knowledge, chunk_parameter=chunk_parameters
        )
        self._chunks: List[Chunk] = []
        self.load_knowledge(knowledge)  # 自动加载

    def load_knowledge(self, knowledge):
        documents = knowledge.load()      # 1. 加载文档
        self._chunks = self._chunk_manager.split(documents)  # 2. 分块

小结：DB-GPT 的 RAG 子系统是一个教科书级的"抽象 + 实现 + 编排"三层设计。
  • 抽象层（core）：5 个抽象类定义了每一层的契约
  • 实现层（ext）：11 种 Knowledge、5+ 种 Embedding、9 种 VectorStore、5 种 Ranker
  • 编排层（AWEL Operator）：9+ 个 Operator 把每一层封装成 DAG 节点
这种设计让 RAG 从"一段写死的流程"变成了"可自由组装的积木"。
