一句话定位：DB-GPT 的数据源子系统用 BaseConnector 这一层抽象， 把 MySQL / PostgreSQL / SQLite / DuckDB / Spark / OceanBase / Neo4j / Redis 等异构数据源， 全部归一成"列出一堆表 → 拿到建表语句 → 跑一段 SQL → 返回 DataFrame"的统一接口。 上层的 chat_db 场景、Agent 的 DBResource、AWEL 的 SqlExecOperator 都只面对这套接口， 不知道底下跑的是哪种方言——这是 DB-GPT 跨库能力的基石。

一、数据源体系总览

DB-GPT 的数据源源码横跨两个包：dbgpt-core 提供抽象与基础实现， dbgpt-ext 提供各厂商方言的具体连接器。整体分三大家族：

家族基类统一抽象典型成员
RDBMS（关系型） RDBMSConnector SQLAlchemy Engine + Inspector MySQL、PostgreSQL、SQLite、DuckDB、Oracle、MSSQL、OceanBase、ClickHouse、StarRocks、Doris、Vertica、GaussDB、openGauss、Hive
文件型 / 大数据 BaseConnector（直接继承） 自建 DataFrame / Session Spark（parquet/csv/json/orc/text）、DuckDB（也是文件型）
NoSQL / 图数据库 BaseConnector（直接继承） 原生驱动 + 方言查询 Neo4j（bolt + Cypher）、TuGraph、Redis（占位）

所有连接器最终都汇聚到 dbgpt_ext/datasource/schema.py 的 DBType 枚举里登记。 这个枚举是整个系统的"数据库名册"，RDBMSConnector._sync_tables_from_db 会用它判断 SQLite / DuckDB 这种文件库的特殊处理分支：

dbgpt_ext/datasource/schema.py DBType 枚举

设计要点：DbInfo 第二个参数 is_file_db 标记"嵌入式文件数据库"—— SQLite、DuckDB、Spark 三者没有独立服务进程，连接串是文件路径。这个标记会影响 _sync_tables_from_db 里 schema 的传参方式（文件库传 None，C/S 库传数据库名）。

二、BaseConnector：所有连接器的抽象基类

dbgpt/datasource/base.py 定义了所有数据源的最顶层抽象。 它通过一系列 raise NotImplementedError 的方法约定了"一个数据源必须能做什么"， 只有 run 是 @abstractmethod——也就是说， 只要你能跑一段查询并返回 List，你就合格了，其他元数据接口都是可选的。

这是典型的"宽接口 + 鸭子默认实现"风格：让 Spark、Neo4j 这种没法走 SQLAlchemy 的异构源也能低成本接入。

dbgpt/datasource/base.py L12-L15 
class BaseConnector(ABC):
    """Base class for all connectors."""
    db_type: str = "__abstract__db_type__"
    driver: str = ""

下面这张表梳理 BaseConnector 的核心方法（按职责分组）。 注意几乎所有方法的默认实现都是 raise NotImplementedError， 这意味着子类按需覆盖，不必全实现：

分组方法作用返回类型
元数据 get_table_names() 列出所有可用表 Iterable[str]
get_columns(table_name) 取某表的列定义（含类型/主键/注释） List[Dict]
get_table_info(table_names?) 取建表语句拼成的 schema 文本（喂 LLM 的核心） str

易错点：run 返回的 List 第一行是列名元组， 后续行才是数据。看 RDBMSConnector.run_to_df： result_lst[0] 当列、result_lst[1:] 当值—— 写新连接器时务必遵守这个约定，否则 DataFrame 列会错位。

三、RDBMSConnector：基于 SQLAlchemy 的关系型基类

dbgpt/datasource/rdbms/base.py 是整个数据源子系统的"重头戏"。 它继承 BaseConnector，内部用 SQLAlchemy 的 Engine + Inspector + MetaData 三件套，把所有"标准关系型库该有的能力"一次性实现好。 MySQL/PostgreSQL/OceanBase 这些具体实现几乎不用写代码——只需声明 driver 和 db_type。

3.1 构造：从 URI 到 Engine

RDBMSConnector 提供三条构造路径，都最终落到 create_engine：

dbgpt/datasource/rdbms/base.py from_uri / from_uri_db / from_parameters 
@classmethod
def from_uri(cls, database_uri: str, engine_args: Optional[dict] = None, **kwargs):
    """Construct a SQLAlchemy engine from URI."""
    _engine_args = engine_args or {}
    return cls(create_engine(database_uri, **_engine_args), **kwargs)

@classmethod
def from_uri_db(cls, host, port, user, pwd, db_name, engine_args=None, **kwargs):
    db_url = f"{cls.driver}://{quote(user)}:{urlquote(pwd)}@{host}:{str(port)}/{db_name}"
    return cls.from_uri(db_url, engine_args, **kwargs)

@classmethod
def from_parameters(cls, parameters: RDBMSDatasourceParameters) -> "RDBMSConnector":
    db_url = parameters.db_url()
    return cls.from_uri(db_url, engine_args=parameters.engine_args())

构造函数里做了一件关键的事——反射整个库的元数据：

关键 _metadata.reflect 会把整库所有表的列、类型、主键、外键全部读进内存， 后续 get_table_info 直接用 CreateTable(table).compile(engine) 拼 DDL—— 这是"零代码生成建表语句"的魔法所在。

3.2 get_table_info：生成给 LLM 看的 schema 文本

这是数据源子系统对接 LLM 的核心方法。它遵循 Rajkumar et al, 2022 论文的最佳实践 （论文里证明了：在 prompt 里给建表语句 + 少量样本行，能显著提升 Text-to-SQL 准确率）。

dbgpt/datasource/rdbms/base.py · get_table_info L324-L370 节选 
def get_table_info(self, table_names: Optional[List[str]] = None) -> str:
    """Get information about specified tables.

    Follows best practices as specified in: Rajkumar et al, 2022
    (https://arxiv.org/abs/2204.00498)
    """
    all_table_names = self.get_usable_table_names()
    if table_names is not None:
        # 校验传入的表名必须存在
        missing_tables = set(table_names).difference(all_table_names)
        if missing_tables:
            raise ValueError(f"table_names {missing_tables} not found")
        all_table_names = table_names

    meta_tables = [tbl for tbl in self._metadata.sorted_tables
                   if tbl.name in set(all_table_names)
                   and not (self.dialect == "sqlite" and tbl.name.startswith("sqlite_"))]

    tables = []
    for table in meta_tables:
        # 优先用自定义表信息
        if self._custom_table_info and table.name in self._custom_table_info:
            tables.append(self._custom_table_info[table.name])
            continue
        # 用 SQLAlchemy 编译 CREATE TABLE 语句
        create_table = str(CreateTable(table).compile(self._engine))
        table_info = f"{create_table.rstrip()}"
        if self._indexes_in_table_info:
            table_info += f"\n\n{self._get_table_indexes(table)}"
        if self._sample_rows_in_table_info:
            table_info += f"\n{self._get_sample_rows(table)}"
        tables.append(table_info)
    return "\n\n".join(tables)

三个开关控制输出丰富度（都在构造函数里设）：

  • sample_rows_in_table_info（默认 3）：附加 N 行样本数据，让 LLM 看到真实值。
  • indexes_in_table_info（默认 False）：附加索引信息，帮 LLM 写出能命中索引的 SQL。
  • custom_table_info（Dict）：手工覆盖某张表的描述，用于纠偏或补充业务语义。

3.3 run：SQL 分流执行

run 是真正"跑 SQL"的入口。它用 sqlparse 解析 SQL 类型， 分流到 _query（SELECT）或 _write（INSERT/UPDATE/DELETE）， DDL 则走第三条路。一个巧思：写操作执行完后， 会用 convert_sql_write_to_select 反推一个 SELECT 回查结果。

dbgpt/datasource/rdbms/base.py · run L620-L654 节选 
def run(self, command: str, fetch: str = "all") -> List:
    if not command:
        return []
    parsed, ttype, sql_type, table_name = self.__sql_parse(command)
    command = self._format_sql(command)
    if ttype == sqlparse.tokens.DML:
        if sql_type == "SELECT":
            return self._query(command, fetch)
        else:                                       # INSERT/UPDATE/DELETE
            self._write(command)
            select_sql = self.convert_sql_write_to_select(command)
            return self._query(select_sql)          # 回查结果
    else:                                           # DDL
        with self.session_scope(commit=False) as session:
            cursor = session.execute(text(command))
            if cursor.returns_rows:
                return list(cursor.fetchall())
            return self.get_simple_fields(table_name)

3.4 query_ex：带超时的查询

不同方言的"查询超时"机制完全不同。query_ex 把这个差异收敛到一处—— 这是非常值得学习的"方言适配层"写法：

dbgpt/datasource/rdbms/base.py · query_ex 超时分支 L536-L584 节选 
if timeout is not None:
    if self.dialect == "mysql":
        # MySQL: 毫秒级 MAX_EXECUTION_TIME
        session.execute(text(f"SET SESSION MAX_EXECUTION_TIME = {int(timeout*1000)}"))
        return _execute_query(session, sql, params)
    elif self.dialect == "postgresql":
        # PG: statement_timeout 毫秒
        session.execute(text(f"SET statement_timeout = {int(timeout*1000)}"))
        return _execute_query(session, sql, params)
    elif self.dialect == "oceanbase":
        # OceanBase: ob_query_timeout 微秒
        ob_timeout = int(timeout * 1000000)
        session.execute(text(f"SET SESSION ob_query_timeout = {ob_timeout}"))
        return _execute_query(session, sql, params)
    elif self.dialect == "duckdb":
        # DuckDB 不支持原生超时，用线程池兜底
        with ThreadPoolExecutor(max_workers=1) as executor:
            future = executor.submit(_execute_query, session, sql, params)
            return future.result(timeout=timeout)

四、用 SQLAlchemy 统一不同方言的设计

DB-GPT 选择 SQLAlchemy 而非各家原生驱动，是一次关键的"工程决策"。 收益是用一套 Engine/Session/MetaData 抹平了 MySQL/PG/SQLite/Oracle 的差异， 代价是引入了对 SQLAlchemy 的强依赖。具体体现在：

  • Engine：所有连接池、驱动、方言差异隐藏在 create_engine(url) 一行里。 url 的前缀（mysql+pymysql / postgresql+psycopg2 / duckdb）决定一切。
  • Inspector：inspect(engine) 提供方言无关的 get_columns / get_indexes / get_table_comment——不用为每家库写 INFORMATION_SCHEMA 查询。
  • MetaData + CreateTable：MetaData.reflect(engine) 一次性把全库 schema 读进内存， 再用 CreateTable(table).compile(engine) 按目标方言反编译 DDL。
  • Session + scoped_session：线程安全的会话管理， 配合 session_scope 上下文管理器自动提交/回滚。

但也因此带来副作用：像 SQLite 的 PRAGMA、PostgreSQL 的 pg_catalog、 Neo4j 的 Cypher 这种方言特有能力，必须靠覆盖方法来补充。 下一节会看到，SQLiteConnector.get_indexes 就是直接调 PRAGMA index_list， 绕过了 Inspector——因为 SQLite 的索引在 SQLAlchemy Inspector 里不全。

五、ext 层数据源实现概览

dbgpt_ext/datasource/rdbms/ 下每个 conn_*.py 就是一家数据库。 大部分实现极其精简——只声明 db_type / db_dialect / driver 三个类属性， 其他全部继承自 RDBMSConnector。下面看几个典型。

5.2 SQLite：覆盖方言特有方法

SQLite 因为没有 SHOW COLUMNS、没有 information_schema， 所以要覆盖一批方法用 PRAGMA 替代：

dbgpt_ext/datasource/rdbms/conn_sqlite.py · 方言特有方法 L106-L144 节选 
def get_indexes(self, table_name):
    with self.session_scope() as session:
        cursor = session.execute(text(f"PRAGMA index_list({table_name})"))
        indexes = cursor.fetchall()
        result = []
        for idx in indexes:
            index_name = idx[1]
            cursor = session.execute(text(f"PRAGMA index_info({index_name})"))
            column_names = [info[2] for info in cursor.fetchall()]
            result.append({"name": index_name, "column_names": column_names})
        return result

def get_show_create_table(self, table_name):
    with self.session_scope() as session:
        cursor = session.execute(text(
            "SELECT sql FROM sqlite_master WHERE type='table' AND name='{table_name}'"
        ))
        return cursor.fetchall()[0][0]

def get_fields(self, table_name, db_name=None):
    # PRAGMA table_info 取代 information_schema.COLUMNS
    with self.session_scope() as session:
        cursor = session.execute(text(f"PRAGMA table_info('{table_name}')"))
        return [(f[1], f[2], f[3], f[4], f[5]) for f in cursor.fetchall()]

5.3 PostgreSQL：schema 感知

PG 的特色是 schema（命名空间）概念。默认 public， PostgreSQLConnector._sync_tables_from_db 会按 schema 过滤表：

dbgpt_ext/datasource/rdbms/conn_postgresql.py L95-L130 节选 
def _sync_tables_from_db(self) -> Iterable[str]:
    schema = self._schema or "public"
    with self.session_scope() as session:
        table_results = session.execute(text(
            "SELECT tablename FROM pg_catalog.pg_tables WHERE schemaname = :schema"
        ), {"schema": schema})
        view_results = session.execute(text(
            "SELECT viewname FROM pg_catalog.pg_views WHERE schemaname = :schema"
        ), {"schema": schema})
        self._all_tables = set(r[0] for r in table_results).union(
            set(r[0] for r in view_results))
        self._metadata.reflect(bind=self._engine, schema=schema)
        return self._all_tables

六、AWEL Operator 化：SqlExecOperator / DatasourceOperator

数据源要进 AWEL DAG，需要被包装成 MapOperator。 DB-GPT 提供两层封装：

6.1 核心层的 DatasourceOperator（通用）

dbgpt/datasource/operators/datasource_operator.py · 完整实现 L13-L28 全文 
class DatasourceOperator(MapOperator[str, Any]):
    """The Datasource Operator."""

    def __init__(self, connector: BaseConnector, **kwargs):
        super().__init__(**kwargs)
        self._connector = connector

    async def map(self, input_value: str) -> Any:
        """Execute the query."""
        return await self.blocking_func_to_async(self.query, input_value)

    def query(self, input_value: str) -> Any:
        return self._connector.run_to_df(input_value)
关键 blocking_func_to_async 把同步的 run_to_df 包成异步—— 因为 SQLAlchemy 是阻塞 API，而 AWEL 是 asyncio 事件循环。 这是 DB-GPT 跨"同步/异步"的通用桥。

6.2 examples 里的 SqlExecOperator（端到端示例）

在 examples/awel/simple_nl_schema_sql_chart_example.py 里， SqlExecOperator 是用户自己写的一个更具体的 Operator， 它接收上游生成的 SQL，调 connector.run_to_df 返回 DataFrame， 再喂给下游 ChartDrawOperator 画图：

examples/awel/simple_nl_schema_sql_chart_example.py L179-L199 
class SqlExecOperator(MapOperator[Any, Any]):
    """The Sql Execution Operator."""

    def __init__(self, connector: Optional[RDBMSConnector] = None, **kwargs):
        super().__init__(**kwargs)
        self._connector = connector

    def map(self, sql: str) -> DataFrame:

        print(f"sql data is \n{dataframe}")
        return dataframe

七、给 LLM 提供 schema 的机制

Text-to-SQL 要 work，LLM 必须知道"有哪些表、字段长什么样"。 DB-GPT 提供三条注入路径：

7.1 路径 A：get_table_info 直接拼 prompt

最朴素的方式——调 connector.get_table_info() 拿到全部建表语句， 塞到 prompt 模板的 {table_info} 占位符里（见第八节 chat_db 的模板）。

7.2 路径 B：DBSummary + 向量检索（schema linking）

表太多时全塞 prompt 会超 token。DB-GPT 的做法是： 用 RdbmsSummary 把每张表压成一行摘要（表名+列名+注释+索引）， 向量化后存到 ChromaVectorStore；用户提问时只召回 top-k 相关表的 DDL。 这套机制在 dbgpt_ext/rag/summary/rdbms_db_summary.py：

dbgpt_ext/rag/summary/rdbms_db_summary.py · _parse_table_summary L261-L307 节选 
def _parse_table_summary(conn: BaseConnector, summary_template, table_name) -> str:
    """生成单表摘要，用于向量化。"""
    columns = []
    for column in conn.get_columns(table_name):
        if column.get("comment"):
            columns.append(f"{column['name']} ({column.get('comment')})")
        else:
            columns.append(f"{column['name']}")
    column_str = ", ".join(columns)

    # 索引信息
    index_keys = []
    for index in conn.get_indexes(table_name):
        key_str = ", ".join(index["column_names"])
        index_keys.append(f"{index['name']}(`{key_str}`) ")

    table_str = summary_template.format(table_name=table_name, columns=column_str)
    if index_keys:
        table_str += f", and index keys: {', '.join(index_keys)}"
    # 表注释
    comment = conn.get_table_comment(table_name)
    if comment.get("text"):
        table_str += f", and table comment: {comment.get('text')}"
    return table_str

摘要进向量库 → 用户提问召回 → 命中的表才调 get_table_info(hit_tables) 取完整 DDL → 拼 prompt。 这是把 RAG 思路用在"schema 检索"上的精巧设计。

7.3 路径 C：Agent 的 DBResource.get_prompt

Agent 路径走 DBResource.get_prompt， 内部带 10 秒 TTL 缓存（避免每次问都重算 schema）：

dbgpt/agent/resource/database.py · DBResource.get_prompt L77-L96 节选 
@cached(cachetools.TTLCache(maxsize=100, ttl=10))
async def get_prompt(self, *, lang="en", prompt_type="default",
                   question=None, resource_name=None, **kwargs):
    if not self._db_name:
        return "No database name provided.", None
    schema_info = await blocking_func_to_async(
        self._executor, self.get_schema_link, db=self._db_name, question=question
    )
    return (
        self._prompt_template.format(db_type=self._db_type, schemas=schema_info),
        None,
    )
缓存 @cached(TTLCache(maxsize=100, ttl=10))——同一个 DB 10 秒内的多次 prompt 复用同一份 schema。 这对多轮对话很关键：用户连续追问"再按年龄分组"，不用每次都全表反射。

Agent 系统里数据源被封装成 Resource。DBResource 是抽象， RDBMSConnectorResource 是把 RDBMSConnector 包装成 Resource 的具体类， SQLiteDBResource 是它的特化（自动用 SQLiteConnector 建连）：

dbgpt_app/scene/chat_db/data_loader.py L8-L37 节选 
class DbDataLoader:
    def get_table_view_by_conn(self, data, speak, sql=None):
        param = {}
        api_call_element = ET.Element("chart-view")
        try:
            param["type"] = "response_table"
            param["sql"] = sql
            param["data"] = json.loads(
                data.to_json(orient="records", date_format="iso", date_unit="s")
            )
            view_json_str = json.dumps(param, default=serialize, ensure_ascii=False)
        except Exception as e:
            err_param = {"sql": sql, "type": "response_table", "err_msg": str(e), "data": []}
            view_json_str = json.dumps(err_param, ...)

        return speak + "\n" + ET.tostring(api_call_element, encoding="utf-8").decode("utf-8")

九、本章测验

题 1（基础）

BaseConnector 里唯一标了 @abstractmethod 的方法是什么？为什么是它？

参考答案

是 run(command, fetch=)。因为它代表"执行查询并返回结果"这个所有数据源都必须具备的能力。 其他元数据方法（get_table_names / get_columns 等）默认 raise NotImplementedError， 允许子类按需覆盖——这是"宽接口 + 鸭子默认"的设计取舍：让 Spark、Neo4j 这种没有完整元数据的源也能低成本接入。

题 2（实现）

MySQLConnector 的实现只有 53 行就能 work，靠的是什么机制？

参考答案

靠 SQLAlchemy 的 Engine + Inspector + MetaData 三件套。 父类 RDBMSConnector 在 __init__ 里调 MetaData.reflect(engine) 把全库 schema 读进内存，后续 get_columns / get_indexes / get_table_info 全部由 Inspector 或反射后的 MetaData 完成。 MySQL 子类只需要声明 driver = "mysql+pymysql"， create_engine 就会自动选择 pymysql 驱动和 MySQL 方言。

题 3（设计）

sample_rows_in_table_info（默认 3）这个参数解决的是什么问题？ 它的依据是哪篇论文？

参考答案

解决的是 Text-to-SQL 的准确率问题。仅给 LLM 看建表语句，模型不知道列里实际存的是什么类型/格式/枚举值的数据， 容易写出语法对但语义错的 SQL。在 get_table_info 末尾追加 3 行真实数据 （如看到 status 列实际是 "active"/"inactive"，就能写出正确的 WHERE）， 能显著提升准确率。依据是 Rajkumar et al, 2022（arxiv.org/abs/2204.00498）， 这篇论文实验证明了"建表语句 + 样本行"的 prompt 组合效果最好。

题 4（跨方言）

query_ex 里 MySQL / PostgreSQL / OceanBase / DuckDB 的超时机制各是什么？ 为什么 DuckDB 要用 ThreadPoolExecutor？

参考答案
  • MySQL：SET SESSION MAX_EXECUTION_TIME = ms（毫秒）
  • PostgreSQL：SET statement_timeout = ms（毫秒）
  • OceanBase：SET SESSION ob_query_timeout = us（微秒，量级不同！）
  • DuckDB：用 ThreadPoolExecutor 提交任务 + future.result(timeout=)

DuckDB 走线程池是因为它没有原生的查询超时 SQL——它是嵌入式库， 查询在调用线程里同步执行，只能靠外部线程池的 future.result(timeout) 强制超时。 这也意味着 DuckDB 的超时是"尽力而为"，超时后查询线程可能仍在后台跑。

题 5（联动）

从用户在前端选"MySQL 库 chat_db"到 LLM 收到带 schema 的 prompt， 中间经过哪些组件？画出数据流。

参考答案

数据流（以 schema linking 模式为例）：

  1. 前端选库 → 后端取 ConnectorManager.get_connector(db_name) 拿到 RDBMSConnector
  2. 调 _parse_db_summary(connector) 把每张表压成"表名(列1, 列2, ...) and index keys: ..."格式的摘要
  3. 摘要写入 ChromaVectorStore 向量化
  4. 用户提问 → 用问题 embedding 在向量库召回 top-k 相关表
  5. 对命中的表调 connector.get_table_info(hit_tables) 取完整 DDL（含建表语句 + 样本行）
  6. DDL 填入 prompt 模板的 {table_info}，connector.dialect 填入 {dialect}
  7. 完整 prompt 发给 LLM → 返回 SQL
  8. connector.run_to_df(sql) 执行 → DataFrame
  9. DbDataLoader.get_table_view_by_conn 转成 <chart-view> XML 回前端
