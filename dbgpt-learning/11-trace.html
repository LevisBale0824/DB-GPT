一、场景设定

用户在 DB-GPT 前端选中了一个已配置的 MySQL 数据源（sales_db）， 并选择聊天模式为 chat_db（NL2SQL 自动执行）。在输入框敲下：

# 用户输入
查询近 30 天销售额 Top10 的客户，并画出柱状图

点发送。接下来 16 步会在大约 3–8 秒内完成。

为什么选这个场景？ 它一次串起了 Scene、AWEL、LLM、DataSource、Vis 五大子系统， 是验证你"是否真懂 DB-GPT"的最佳测试题。如果跟得上这 16 步，本书就达到目的了。

三、阶段 1：前端发起（Step 1–2）

Step 1 · 前端构造请求

前端 web/pages/chat 的 Chat 输入框组件收集：

// web/client/api/chat.ts (示意)
{
  "chat_mode": "chat_db",          // 决定走哪个 Scene
  "chat_scene": "chat_db",
  "model_name": "gpt-4o-mini",
  "chat_param": {
    "db_name": "sales_db",         // 目标数据源
    "model_type": "llm_with_result"  // 让 LLM 自动执行 SQL
  },
  "user_input": "查询近 30 天销售额 Top10 的客户...",
  "stream": true
}

关键点：chat_mode 是路由的核心字段，前端只是透传， 后端 SceneResolver 根据它选场景。

Step 2 · 发起 HTTP 请求

前端用 fetch + ReadableStream 发起流式请求：

POST /api/v1/chat/completions
Content-Type: application/json
Accept: text/event-stream

对应后端入口见 Step 3。

四、阶段 2：路由 + Scene 选择（Step 3–5）

Step 3 · FastAPI 路由命中

请求落到 dbgpt-app/openapi/api_v1/chat/__init__.py 注册的 router 上。

packages/dbgpt-app/src/dbgpt_app/openapi/api_v1/chat/__init__.py关键路由
@router.post("/chat/completions", streaming=True)
async def chat_completions(request: ChatSceneV2):
    # 1. 根据 flow_id 加载持久化的 AWEL DAG
    if request.flow_id:
        dag = await dag_loader.load_dag(request.flow_id)
        return await dag._run(
            http_request=request, user_input=request.user_input, ...
        )
    # 2. 兜底路径：用 Scene 兼容老接口
    return await run_scene_v2(request)

这里有一个分支：如果请求带 flow_id，直接走持久化的 AWEL DAG（来自前端的"工作流"页面）； 否则走传统 Scene 路径（下面 Step 4–5）。

Step 4 · 加载 Scene 模板

传统路径下，由 scene/chat_factory.py 根据 chat_mode 实例化对应 ChatScene：

chat_mode = "chat_db"
→ ChatFactory.create(chat_mode="chat_db")
→ return ChatDB(...)

Step 5 · Scene 装配 prompt 与依赖

ChatScene 在 from_scene() 阶段做三件事：

  1. 拉取 DB Schema：调 RDBMS.table_info() 把表结构（CREATE TABLE 语句 + 索引 + 注释）注入 prompt
  2. 加载 prompt 模板：从 scene/chat_db/auto_execute/prompt.py 读取 SUFFIX 双分隔符模板
  3. 选择 Agent：默认走 default_awel_layout_agent（基于 AWEL Layout 的 Agent）

此时 Scene 已具备 prompt + 上下文，但它不直接调 LLM——而是交给一个 AWEL DAG。

五、阶段 3：AWEL DAG 编排（Step 6）⭐

这是整个链路的枢纽。DB-GPT 0.6+ 的设计哲学：所有业务逻辑都是 DAG。

scene/operators/app_operator.pyAppChatComposerOperator
# 简化的 DAG 拼装
input_op = InputOperator(req_dict)
history_op = history_load_op(input_op)              # 加载历史对话
prompt_op = prompt_build_op(history_op, db_info)    # 拼接最终 prompt
llm_op    = LLMOperator(prompt_op, model_instance)  # 调 LLM
sql_op    = SQLOutParser(llm_op)                    # 解析 SQL
exec_op   = SqlExecOperator(sql_op, datasource)     # 执行 SQL
vis_op    = ChartVisOperator(exec_op, chart_type="bar")
stream_op = ResponseStreamOperator(vis_op)          # 转 SSE

dag = DAG("chat_db_pipeline", nodes=[...], edges=[...])
dag.compile()

DAG 一旦编译完成，就触发 _run_node 递归执行。 每个 Operator 都是 async def __call__(self, *args)，输入是上游 Output，输出给下游。

关键洞察：从这一步开始，控制权完全交给 AWEL Runner。 DefaultWorkflowRunner._execute_node 会按拓扑序遍历每个节点， 流式输出通过 SimpleStreamTaskOutput 在节点间传递。

六、阶段 4：LLM 推理（Step 7–10）

Step 7 · LLMOperator 调 Controller

LLMOperator 不直接对接模型，而是通过 ModelControllerClient 找 Worker：

# LLMOperator 内部
async def _call_llm(self, prompt, model_name):
    instance = await self.controllerClient.select_model(model_name)
    return await instance.generate_stream(prompt)

Step 8 · Controller 路由到 Worker

ModelController 在所有注册的 Worker 里挑一个健康的，返回 WorkerRunData。 如果是 proxy 模式，Worker 实际上是 ProxyLLMClient， 会把请求转给 OpenAI/Qwen/Tongyi 等 API。

Step 9 · ModelAdapter 转换格式

不同模型 prompt 格式不一样（chat template / system + user 分隔符 / 特殊 token）。 ModelAdapter 负责：

["system", "You are a SQL expert..."]
["user",   "<schema>CREATE TABLE sales...</schema>\n查询 Top10..."]

→ apply_chat_template() → 模型最终输入 tokens

Step 10 · 流式返回 token chunks

Worker 用 async for chunk in generate_stream(...) 把每个 token chunk yield 给 LLMOperator。 LLMOperator 是 StreamingLLMOperator，它会原样把 stream 推给下游， 实现"边出 token 边执行下一步"的效果。

LLM 大概会用 2–4 秒生成形如：

SELECT customer_name, SUM(amount) AS total
FROM sales
WHERE sale_date >= DATE_SUB(CURDATE(), INTERVAL 30 DAY)
GROUP BY customer_name
ORDER BY total DESC
LIMIT 10;

七、阶段 5：SQL 执行（Step 11–13）

Step 11 · SQLOutParser 解析

scene/chat_db/auto_execute/out_parser.py 用正则 + Markdown code fence 提取 SQL：

llm_output = """```sql\nSELECT ...\n```"""
sql = parse_sql_from_markdown(llm_output)
# → "SELECT customer_name, SUM(amount) ... LIMIT 10"

Step 12 · SqlExecOperator 执行

调用 RDBMSConnector.query_to_df(sql)，背后是 SQLAlchemy：

df = pd.read_sql(sql, self.engine)  # 10 行 DataFrame
注意：默认实现 不防 SQL 注入（让 LLM 直接执行）， 生产环境需要 sqlparse 解析后拒绝 DDL/DROP。社区版本会做基本的白名单过滤， 企业部署建议用只读账号 + 表级权限。

Step 13 · 返回 DataFrame

10 行 × 2 列：customer_name + total。这个 DataFrame 会作为 SqlExecOperator 的输出，传给下游 Vis。

八、阶段 6：可视化与流式回传（Step 14–16）

Step 14 · ChartVisOperator 生成图表配置

调用 dbgpt_core.vis.tags.vis_chart.VisChart，把 DataFrame 转成 ECharts option JSON：

{
  "echarts_type": "bar",
  "title": "近 30 天 Top10 客户销售额",
  "x_axis": ["客户A", "客户B", ...],
  "series": [{ "name": "销售额", "data": [...] }]
}

同时还有 VisTable 把 DataFrame 转 HTML 表格片段。

Step 15 · 流式回传（SSE）

整条 DAG 的最终输出是一个 多 chunk 的流， FastAPI 用 StreamingResponse 推给前端：

data: {"vis_type": "llm_response", "content": "SELECT customer_name..."}

data: {"vis_type": "table", "content": "<table>...</table>"}

data: {"vis_type": "chart", "content": {...echarts option...}}

data: [DONE]

Step 16 · 前端流式渲染

前端根据 vis_type 分发到不同渲染组件：

  • llm_response → Markdown 渲染器（显示 SQL）
  • table → Ant Design Table
  • chart → ECharts React 组件（画出柱状图）

用户看到完整答案。整条链路结束。

九、失败路径：当 LLM 给出错 SQL

真实场景远比上面复杂。下面看几种典型失败路径，理解 DB-GPT 的容错设计。

路径 A · SQL 语法错

LLM 生成 SELET * 这种语法错 → Step 12 SQLAlchemy 抛异常 → 如果配置了 SQLFixAgent，错误信息会回灌给 LLM 让它修正， 最多重试 N 次（默认 3）。

路径 B · Schema 不匹配

LLM 引用了不存在的字段 customer_phone。 这通常说明 Step 5 的 schema 注入不完整（比如大表做了截断）。 调试方法：打开 scene/chat_db/auto_execute/prompt.py 看 SUFFIX 模板的截断阈值。

路径 C · LLM 不返回 SQL

有时 LLM 直接回答"这个查询应该用 ..."却没出 SQL。 SQLOutParser 解析失败 → 抛 ValueError → DAG 中断 → Runner 把错误包成 error vis_type 推给前端。

路径 D · 流式中断

网络中断 / Worker 崩溃。前端 SSE 连接断开，但后端 DAG 还在跑（除非触发 asyncio.CancelledError）。 这是 DB-GPT 的已知技术债，社区在研究如何把 DAG 执行做成可断点续传。

十、实战调试指南

1. 打开 AWEL 调试日志

设置环境变量 DBGPT_AWEL_LOG_LEVEL=DEBUG， 可以看到每个 Operator 的输入输出、节点执行时间。

2. 用 AWEL 可视化编辑器单步

前端 /flow 页面打开任意 DAG，可以单节点运行。 跑到 LLMOperator 时点 "暂停"，就能看到完整 prompt 和 LLM 返回。

3. 抓 LLM 实际 prompt

在 LLMOperator._call_llm 入口加日志： logger.debug("LLM input: %s", prompt)

4. 抓 SQL 实际执行

在 RDBMSConnector.query_ex 加日志，看真实 SQL 是什么、耗时多少。

5. 启用 SQLite + 本地模型

调试时用 dbgpt-local-qwen.toml + SQLite 数据源， 完全本地跑通链路，避免网络/鉴权干扰。

6. 常见问题速查表

症状排查方向
LLM 一直不出 SQL检查 prompt 模板 + DB schema 注入是否完整
SQL 执行报错看 Step 12 的 SQLAlchemy 异常堆栈
图表不显示看 vis_type=chart 的 chunk 是否到达前端
流式卡住Worker 是否健康 / Controller 是否选错 Worker
整个请求 500看后端 stack trace，通常在 DAG 编译或 Operator 执行

十一、全书总结

恭喜你读完了全部 11 章。现在你应该能回答这些核心问题：

核心检查点

  • DB-GPT 的灵魂是什么？ → AWEL（DAG 编排）。所有功能都是 DAG 节点。
  • 为什么分包？ → 依赖倒置：core 抽象、ext 实现、serve/app 依赖 core。
  • SystemApp 解决什么问题？ → 组件注册 + 生命周期 + 避免循环依赖。
  • Scene 和 AWEL 是什么关系？ → Scene 是模板，最终会被编译成 AWEL DAG 执行。
  • Agent 在哪用 AWEL？ → WrappedAgentOperator 把 Agent 包装成 DAG 节点。
  • RAG 怎么进 DAG？ → EmbeddingOperator / RetrievalOperator / RerankOperator。
  • 多模型怎么调度？ → ModelController + WorkerManager + Registry。
  • 怎么扩展一个新功能？ → 写一个 Operator 挂到 DAG；或写一个 ext 实现 core 接口。

推荐下一步学习

  1. 跑通 examples/ 至少 3 个示例，加深对 AWEL API 的肌肉记忆
  2. 写一个新 Operator（比如"调一个外部天气 API"），挂到一个 chat_scene DAG 里
  3. 写一个新 Scene：参考 chat_normal 模板，加一个 chat_weather
  4. 读源码社区的 PR：看大牛怎么给 DB-GPT 加 datasource / LLM adapter
  5. 读论文：arxiv.org/pdf/2312.17449.pdf（DB-GPT 原始论文）
  6. 关注 AWEL 文档：docs.dbgpt.cn 最新版有 AWEL 教程
最后一句：理解 DB-GPT 的最快路径是动手改一行代码。 现在打开 dbgpt_server.py，在 initialize_app 里加一行 print("components:", system_app.list_components())， 启动服务，你就能看到整个 IoC 容器的全貌。
