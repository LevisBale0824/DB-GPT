10 · Ext 扩展层 + 沙箱 + 加速器 Ext

按需安装、插件化、沙箱隔离、GPU 加速 —— DB-GPT 如何用 extras 机制把"大而全"变成"按需装"

一句话定位：Ext 层是 DB-GPT 的"实现仓库"——core 只定义抽象接口 （BaseConnector / VectorStoreBase / ProxyLLMClient）， 所有具体的数据源驱动、向量库适配、LLM 厂商 SDK 对接、RAG 知识源解析器，都放在 dbgpt-ext 包里。 通过 pyproject.toml 的 optional-dependencies 机制，用户只安装自己需要的那部分依赖， 避免"装一个包拉半个 PyPI"的问题。

一、Ext 层定位：core 抽象 → ext 实现

DB-GPT 的代码分为三个层次：

包角色典型内容
dbgpt-core抽象层 + 核心引擎Interface、AWEL、Agent、Scene、Serve、Model Adapter 基类
dbgpt-ext扩展实现层数据源驱动、向量库适配、LLM 厂商、RAG 知识源、Embedding 厂商
dbgpt-sandbox沙箱隔离本地进程 / Docker / Podman / Nerdctl 容器执行
dbgpt-acceleratorGPU/CPU 加速PyTorch 自动检测、vLLM、Flash Attention、量化

关键设计原则：ext 包只依赖 core 的抽象，不依赖任何第三方驱动。 所有的 pymysql、chromadb、pymilvus 等重依赖， 都放进 optional-dependencies，由用户按需安装。

安装方式非常灵活：

# 只装 MySQL 数据源
pip install "dbgpt-ext[datasource_mysql]"

# 装 RAG + ChromaDB 向量库
pip install "dbgpt-ext[rag,storage_chromadb]"

# 全家桶（开发环境）
pip install "dbgpt-ext[datasource_mysql,datasource_postgres,storage_milvus,\
storage_chromadb,rag,graph_rag]"

这种设计的好处显而易见：

  • 最小依赖：只装 MySQL 的用户不需要拉 chromadb（它依赖 onnxruntime，几百 MB）
  • 避免冲突：不同向量库的底层依赖可能版本冲突（如 Faiss 和 Milvus 的 numpy 版本要求不同）
  • 按需启用：扩展器代码在运行时才 import，缺依赖时报清晰的 ImportError

以 MySQL 为例，看看一个数据源驱动有多薄——整个文件只有 53 行：

# dbgpt_ext/datasource/rdbms/conn_mysql.py（完整）
from dbgpt.datasource.rdbms.base import RDBMSConnector, RDBMSDatasourceParameters

@auto_register_resource(
    label=_("MySQL datasource"),
    category=ResourceCategory.DATABASE,
    tags={"order": TAGS_ORDER_HIGH},
    description=_("Fast, reliable, scalable open-source relational DBMS."),
)
@dataclass
class MySQLParameters(RDBMSDatasourceParameters):
    __type__ = "mysql"
    driver: str = field(default="mysql+pymysql",
        metadata={"help": _("Driver name for MySQL, default is mysql+pymysql.")})

    def create_connector(self) -> "MySQLConnector":
        return MySQLConnector.from_parameters(self)

class MySQLConnector(RDBMSConnector):
    db_type: str = "mysql"
    db_dialect: str = "mysql"
    driver: str = "mysql+pymysql"
    default_db = ["information_schema", "performance_schema", "sys", "mysql"]

由于 MySQL 完全兼容 SQLAlchemy 标准，MySQLConnector 几乎不需要覆盖任何方法—— 只需声明 db_type、db_dialect、driver， 剩下的 SQL 执行、表结构查询、字段提取都由 RDBMSConnector 基类统一处理。

相比之下，DuckDB 因为有自己的系统表结构（sqlite_master），需要覆盖几个方法：

# dbgpt_ext/datasource/rdbms/conn_duckdb.py（节选）
class DuckDbConnector(RDBMSConnector):
    db_type: str = "duckdb"
    db_dialect: str = "duckdb"

    def get_table_comments(self, db_name: str):
        with self.session_scope() as session:
            cursor = session.execute(text(
                "SELECT name, sql FROM sqlite_master WHERE type='table'"
            ))
            return [(row[0], row[1]) for row in cursor.fetchall()]

    def table_simple_info(self) -> Iterable[str]:
        with self.session_scope() as session:
            cursor = session.execute(
                text("SELECT name FROM sqlite_master WHERE type='table'")
            )
            for row in cursor.fetchall():
                table_name = row[0]
                cursor_cols = session.execute(
                    text(f"PRAGMA table_info({table_name})")
                )
                cols = [col[1] for col in cursor_cols.fetchall()]
                results.append(f"{table_name}({','.join(cols)});")
            return results

Spark 则因为不是传统关系型数据库，直接继承 BaseConnector，自己实现 run 方法：

# dbgpt_ext/datasource/conn_spark.py（节选）
class SparkConnector(BaseConnector):
    db_type: str = "spark"
    driver: str = "spark"
    dialect: str = "sparksql"

    def __init__(self, file_path, spark_session=None, **kwargs):
        from pyspark.sql import SparkSession
        self.spark_session = (spark_session or
            SparkSession.builder.appName("dbgpt_spark").getOrCreate())
        self.path = file_path
        self.table_name = "temp"
        self.df = self.create_df(self.path)

    def run(self, sql: str, fetch: str = "all"):
        self.df.createOrReplaceTempView(self.table_name)
        df = self.spark_session.sql(sql)
        rows = [df.first().asDict().keys()]
        for row in df.collect():
            rows.append(row)
        return rows

每个数据源都用 @auto_register_resource 装饰器自动注册到 AWEL 资源注册表， 这样在可视化 DAG 编排界面就能拖拽使用。

Milvus 则复杂得多——它支持多种索引类型（HNSW / IVF_FLAT / IVF_SQ8 / ANNOY...）， 还能检测服务端版本，在 2.5.0 以上自动启用全文检索：

# dbgpt_ext/storage/vector_store/milvus_store.py（节选）
class MilvusStore(VectorStoreBase):
    def __init__(self, vector_store_config, name, embedding_fn=None, ...):
        self.index_params = {"index_type": "HNSW", "metric_type": "COSINE"}
        self.index_params_map = {
            "IVF_FLAT": {"params": {"nprobe": 10}},
            "IVF_SQ8":  {"params": {"nprobe": 10}},
            "HNSW":     {"params": {"M": 8, "efConstruction": 64}},
            "ANNOY":    {"params": {"search_k": 10}},
            # ...更多索引类型
        }
        url = f"http://{self.uri}:{self.port}"
        self._milvus_client = MilvusClient(uri=url, user=self.username)

    def is_support_full_text_search(self) -> bool:
        """Milvus 2.5+ 才支持全文检索"""
        milvus_version = self._milvus_client.get_server_version()
        return milvus_version >= "2.5.0"

6.2 组装器（Assembler）

组装器把"知识源 → 切分 → 向量化 → 存储"这条链路封装起来：

# dbgpt_ext/rag/assembler/base.py（节选）
class BaseAssembler(ABC):
    def __init__(self, knowledge: Knowledge,
                 chunk_parameters=None, extractor=None, **kwargs):
        self._knowledge = knowledge
        self._chunk_manager = ChunkManager(
            knowledge=self._knowledge, chunk_parameter=chunk_parameters)
        self._chunks: List[Chunk] = []
        # 加载知识 → 切分
        documents = knowledge.load()
        self._chunks = self._chunk_manager.split(documents)

    @abstractmethod
    def as_retriever(self, **kwargs) -> BaseRetriever:
        """返回一个检索器"""

    @abstractmethod
    def persist(self, **kwargs) -> List[str]:
        """将 chunks 持久化到索引存储"""

EmbeddingAssembler 是最常用的实现，它把 chunks 灌入向量库：

# dbgpt_ext/rag/assembler/embedding.py（节选）
class EmbeddingAssembler(BaseAssembler):
    def __init__(self, knowledge, index_store: IndexStoreBase,
                 retrieve_strategy=RetrieverStrategy.EMBEDDING, **kwargs):
        self._index_store = index_store
        self._retrieve_strategy = retrieve_strategy
        super().__init__(knowledge=knowledge, **kwargs)

    def persist(self, **kwargs) -> List[str]:
        return self._index_store.load_document_with_limit(
            self._chunks, max_chunks_once_load, max_threads)

    def as_retriever(self, top_k=4, **kwargs) -> BaseRetriever:
        return EmbeddingRetriever(
            top_k=top_k, index_store=self._index_store,
            retrieve_strategy=self._retrieve_strategy)

其他 Assembler：BM25Assembler（关键词检索）、 DBSchemaAssembler（数据库 Schema 检索）、 SummaryAssembler（摘要检索）。

Ext 层还提供了 Embedding 模型的厂商适配（独立于 LLM）：

# dbgpt_ext/rag/embeddings/tongyi.py（节选）
class TongYiEmbeddings(BaseModel, Embeddings):
    def embed_documents(self, texts: List[str], max_batch_chunks_size=25):
        from dashscope import TextEmbedding
        # text-embedding-v3 的 batch 不超过 6
        if str(self.model_name) == "text-embedding-v3":
            max_batch_chunks_size = 6
        for i in range(0, len(texts), max_batch_chunks_size):
            batch = texts[i:i+max_batch_chunks_size]
            resp = TextEmbedding.call(model=self.model_name, input=batch)
            sorted_emb = sorted(resp["output"]["embeddings"],
                                key=lambda e: e["text_index"])
            embeddings.extend([r["embedding"] for r in sorted_emb])
        return embeddings

基类 Vis 很简单——核心就是把结构化数据序列化成带 vis_tag 标记的 JSON：

# dbgpt-core/src/dbgpt/vis/base.py（完整核心）
class Vis:
    def sync_display(self, **kwargs) -> Optional[str]:
        content = json.dumps(
            self.sync_generate_param(**kwargs),
            default=serialize, ensure_ascii=False)
        return f"```{self.vis_tag()}\n{content}\n```"

    @classmethod
    def vis_tag(cls) -> str:
        return ""   # 子类覆盖

8.2 核心抽象

SandboxRuntime 是运行时抽象，SandboxSession 是会话抽象：

# dbgpt-sandbox/src/dbgpt_sandbox/sandbox/execution_layer/base.py（节选）
class ExecutionStatus(Enum):
    SUCCESS = "success"
    ERROR = "error"
    TIMEOUT = "timeout"
    RESOURCE_LIMIT = "resource_limit"

@dataclass
class SessionConfig:
    language: str = "python"
    timeout: int = 30           # 秒
    max_memory: int = 256MB
    max_cpus: int = 1
    working_dir: str = "/workspace"
    network_disabled: bool = False

class SandboxSession(ABC):
    @abstractmethod
    async def execute(self, code: str) -> ExecutionResult:
        """执行代码，返回结果"""

class SandboxRuntime(ABC):
    @abstractmethod
    async def create_session(self, session_id, config) -> SandboxSession:
        """创建沙箱会话"""

8.3 本地运行时（LocalRuntime）

本地运行时用 subprocess + psutil 实现进程隔离和资源限制。 它支持 8 种语言：

# dbgpt-sandbox/src/dbgpt_sandbox/sandbox/execution_layer/local_runtime.py（节选）
class LocalSandboxSession(SandboxSession):
    async def execute(self, code: str) -> ExecutionResult:
        # 1. 安全检查
        warnings = self.security_utils.validate_code(code, self.config.language)
        if any("危险操作" in w for w in warnings):
            return ExecutionResult(status=ExecutionStatus.ERROR,
                                   error=f"代码安全检查失败")

        # 2. 写入临时文件
        code_file = self._create_code_file(code)

        # 3. 获取执行命令（python/node/javac/g++/go/rustc/bash）
        command = self._get_exec_command(code_file)

        # 4. 在资源限制下执行
        result = await self._run_with_limits(command)
        # → asyncio.create_subprocess_exec + asyncio.wait_for(timeout)

        return ExecutionResult(
            status=SUCCESS if result["returncode"] == 0 else ERROR,
            output=result["stdout"], error=result["stderr"],
            execution_time=execution_time,
            memory_usage=result.get("memory_usage", 0))

本地运行时是默认禁用的——因为它直接在宿主机执行代码， 有安全风险。需要显式开启：

# 环境变量
export SANDBOX_RUNTIME=local
export SANDBOX_ALLOW_LOCAL_RUNTIME=true

Docker 会话的 execute 方法会把代码 put_archive 到容器， 然后 exec_run 执行：

# dbgpt-sandbox/src/dbgpt_sandbox/sandbox/execution_layer/docker_runtime.py（节选）
class DockerSandboxSession(SandboxSession):
    async def start(self) -> bool:
        self.container = await asyncio.to_thread(
            self.docker_client.containers.run,
            image=self.image_name,
            command="tail -f /dev/null",     # 保持容器运行
            mem_limit=self.config.max_memory,
            cpuset_cpus=str(self.config.max_cpus),
            network_disabled=self.config.network_disabled,
            name=f"sandbox_{self.session_id}")

    async def execute(self, code: str) -> DisplayResult:
        code_file = self._create_code_file(code)
        tar_data = self._create_tar_from_file(code_file)
        self.container.put_archive(self.config.working_dir, tar_data)
        result = self.container.exec_run(
            self._get_exec_command(filename),
            workdir=self.config.working_dir)
        return DisplayResult(
            status="success" if result.exit_code == 0 else "error",
            output=result.output[0].decode("utf-8"),
            error=result.output[1].decode("utf-8"))

8.5 运行时工厂（RuntimeFactory）

工厂负责自动检测本机环境，按优先级选择最佳运行时：

# dbgpt-sandbox/src/dbgpt_sandbox/sandbox/execution_layer/runtime_factory.py（节选）
class RuntimeFactory:
    @staticmethod
    def create(runtime_preference: str = None):
        # 优先读环境变量
        env_choice = SANDBOX_RUNTIME  # os.getenv("SANDBOX_RUNTIME")
        if env_choice:
            runtime_preference = env_choice.lower()

        if runtime_preference == "docker":
            return DockerRuntime()
        elif runtime_preference == "podman":
            return PodmanRuntime()
        elif runtime_preference == "nerdctl":
            return NerdctlRuntime()
        elif runtime_preference == "local":
            return RuntimeFactory._local_runtime()

        # 自动检测（无显式指定时）
        if EnvironmentDetector.is_docker_sdk_available():
            return DockerRuntime()
        if EnvironmentDetector.is_podman_available():
            return PodmanRuntime()
        if EnvironmentDetector.is_nerdctl_available():
            return NerdctlRuntime()
        if SANDBOX_ALLOW_LOCAL_RUNTIME:
            return LocalRuntime()
        raise RuntimeError("No container runtime available")

关键约定：所有扩展都用 @auto_register_resource 或 @register_resource 装饰器注册到 AWEL 资源注册表。 注册后，扩展会自动出现在三个地方：

本章测验

  1. dbgpt-ext 的默认依赖里只有哪两个包？为什么 pymysql 是默认而 mysqlclient 不是？
    参考答案

    默认依赖是 dbgpt>=0.7.0 和 pymysql。 pymysql 是纯 Python 实现的 MySQL 驱动，安装时不需要编译环境； 而 mysqlclient 是 C 扩展，需要系统的 libmysqlclient-dev， 所以放进 datasource_mysql extra 里按需安装。 pymysql 用于 SQLAlchemy 的 mysql+pymysql 驱动即可满足基本需求。

  2. 安装 pip install "dbgpt-ext[rag,storage_chromadb]" 会拉入哪些关键依赖？
    参考答案

    RAG 的依赖：spacy, markdown, bs4, python-pptx, python-docx, olefile, pypdf, pdfplumber。 ChromaDB 的依赖：chromadb（>=0.4.22）, onnxruntime（>=1.14.1,<=1.18.1）。 其中 onnxruntime 约 200MB，是 ChromaDB 默认嵌入模型（all-MiniLM-L6-v2）的推理引擎。

  3. MySQLConnector 只有 53 行代码就能完整工作，靠的是什么设计？
    参考答案

    靠的是 RDBMSConnector 基类的 SQLAlchemy 统一抽象。 所有标准关系型数据库只要兼容 SQLAlchemy，声明 db_type / db_dialect / driver 后，SQL 执行、表结构提取、字段查询、索引获取等全部由基类处理。 只有当数据库的系统表结构与标准不同时（如 DuckDB 用 sqlite_master）， 才需要覆盖特定方法。

  4. 沙箱的 RuntimeFactory 如何选择运行时？为什么 LocalRuntime 默认禁用？
    参考答案

    选择优先级：环境变量 SANDBOX_RUNTIME 显式指定 > Docker SDK > Podman > Nerdctl > LocalRuntime。 LocalRuntime 默认禁用是因为它直接在宿主机用 subprocess 执行代码， 有安全风险（文件读写、网络访问、进程创建无隔离）。 需要设置 SANDBOX_RUNTIME=local 和 SANDBOX_ALLOW_LOCAL_RUNTIME=true 两个环境变量才能启用。

  5. dbgpt-acc-auto 的 cuda118 / cuda121 / cuda124 三个 extras 为什么互斥？
    参考答案

    因为它们对应不同版本的 CUDA Toolkit，会拉入不同版本的 PyTorch wheel。 同一环境下只能安装一个 CUDA 版本的 PyTorch（GPU 驱动只对应一个 CUDA 版本）。 在 pyproject.toml 中用 [tool.uv] conflicts 声明互斥， 安装时会报错提示。用户需根据本机 GPU 驱动的 CUDA 版本选择正确的 extra。
