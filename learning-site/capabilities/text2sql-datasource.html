核心能力 / 自主 SQL 与数据源

把自然语言问题转成可执行的数据查询

DB-GPT 的数据源能力围绕数据库连接、schema summary、Text2SQL、查询执行和结果解释展开。它让业务用户可以通过自然语言访问结构化数据。

API：/api/v1/chat/db/*场景：ChatWithDbQA / ChatWithDbExecute核心：DBConfig / DBSummaryClient

能力定位

Text2SQL 不是简单让模型写 SQL。DB-GPT 先管理数据源，再抽取表结构摘要，让模型基于真实 schema 查询。

连接管理

通过 DBConfig 保存数据库类型、连接参数、用户归属和备注信息。

结构理解

刷新数据源会触发 async_db_summary_embedding，为 Text2SQL 提供表结构上下文。

查询闭环

模型生成 SQL 后执行查询，再把表格结果解释成自然语言或图表。

实现链路

从数据源页面到后端数据服务，核心链路如下。

1. 新增连接前端提交数据库类型、host、port、用户名等参数。
2. API 接收db_connect_add 调用 local_db_manager.add_db。
3. 测试连接test_connect 验证数据库参数是否可用。
4. 刷新摘要db_connect_refresh 删除旧 profile 并重新生成 summary embedding。
5. 聊天使用ChatWithDbQA/Execute 获取表结构上下文，生成 SQL 并解释结果。

流程图：自然语言如何变成 SQL 查询

这个流程的重点是先让系统理解数据库结构，再让模型生成查询。

数据源配置
用户在控制台新增数据库

前端提交 DB 类型、host、port、认证信息，后端用 DBConfig 接收并保存。

连接验证
后端测试连接可用性

test_connect 调用 local_db_manager，确认参数可连接真实数据库。

Schema 摘要
刷新 summary embedding

db_connect_refresh 删除旧 profile，再调用 async_db_summary_embedding 生成表结构上下文。

聊天生成
ChatWithDbQA / Execute 使用 summary

模型基于用户问题和表结构生成 SQL，而不是脱离数据库猜测字段。

执行解释
执行 SQL 并组织答案

查询结果被转成表格、文本解释或图表输出。

如何实现：源码分层拆解

1. API 层

api_v1.py 提供 db_connect_list/add/edit/delete/refresh/test 等接口。

2. 数据源服务层

CFG.local_db_manager 负责数据源持久化、连接测试、summary 生成和数据库 profile 管理。

3. 数据库适配层

dbgpt_ext/datasource 放不同数据库的连接器实现，新增数据库类型通常从这里开始。

4. Chat 场景层

dbgpt_app/scene/chat_db 定义数据库问答和数据库执行场景的 prompt、parser 和 data loader。
