核心能力

DB-GPT 的核心不是“提供一个聊天框”，而是把数据接入、语义理解、SQL/代码执行、知识检索、Agent 编排和结果呈现组织成一条完整的数据分析链路。

AI

Agentic 数据分析

面向复杂分析问题，DB-GPT 不只生成一句回答，而是让 Agent 先理解目标，再拆解任务、选择资源、调用工具并逐步完成分析。

  • 解决问题：把“帮我分析销售下降原因”这类开放问题转成可执行步骤。
  • 实现链路：ChatScene → Agent → Plan/Action → SQL/Code/Tool → Result。
  • 适用场景：经营分析、指标诊断、临时数据探索、自动报告生成。
SQL

自主 SQL 生成与数据查询

系统可以读取数据库连接和 schema summary，让模型围绕真实表结构生成 SQL，执行查询后再解释结果，而不是凭空猜测。

  • 解决问题：业务用户不会写 SQL，也能直接询问数据库。
  • 实现链路：Datasource → DBConfig → Schema Summary → Text2SQL → Query Result。
  • 适用场景：BI 问答、报表取数、数据核查、指标追踪。
RAG

知识库检索增强问答

DB-GPT 支持把文档、资料、报告和制度沉淀为知识空间，在回答时先检索相关片段，再让模型结合上下文生成答案。

  • 解决问题：模型不知道企业内部文档，直接回答容易不准确。
  • 实现链路：Knowledge Space → Chunk → Embedding → Vector Store → Retrieval → Answer。
  • 适用场景：制度问答、文档分析、财报解读、技术资料助手。
SK

Skills 驱动的能力复用

Skills 用来沉淀可复用的分析方法和执行流程，例如 CSV 分析、财报分析、销售分析，让 Agent 不必每次从零开始。

  • 解决问题：团队内重复分析方法无法标准化、复用和分享。
  • 实现链路：Skill Definition → Upload/Import → Agentic Data API → Sandbox/Tool Execution。
  • 适用场景：行业模板、团队方法库、固定报告流程、专家经验沉淀。
BOX

沙箱化代码执行

当 Agent 需要运行 Python 或分析脚本时，DB-GPT 通过独立 sandbox 承载执行环境，降低模型生成代码直接影响主服务的风险。

  • 解决问题：AI 生成代码需要执行，但不能无边界地运行在主服务里。
  • 实现链路：Agent Action → Code Operator → dbgpt-sandbox → Execution Result。
  • 适用场景：数据清洗、图表生成、文件分析、临时计算任务。
UI

可视化数据智能控制台

前端提供聊天、数据源、知识库、模型、Prompt、Flow、Skills、应用构建等入口，让 DB-GPT 不只是 SDK，而是可操作的平台。

  • 解决问题：数据智能能力需要被配置、管理、调试和复用。
  • 实现链路：Next.js Pages → Client/Hooks → FastAPI Routers → Services。
  • 适用场景：团队内部平台、数据助手门户、模型与知识库运维后台。

典型使用场景

把 DB-GPT 放到真实业务里看，它更像一个“数据分析操作系统”：负责接入数据、理解问题、执行分析、沉淀方法。

01

数据库自然语言问答

业务人员输入“本月销售额环比为什么下降”，系统基于表结构生成 SQL，查询数据并解释结果。

  • 适合 BI 问答
  • 核心链路：Datasource + Text2SQL + ChatScene
02

企业知识库分析

将制度、报告、手册、合同等资料构建知识空间，让模型在检索上下文后回答问题。

  • 适合内部知识助手
  • 核心链路：KnowledgeService + RAG + Embedding
03

数据文件分析

上传 CSV、Excel、PDF 等文件，由 Agent 读取、清洗、计算指标并输出图表或结论。

  • 适合临时分析
  • 核心链路：File upload + Sandbox + Report
04

可复用分析技能

把固定分析方法沉淀为 Skills，例如财报分析、销售分析、CSV 数据分析，后续直接复用。

  • 适合团队方法库
  • 核心链路：Skills + Agentic Data API
05

Agent 应用构建

基于 App、Prompt、资源和工作流搭建具体数据应用，面向不同业务场景复用。

  • 适合内部应用平台
  • 核心链路：dbgpt-serve agent app + flow
06

私有化数据助手

通过本地部署、代理模型、沙箱执行和权限控制，构建企业内可控的数据助手。

  • 适合私有数据场景
  • 核心链路：configs + docker + auth + sandbox

系统架构

从用户界面到模型与工具执行，DB-GPT 采用“前端产品层 → 应用装配层 → 业务服务层 → 核心能力层 → 扩展接入层”的结构。

产品访问层用户看到和操作 DB-GPT 的入口。
Web 控制台

web/pages 提供 chat、construct、knowledge、models、skills、flow、share 等页面。

流式聊天交互

web/hooks/use-chat.ts 通过 fetchEventSource 消费 SSE，逐步渲染回答和工具输出。

管理操作入口

数据源、知识库、Skills、模型和应用构建页面通过 axios/fetch 调用后端 REST API。

请求流向：页面行为 → API client / SSE → FastAPI router返回流向：JSON / SSE chunk → Hook 状态 → UI 组件
应用装配层负责启动服务、注册路由和组织聊天场景。
FastAPI 服务

dbgpt_app/dbgpt_server.py 创建应用，挂载 OpenAPI、知识库、静态资源和中间件。

Chat 路由

api_v1.py 的 chat_completions 根据 ChatScene 分流普通聊天、知识库、Agent 和 Flow。

Agentic Data API

agentic_data_api.py 编排 SQL、知识检索、代码解释器、shell、skills 和步骤流。

业务服务层把产品概念落到可复用服务。
Datasource Service

dbgpt_serve/datasource 管理数据库连接、连接测试、schema summary 和 profile。

Knowledge Service

knowledge/api.py 与 knowledge/service.py 管理 space、document、chunk、embedding 和召回。

App / Flow / Conversation

dbgpt_serve 承载应用、会话、工作流、模型服务和反馈等业务对象。

核心能力层DB-GPT 的推理、执行和数据智能底座。
Agent Runtime

dbgpt-core/agent 定义 Agent、ConversableAgent、ReActAgent 和工具调用闭环。

RAG / Storage / Model

dbgpt/rag、storage、model adapter 负责检索增强、向量存储、模型接入与上下文构造。

Sandbox Execution

dbgpt-sandbox 用 SandboxRuntime 和 SandboxSession 隔离执行代码与脚本。

扩展接入层连接外部模型、数据库、向量库和运行环境。
数据库与数据仓库

dbgpt_ext/datasource 接入 MySQL、Postgres、ClickHouse、DuckDB、Hive 等结构化数据源。

模型与向量库

通过 provider 配置接入 OpenAI、DeepSeek、Qwen、Ollama，以及 Chroma、Milvus、Qdrant 等向量库。

部署与运行

configs、docker、docker-compose.yml 负责私有化部署、元数据库和服务编排。

源码模块地图

下面按职责解释核心目录，适合作为源码阅读和二次开发的入口页。

packages/dbgpt-core

核心框架层

Agent、模型、RAG、数据源、存储、CLI、可视化和工具函数。改底层抽象时先看这里。

packages/dbgpt-app

应用装配层

FastAPI 服务入口、OpenAPI 路由、聊天场景、初始化流程、默认技能和静态资源。

packages/dbgpt-serve

业务服务层

Agent 应用、数据源、会话、Flow、模型服务、知识库、定时任务和反馈服务。

packages/dbgpt-ext

扩展接入层

MySQL、Postgres、ClickHouse、DuckDB、Milvus、Chroma、Qdrant、LLM provider 等外部能力。

packages/dbgpt-client

SDK 与客户端

提供 Python client，方便外部脚本或应用调用 DB-GPT 的 app、knowledge、flow、datasource 等能力。

packages/dbgpt-sandbox

安全执行环境

Agent 生成代码或分析脚本时的执行环境，适合重点理解权限、隔离和执行结果返回。

web

前端产品层

Next.js + React 页面，包括 chat、construct、knowledge、models、skills、flow、share 等页面。

configs / docker

运行与部署

不同模型 provider、本地模型、MySQL、容器镜像、数据卷和一键启动示例。

技术栈

DB-GPT 是一个典型的 AI 应用 monorepo：Python 负责 AI、服务和数据能力，TypeScript/React 负责产品界面，Docker 负责快速交付。

PY

Python 3.10+

后端主体语言，使用 FastAPI、Click、SQLAlchemy、Pydantic、Pandas 等生态支撑服务、CLI 和数据处理。

UV

uv workspace

根目录 pyproject.toml 管理多个 workspace package，适合 monorepo 开发和统一依赖锁定。

API

FastAPI

dbgpt_app/dbgpt_server.py 创建应用，openapi 目录承载 Chat、Editor、Feedback、AgenticData 等路由。

AI

LLM Provider

通过配置接入 OpenAI、DeepSeek、Qwen、Ollama、LiteLLM、本地模型和 vLLM 等推理后端。

RAG

Vector Storage

扩展包支持 Chroma、Milvus、Qdrant、Weaviate、Elasticsearch 等向量检索后端。

WEB

Next.js + React

前端位于 web，使用 Ant Design、MUI、AntV、Monaco、React Flow 等构建交互式控制台。

DB

MySQL / SQLite

元数据库保存连接配置、会话、应用、知识库等信息；示例数据也包含 SQLite 文件。

OPS

Docker + Compose

提供 MySQL、Webserver、数据卷、模型挂载和配置挂载，适合快速试用和私有化验证。

QA

Ruff / Pytest / Mypy

项目提供格式化、单测、doctest、类型检查等质量工具，贡献前应按改动范围运行。

1

新增 API

先在 dbgpt-app/openapi 定义路由，再确认是否在 mount_routers() 中注册。

2

新增数据源

优先看 dbgpt-ext/datasource 和 dbgpt_serve/datasource，补连接参数、测试和 summary 逻辑。

3

新增 Skill

4

改前端页面

从 web/pages 定位页面，再追 components、hooks、client，最后回搜后端 API。

常见问题

快速回答阅读和改造 DB-GPT 时最容易卡住的问题。

我应该从前端还是后端开始？

如果目标是理解产品，先从前端页面开始；如果目标是改功能，先从 API 和服务入口开始；如果目标是理解 Agent 能力，直接进入 dbgpt-core/src/dbgpt/agent。

DB-GPT 最核心的源码文件是哪几个？

cli_scripts.py、_cli.py、dbgpt_server.py、api_v1.py、agent/core/agent.py、web/pages/chat/index.tsx 是第一批入口。

如何判断一个功能改动应该放在哪个包？

HTTP 入口放 dbgpt-app；业务服务放 dbgpt-serve；基础抽象放 dbgpt-core；外部接入放 dbgpt-ext；界面交互放 web。
